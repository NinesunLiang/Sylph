# Web 性能审查 — 危险信号自检清单

在输出最终报告前，逐条自检。任何未通过项 → 回溯对应步骤修正。

## 准确性检查- [ ] 是否在未运行 `next build` / `next/bundle-analyzer` 的情况下凭经验估算 bundle 大小？（必须有实际数据）- [ ] 是否把 Next.js 自动优化的图片（`next/image`）误标为未优化？（检查是否已使用 `<Image>` 组件）- [ ] 是否确认了项目的渲染策略（SSR/SSG/ISR/CSR）后才判定渲染方式不当？（读取 route segment config / `getServerSideProps` / `getStaticProps`）- [ ] 是否区分了首屏关键资源和非首屏懒加载资源？（非首屏资源不报 LCP 相关 P0）- [ ] 是否检查了 `dynamic(() => import(...))` 的 `ssr: false` 选项后才判定 SSR bundle 膨胀？（动态导入可能已排除 SSR）

## 证据完整性检查- [ ] 6 大类别所有规则是否每条都有实际执行的命令 + 原始输出？- [ ] 是否有规则被"跳过"而未说明原因？- [ ] 严重度计数（P0/P1/P2/P3）是否与详情条目数一致？- [ ] re-scan 的 before/after 对比表是否完整（每条规则一行）？- [ ] bundle 分析数据是否引用了实际的构建输出或 `.next/analyze` 报告？

## 误报防护检查- [ ] 每个标记为 FP 的命中项是否有具体理由（非笼统"已排除"）？- [ ] 第三方库强制要求的加载方式（如 `<script>` 标签）是否已排除？- [ ] `use client` 指令是否因交互需求而合理使用？（不应盲目标记为 P1）- [ ] 大依赖（如 `lodash`、`moment`）是否已通过 tree-shaking 按需引入？（确认 import 方式后再判定）- [ ] 非用户可见的后台页面/管理端是否降低了 Web Vitals 严重度？

## 修复质量检查- [ ] 修复是否复用了项目已有的优化模式（如已有的 `dynamic()` 用法、已有的 Image 配置）？- [ ] 修复是否引入了新的性能问题（如过度拆分导致请求瀑布、过度懒加载导致 CLS）？- [ ] 建议的 `next/image` 配置是否与项目 `next.config.js` 中的 `images` 配置兼容？- [ ] blocked 项是否有完整的失败原因 + 建议代码？

## 不确定性处理- [ ] Lighthouse 未实际运行时，Web Vitals 数据是否标记为"需实测验证"？- [ ] 服务端渲染性能（TTFB）是否标记为"需生产环境验证"？- [ ] 第三方脚本影响是否标记为"需 RUM 数据确认"？- [ ] 是否以确定语气报告了实际不确定的结果？（= 过度自信违规）
