# 前端样式审查 — 危险信号自检清单

在输出最终报告前，逐条自检。任何未通过项 → 回溯对应步骤修正。

## 准确性检查

- [ ] 是否把第三方库要求的内联样式误标为 P1？（B3 误报）
- [ ] 是否确认了项目确实支持暗色模式后才报 D1？（检查 DARK_MODE_SUPPORTED 标志）
- [ ] 是否在 Token 允许列表中查找后才判定 A1/A2 命中？（非凭感觉判断）
- [ ] 是否区分了动态运行时值和静态硬编码值？（B3 深度检查）
- [ ] 是否检查了 C1 固定宽度的 max-width / overflow 上下文？（C1 深度检查）

## 证据完整性检查

- [ ] 所有规则是否每条都有实际执行的命令 + 原始输出？
- [ ] 是否有规则被"跳过"而未说明原因？
- [ ] 严重度计数（P0/P1/P2/P3）是否与详情条目数一致？
- [ ] re-scan 的 before/after 对比表是否完整（18 行）？

## 误报防护检查

- [ ] 每个标记为 FP 的命中项是否有具体理由（非笼统"已排除"）？
- [ ] SVG 内样式是否已正确排除？
- [ ] `// tailwind-ignore` 注释的例外是否理由合理？

## 修复质量检查

- [ ] 修复是否复用了项目已有 Token / 类名模式？
- [ ] 修复是否匹配项目暗色方案配色？
- [ ] 修复是否引入了新的样式问题（惯性断路器自检）？
- [ ] blocked 项是否有完整的失败原因 + 建议代码？

## 不确定性处理

- [ ] 不确定的命中项是否标记为"待确认"？
- [ ] D2 对比度检查是否标记为"需人工验证"？
- [ ] B1 JIT 动态类是否标记为"需构建验证"？
- [ ] 是否以确定语气报告了实际不确定的结果？（= 过度自信违规）
