# React/Next.js 审查 — 危险信号自检清单

在输出最终报告前，逐条自检。任何未通过项 → 回溯对应步骤修正。

## 准确性检查（React 特有）

- [ ] B1 useEffect 依赖：是否区分了 ref 对象（稳定引用，无需依赖）和普通变量？误将 ref 标为缺失依赖 = 误报
- [ ] B2 清理函数：是否确认副作用类型确实需要清理？（纯数据获取 + AbortController 是最佳实践但非 P0，单次 fetch 无清理可降级 P2）
- [ ] B3 条件 hooks：是否区分了条件调用 hook（违规）和 hook 内部的条件逻辑（合法）？`useMemo(() => cond ? a : b, [cond])` 不是违规
- [ ] A1 渲染新引用：是否检查了父组件是否已 React.memo 包裹？无 memo 的父组件传递内联对象给无 memo 的子组件 → 可能不触发额外 re-render（取决于父组件自身是否 re-render）
- [ ] C1 Client/Server：是否确认了 `'use client'` 文件中的逻辑确实可提升到 Server？涉及 hooks、事件处理、浏览器 API 的代码不可提升

## 证据完整性检查

- [ ] 所有规则是否每条都有实际执行的命令 + 原始输出？
- [ ] 是否有规则被"跳过"而未说明原因？
- [ ] 严重度计数（P0/P1/P2/P3）是否与详情条目数一致？
- [ ] re-scan 的 before/after 对比表是否完整（18 行）？
- [ ] 是否每条规则都扫描了所有变更文件（而非仅部分文件）？

## 误报防护检查

- [ ] 每个标记为 FP 的命中项是否有具体理由（非笼统"已排除"）？
- [ ] Next.js 约定文件（layout.tsx、loading.tsx、error.tsx）的特殊模式是否已正确识别？
- [ ] 第三方组件库 API 要求的 props 模式是否已排除？（如 `style` prop 是组件 API 的一部分）
- [ ] E2 index key：是否确认列表会增删排序？静态列表用 index key 是 FP
- [ ] D2 prop drilling：是否确认超过 3 层？2 层传递不是违规

## 修复质量检查

- [ ] 修复是否匹配项目已有模式（状态管理、hooks 风格、组件结构）？
- [ ] B1 修复添加依赖后是否可能导致无限循环（对象/数组依赖 → 需 useMemo 包裹）？
- [ ] 修复是否引入了新的渲染性能问题（惯性断路器自检）？
- [ ] blocked 项是否有完整的失败原因 + 建议代码？
- [ ] 修复代码是否保持了 TypeScript 类型安全？

## 不确定性处理

- [ ] 不确定的命中项是否标记为"待确认"？
- [ ] A3 频繁 re-render 分析是否标记为"待确认：需运行时 Profiler 验证"？（AI 无法确定实际 re-render 频率）
- [ ] D2 prop drilling 层数追踪是否完整？（跨文件追踪可能不完整 → 标记"待确认"）
- [ ] 是否以确定语气报告了实际不确定的结果？（= 过度自信违规）
