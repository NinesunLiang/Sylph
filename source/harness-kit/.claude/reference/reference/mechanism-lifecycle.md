# 机制生命周期 — 保留/删除/激活判定

> Carror OS 机制治理的核心判定原则。评估任何 hook/skill/机制时遵循。

## 标准一：哲学 > 铁律 > 需求

```
哲学 7条 → 铁律 8条 → 项目需求 → 增益 vs 噪声
```

- 三级筛选：任何机制必须能追溯到至少一条哲学或铁律
- 增益 >> 噪声（维持成本、运行时开销、维护负担）
- 增益/噪声比为负 → 即使有哲学支撑也应删除

## 标准二：区分「意图不对」vs「实现不对」

| 情况 | 处理 |
|------|------|
| **意图不对** — 哲学/铁律无支撑，原始设计目标本身错误 | **删除** |
| **实现不对** — 意图正确，但 bug/埋点路径错/未注册/观测不到 | **改实现，不删除** |

> **read-tracker.sh 零 flywheel 事件案例**：初始误判为「无价值」。追溯后确认它是 edit-guard 的数据核心依赖。问题不在意图（Read-before-Edit = 铁律#1），而在观测方法（数据型 hook 不写 flywheel，事件数不反映效果）。

**操作约束**：评估任何机制前，先查 git log 追溯原始创建意图。禁止仅凭 flywheel 零事件 = 无价值的粗暴判断。

## 标准三：机制采纳门禁三问

新增机制前必答：

1. **收益可证伪吗？** 该机制成功时产生什么可观测信号？失败时呢？
2. **噪声上限明确吗？** false positive 率上限？超过后谁负责关闭？
3. **如果 0 收益，多久能发现？** 定义观察期和终止条件。

| 案例 | 判决 | 依据 |
|------|------|------|
| **ED-01**: error-dna-auto-fix v1 (83.5%噪声) | 搁置归档 | 标准一：增益<噪声。标准三：0次repair_success |
| **ED-01 v6.3.27**: error-dna-auto-fix 重新激活 | 重新评估→激活 | 标准二：意图正确(跨会话回顾)但旧实现观测不对。新评估：知识管道成熟后角色转变 |

| **5个幽灵hook** (anti_pattern_detect等): 文件从不存在 | 清理harness声明 | 标准一：无实现=无增益。标准二：不适用(从未实现过) |
