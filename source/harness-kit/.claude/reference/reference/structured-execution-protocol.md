# 结构化执行协议 (Structured Execution Protocol)

> 本文档是五阶段结构化执行协议的单一权威定义。所有其他文档引用本文档。任何不一致以本文档为准。

> 哲学 #4(验证) + #6(0信任) + #5(人) 的物化: 每个非 trivial 任务走 5 阶段双法官门禁

## 5 阶段流程

```
人类参与                          无人区(自主执行)                      人类参与
  ↓                                   ↓                               ↓
Phase 1: 调研 → Phase 2: 方案双审 → Phase 3: 执行 → Phase 4: 结果双审 → Phase 5: 验收报告
  ↑              ↑ Oracle+Meta        ↑ 不暂停       ↑ Oracle+Meta          ↑ 交付人类
  需求+哲学        ← ACCEPT →          不提问         ← ACCEPT →            结构化报告
  +铁律+现状                           不中断
  +ROI                               只记录
```

**Phase 3-4 = 无人区**: AI 全自动执行，不暂停、不提问、不中断、只记录。
卡点处理: 硬边界→跳过记录 | 可绕行→自动降级 | 真阻断→记录 blocked 继续其他。
危险操作走三级裁决链(哲学→Oracle→Meta-Oracle)，不可逆操作记录 skip-risk 在退出报告中。

## 各阶段输入/输出

| Phase | 输入 | 输出 | 门禁 |
|-------|------|------|:----:|
| 1. 调研 | 需求 + 哲学7条 + 铁律8条 + 现状 + ROI | 执行方案(调研结论+改动清单+验收标准) | — |
| 2. 方案双审 | 执行方案 | Oracle ACCEPT + Meta-Oracle ACCEPT | 🔴 双签强制 |
| 3. 执行 | 批准方案 | 改动代码 + 验证结果 | — |
| 4. 结果双审 | 执行记录(过程+结果+验证证据) | Oracle ACCEPT + Meta-Oracle ACCEPT | 🔴 双签强制 |
| 5. 验收报告 | 全部通过 | 结构化报告交付人类 | 人类确认 |

## Phase 1: 调研

AI 基于 4 个判断条件:
- **需求**: 用户明确目标
- **哲学**: 哲学7条优先级裁决 (#4>#6>#3>#7>#5>#2>#1)
- **铁律**: 8条铁律约束检查
- **现状**: 当前代码/配置状态 (Read 源码验证)
- **ROI**: 收益/成本判断

输出: 执行方案文件 (.omc/plans/ 或指定路径)

## Phase 2: 方案双审

- Oracle (critic agent): 静态检查 + 文件存在性 + 合规审查
- Meta-Oracle (critic agent): 运行时验证 + 烟雾日志 + 对抗性审查 + 设计盲区扫描
- 双签 ACCEPT → 进入执行; REVISE → 修方案重审; REJECT → 终止

## Phase 3: 执行

- 按批准方案执行
- 改动记录: 文件清单 + 行数 + 操作类型
- 验证证据: 烟雾测试 + bash -n + audit-hooks

## Phase 4: 结果双审

- 提交执行记录 (实际改动 vs 计划改动 + 验证结果)
- Oracle + Meta-Oracle 审核执行质量
- ACCEPT → 进入验收报告

## Phase 5: 验收报告

提交给人类:
- 执行摘要 (做了什么)
- 验收证据 (验证结果)
- 决策记录 (关键选择+理由)
- 未解决项 (延后/放弃)
- 下一步建议

## 触发条件

- 非 trivial 任务 (>3文件 或 新功能 或 架构变更)
- 用户说 "do" / "执行" / "方案" 等触发词
- 自主模式 goal/ghost 自动遵循

## 豁免

- L1 简单变更 (单文件/typo/单行修复) → 可跳过双审但必须自证
- 紧急修复 → 可先执行后补审
