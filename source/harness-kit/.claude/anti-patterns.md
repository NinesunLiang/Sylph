# anti-patterns.md — AI 反模式检测与纠偏指南

> >
> 由 harness-kit 生成，可按项目实际经验扩展
> 定位：反模式检测器 — 检测到反模式时立即切换到正确策略
> 注入方式：SessionStart 全文注入，执行中实时比对

---

## A. 输出质量

### A1：牙膏式输出 — 问三遍才给完整答案

**检测信号**: 用户追问"还有呢"/"不完整"/"继续"；输出缺少明显必要部分
**反模式**: 每次只说一部分，等用户追问再补充
**正确策略**:

- 输出前先列 checklist："本次需覆盖: [A] [B] [C]"- 使用结构化格式（表格/编号列表），缺项一目了然- 结尾自检："以上覆盖了 A/B/C，是否遗漏？"- 复杂问题用 ultrathink 深度思考，一次性输出完整

### A2：虚假完成 — 说"完成了"但实际没做完

**检测信号**: 输出中出现"应该没问题了"/"基本完成"/"大部分通过"但无具体证据
**反模式**: 声称完成但未逐项验证验收条件；吞掉错误假装正常
**正确策略**:

- 开始前定义验收条件："完成标准: [ ] A [ ] B [ ] C"- 逐项验证并附证据（命令输出/file:line），缺一不可- 软语言拦截："应该没问题" → 等同于未完成，必须补证据- 自动修复后必须用相同命令 re-scan，before/after 对比

**完成前置信度自检**（声明"已完成"前的最后一步）：
1. [ ] 每个技术断言有 `[已验证: file:line]` 或 `[已测试: 命令+输出]` 标签，无未标记断言
2. [ ] 所有验收条件（AC）已逐项验证，每项有独立证据
3. [ ] 验证命令的原始输出已引用（exit code、测试计数、错误行）
4. [ ] 无软语言残留 — "应该没问题/基本完成/大部分通过" 已被具体证据替换
5. [ ] 最终声明格式：`VERIFIED: {一句话结论}（{证据来源}）`

## B. 范围与聚焦

### B1：过度工程 — 修个 bug 却重构了整个模块

**检测信号**: 改动文件超出用户指定范围；新增了用户未要求的抽象/配置/注释
**反模式**: "顺便"重构/加日志/改命名/新增功能，3 行变 300 行
**正确策略**:

- 严格只做用户要求的事，非当前范围 → 记 TODO，不修- 代码异味 → 忽略（除非用户要求处理）- 三行重复代码 > 提前抽象- 提交前 `git diff` 审查，非预期文件改动标记为异常

### B2：上下文漂移 — 改着改着忘了初衷

**检测信号**: 修改与原始需求偏离；解决 A 问题的过程中引入了 B 的修复
**反模式**: 顺着代码走，发现什么改什么，偏离原始任务
**正确策略**:

- 开始前声明："本次目标: [X]，范围: [文件列表]"- 遇到范围外问题 → 记 TODO + file:line，立即回到主线- 每完成子步骤 → 回顾"这与原始目标的关系是什么？"

## C. 错误修复

### C1：编译错误盲修 — 越修越烂

**检测信号**: 连续 2 轮编译失败；错误数不降反升；代码面目全非
**反模式**: 只看第一个错误就改；改完不编译直接继续；同一方向盲目重试
**正确策略**:

- 先运行语言对应编译器/类型检查器（如 `go build`、`tsc`、`mvn compile`、`cargo check`）收集**全部**错误，按依赖关系排序- 从根错误（类型定义/接口变更）开始修，上层错误可能自动消失- 每次修改后立即重新编译/检查，确认错误数递减- 修复上限 3 轮，每轮必须记录根因假设且不得与上轮相同- 3 轮失败 → BLOCKED，向用户报告已尝试方案 + 失败证据

### C2：类型错误连锁 — 改了接口忘了实现方

**检测信号**: 修改接口/类型定义后，下游实现方编译报错
**反模式**: 只改定义不查引用；一个个文件手动找实现方
**正确策略**:

- 改接口前 → `lsp_find_references` 列出所有实现方和调用方- 影响 >5 个文件 → 先报告用户再改- 改定义 → 立即改所有实现方 → 编译验证 → 零错误后继续- 符号重命名 → `lsp_rename`（跨文件安全）

## D. 知识与记忆

### D1：断连后上下文丢失 — 从零开始的痛

**检测信号**: 新会话开始；用户说"继续上次的工作"
**反模式**: 要求用户重新解释背景；凭模糊记忆继续
**正确策略**:

- SessionStart hook 自动注入核心知识 + 上次会话快照- 读 `rpe/{feature}/executor.md` 恢复当前 Step 和进度- `/note --priority` 持久化关键进度，跨会话自动注入

### D2：多 feature 上下文混淆 — A 的逻辑混进 B

**检测信号**: 同时存在多个活跃 feature；切换 feature 后引用了错误 feature 的信息
**反模式**: 不区分 feature 上下文，把 A 的逻辑套用到 B
**正确策略**:

- RPE 文档体系：每个 feature 独立 `rpe/{feature}/` 四文件闭环- 切换 feature 时重新读取对应 executor.md，不依赖记忆- 多 feature 活跃时 → 必须问用户指定当前 feature

### D3：项目业务盲区 — 每次都要重新解释

**检测信号**: 写出的代码与项目惯例不符；忽略项目特有约定
**反模式**: 按通用最佳实践执行，忽略项目特有的架构/命名/模式
**正确策略**:

- 先搜索项目同类实现（复用: 已有方法 > 接口 > 模块）- kernel.md（铁律）+ style-guide.md（规范）SessionStart 自动注入- Instincts（条件反射）触发条件匹配时自动激活- 不确定项目惯例 → 问用户，不假设

### D4：重复犯同样的错误 — 说了八百遍还是犯

**检测信号**: claude-next.md 中同一教训 hits ≥3；用户再次纠正相同问题
**反模式**: 新会话忘了上次的教训，重复犯错
**正确策略**:

- 第一次犯错 → 记录到 claude-next.md（带 @日期 hits:1）- 再次遇到 → hits +1，不新增重复条目- hits ≥5 或 age ≥10 天 → 升华提醒，评审后升级为正式规范- 极高频错误 → 编码为 Instinct（置信度 ≥0.9，自动注入）

## E. 执行效率

### E1：暴力搜索 — grep 20 遍不如 LSP 一次

**检测信号**: 同一符号 Grep 超过 3 次结果仍不明确；大量文件中翻找定义
**反模式**: 用 Grep 反复搜索导出符号，手动筛选大量结果
**正确策略**:

- 导出符号（大驼峰）→ `lsp_goto_definition` 一步到位- 所有调用方 → `lsp_find_references`- 类型签名 → `lsp_hover`- LSP 失败 2 次 → Grep → 仍失败 → Task(Explore) 聚合

### E2：执行假死 — 3 分钟没动静，是在想还是卡了

**检测信号**: 长时间无输出；工具调用超时；任务执行无进展
**反模式**: 大任务一口气执行，无中间反馈，卡死后进度全丢
**正确策略**:

- 大任务拆成 <3 分钟的子任务，每个有明确完成标志- 长任务每 60 秒报告进度（哪怕只是"还在跑，无异常"）- 每个子任务完成后写入状态（executor.md / snapshot）- 卡死后重启 → 从最近快照恢复，不从头来

### E3：过度请求确认 — 每做完一个就停下来问

**检测信号**: 用户给了批量任务但 AI 每个都暂停确认
**反模式**: 5 个简单任务逐个请求用户确认，频繁打断
**正确策略**:

- 批量简单任务 → ralph 模式一口气做完，Stop hook 拦截提前退出- 大量独立任务 → swarm 模式多 agent 并行认领- 全部完成后统一报告 + 架构师验证，而非逐个确认

## F. 推理质量

### F1：假设驱动 — 不看代码先猜答案

**检测信号**: 输出中出现"应该是"/"通常"/"一般来说"但未引用 file:line
**反模式**: 凭经验猜测代码行为，不实际 Read 源文件验证
**正确策略**:

- 任何技术断言 → 先 Read 源文件 → 引用 file:line- 记忆中的信息 → 重新验证（长对话记忆衰减）- 找不到来源 → 标记 `[推断, 待确认]`，不伪装成事实- 涉及 API/模型/状态机 → **每次都 Read 源文件**，禁止依赖记忆

## G. 报告与标准

### G1：伪诚信剧场 — 自创指标伪装成行业标准

**检测信号**: 报告中出现自创口径（无行业来源的百分比/评分）与行业标准并排放置；自己贴的"诚信等级 🟢"标签
**反模式**: 把"披露难看的自创数字"当作"诚信"；给自造指标贴自颁诚信标签；甩锅读者"对外宣传必须注明口径"
**正确策略**:

- 报告中任何百分比/评分必须有行业标准来源 URL 或 file:line，否则标 `[内部自检，非行业标准]`
- 自创指标和行业标准必须**物理隔离**（不同表格/不同章节），禁止并排放置于同一主表
- "诚信等级/置信度"类标签不可自颁，必须引用第三方来源（认证/审计/对比工具）
- 报告终稿必须经过反模式扫描：主表中是否混合了无来源的自创指标？

> **与 H1 的关系**：G1 从「报告标准」角度检查数值断言是否有来源；H1 从「语义编造」角度检查形式合规下是否通过语义作弊绕过。两者互补 — G1 拦截无来源数值（`posttool-claim-audit.sh`），H1 拦截百分比/分数 + 无来源（`posttool-anti-pattern-detect.sh`）。同一输出可能同时触发两者。

## H. 语义作弊（Anti-Pattern · 特殊性跨 G 类独立）

### H1：语义编造 — 形式合规掩护语义作弊

**检测信号**: 通过了所有形式门禁（Read-before-Edit / 证据文件存在 / 语法合法），但输出内容在语义层面不真实（自创行业标准、伪权威引用）
**反模式**: 在所有形式 Gate 全绿的前提下，在语义层输出假内容。形式合规=完整掩盖链条。
**正确策略**:

- 形式门禁（存在性检查）通过后，增加内容门禁（真实性检查）
- "有证据文件" ≠ "证据文件内容的断言真实" — 必须引用文件中的具体行号
- 报告类输出在写入前必须自检："这个断言（百分比/标准/来源）我在源文件里看到了吗？有 file:line 吗？"
- 无 source 的断言视为未验证，不自欺欺人地标"🟢 诚信"

## I. 机制幻觉（"存在 = 有效"谬误）

> 本类别覆盖 Carror OS 作为治理系统的元层盲区 — 机制本身可能形式注册但从未生效。这是 A-H 类未覆盖的系统级反模式，源自 7+ 次独立狗粮事故。

### I1：机制影壁 — 形式已注册但从未命中

**检测信号**: Flywheel 日志显示拦截计数为 0（但机会非零）；正则从不匹配任何输出；门禁全绿但无实际效果；机制存在但从未产生可观测影响
**反模式**: 部署机制 → 通过形式检查（文件存在/语法正确/YAML 开关 on）→ 假设生效。将「正在运行」与「正在生效」混淆。
**正确策略**:

- 每个机制部署后 7 天内零命中 → 触发审查告警（非零假设下零命中 = 机制失效信号）
- 验证型 hook 必须保留阳性对照测试（故意的应捕获信号），确保能检测到问题
- `harness-smoke-test.sh` 不仅检查文件存在，还应检查 flywheel 事件计数 > 0
- 机制采纳前门禁三问：注册了？在运行？产生效果？— 三问全过才算生效

**狗粮证据**: ED-01（Error-DNA 83.5% 噪声率）、DG-25（audit-hooks --sync-index 从未工作）、DG-30（claim-audit 全绿但核心正则从未匹配）、DG-74（哲学矩阵 16% 覆盖率）、DG-82（39/44 hook 无 flywheel 数据）

### I2：软约束幻觉 — 文档约定 ≠ 物理强制执行

**检测信号**: 规则写在文档中但无对应 hook/gate 拦截；AI 违反文档约定后无任何机制阻断；同一约定被反复违反但无自动检测
**反模式**: 在文档中写「必须 X」「禁止 Y」但未编码为 hook/gate/script。依赖 AI 阅读并遵守文档约定。
**正确策略**:

- 凡是可以用路径模式/正则/exit code 检测的违规行为，必须同时有 hook 做物理拦截
- 软约束（文档）+ 硬拦截（hook）双层防护是 Carror OS 的最低配置
- 新增文档规则时必须自问：「违反这条规则能被自动检测到吗？」— 不能 → 补机制
- 决策链/行为约束 → 物理注入 AI 上下文 + 模式激活脚本 stdout 输出

**狗粮证据**: DG-09（DECISIONS.md 软约束无拦截）、DG-47/DG-62（决策链仅文档约束反复复发）、DG-58（物理门禁 > 文档约定）

## J. 修复回归（"修好 A，弄坏 B"）

> C 类（错误修复）只覆盖修复过程本身，不覆盖修复引入的二次破坏。本类别源自 4 次独立事故。

### J1：修补创伤 — 修复引入更危险的 regression

**检测信号**: 修复一个 bug 时改动 ≥3 个原本正常的文件；修复导致现有测试失败；修复引入新编译器警告/linter 错误；git diff --stat 范围远超预期
**反模式**: 聚焦目标 bug，不检查修复的二次副作用。AI 自证「修好了」但未独立重审。
**正确策略**:

- 每次修复后运行完整回归测试（不仅是目标文件的测试）
- `git diff --stat` 审查改动文件范围，非预期文件改动标记为异常
- AI 修复 → 独立 agent 重审（不自证「修好了」）
- 修复后在相邻功能上运行冒烟测试，确认未破坏

**狗粮证据**: DG-33（AI 修复引入更危险 regression）、DF-04（settings.json 自毁 — json.dump 修复导致全系统瘫痪）、DG-68（sed 空变量修复 = 文件全毁）、DG-80（自动替换破坏 6 个文件注释）

### J2：批量替换范围错误 — 机械文本操作破坏非目标内容

**检测信号**: 使用 `sed`/`str.replace`/正则替换修改代码时命中注释、字符串字面量或非活动代码路径
**反模式**: 对需要语义理解的文本使用纯机械操作（字符串替换/正则），不区分注释与代码
**正确策略**:

- 替换前跳过注释行/字符串字面量（只在结构感知上下文替换）
- 任何批量文本替换后运行 `bash -n` 逐文件语法验证
- 优先使用语言感知工具（autopep8/goimports/ast-grep）而非裸字符串替换
- 替换后 `git diff` 逐文件审查，确认每个命中位置是预期的

**狗粮证据**: DG-80（str.replace 命中注释里的 exit 2，6 个文件注释损毁）、kernel.md §禁止行为（禁止 json.load→str.replace→json.dump 管道）

## K. 跨 Agent 数据链污染（"子 Agent 说的 ≠ 事实"）

> A-H 类假设单一 Agent 工作。Carror OS 的多 Agent 模型中，跨 Agent 信息传递是重复出现的故障点。本类别源自 4+ 次独立事故。

### K1：Agent 轻信 — 子 Agent 输出未经交叉验证即接受

**检测信号**: 子 Agent 返回的数值/统计/声明直接写入方案/报告；子 Agent 证据不被物理验证；子 Agent 输出零过滤即提升为主 Agent 事实
**反模式**: 将委托视为验证 — 子 Agent 被告知「验证 X」，所以 X 被假定已验证。Agent 链中每一跳的置信度衰减被忽略。
**正确策略**:

- 所有子 Agent 输出视为 `[推断, 待确认]` 直到被主 Agent 独立验证
- 子 Agent 返回的任何数值/统计/路径：写入文件前必须 `wc -l`/`ls`/`diff` 物理确认
- `posttool-subagent-audit.sh` 跟踪跨 Agent 声明链，标记哪些已独立验证、哪些未验证
- 关键结论 → Oracle + Meta-Oracle 双签（单审已知漏 4 MAJOR）

**狗粮证据**: DG-44（Explore agent 返回「868+ hook 事件」实际未验证）、DG-63（Explore agent 幻读文件 23,671 行，实际 691 行 — 34x 偏差，第 3 次触发同模式）、DG-61/DG-67（Oracle 一审 + 二审 + Meta-Oracle 三轮发现完全不重叠）

### K2：审查衰减 — 多层审查后仍残留问题，但每层发现递减

**检测信号**: Oracle 一审发现 5 项 → 二审发现 2 项（与一审零重叠）→ Meta-Oracle 发现 2 项（仍不重叠）；修复后仍有审查发现新问题
**反模式**: 假设 N 层审查后零剩余问题。把「已审查」当作「无问题」。
**正确策略**:

- 保持剩余错误概率模型 — N 层之后发现的每个额外问题值得重新评估审查资源分配
- 重大方案必须 ≥2 轮 Oracle + 1 轮 Meta-Oracle（双签是最低配置）
- 多层审查后仍发现新问题 → 增加审查深度而非增加层数

**狗粮证据**: DG-61（三级审查链每轮发现完全不重叠）、DG-64（Oracle 多轮迭代审查 > 单轮）、DG-67（Meta-Oracle 发现 4 个 Oracle 漏掉的 MAJOR）

## L. 静默失败（"没报错 ≠ 正常运行"）

> A-H 类关注显式错误，但不覆盖「exit 0 但实际失败」的静默失败模式。本类别在 bash 密集环境中特别致命。

### L1：静默失败 — exit 0 但实际未成功

**检测信号**: 命令返回 0 但 stderr 含错误；输出为空但预期有内容；预期文件未被创建但无报错；管道中间环节失败被末端 exit 0 掩盖
**反模式**: 仅信任退出码，不验证输出内容或副作用。在 shell 脚本中，`false || exit 0` 掩盖前置语句失败。
**正确策略**:

- 关键路径使用 `set -o pipefail`（管道中任一命令失败 = 整体失败）
- 操作后验证副作用：`[ -f "$OUTPUT" ]` 确认文件存在，`[ -s "$OUTPUT" ]` 确认非空
- `$(grep -c pattern)` 返回 "0" 同时 exit 1 — 用 `VAR=$(grep -c pattern 2>/dev/null); VAR="${VAR:-0}"` 代替 `|| echo 0`
- bash 整数比较前确认变量是纯数字：`[[ "$VAL" =~ ^[0-9]+$ ]]` 先验证再比较
- 浮点数不用于 bash `[ ]` 整数比较（静默失败）— 用 `bc` 或 Python 替代

**狗粮证据**: DG-36（grep -c || echo 0 双输出 bug）、DG-54（harness.yaml 浮点导致 bash 整数比较静默失败）、DG-60（Python 多行输出 + bash $() + 整数比较 = 静默失败三角）、DG-32（非 canonicalized 路径导致 sed 替换静默失败）、kernel.md §禁止行为 已收录 2 条
