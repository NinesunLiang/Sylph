# 模式检测指南

> 由 lx-learner Phase 0 加载。定义 AI 如何从对话中识别可提取为 skill 的重复模式。

---

## 检测启发式

### 1. 重复指令检测

**信号**: 用户在 ≥3 个独立轮次中发出语义相似的指令。

**示例**:
```
轮次 3: "帮我审查这个 Dockerfile"
轮次 7: "检查一下 Dockerfile 的安全性"
轮次 12: "扫描 Dockerfile 有没有漏洞"
```

**提取策略**: 合并指令变体 → 识别核心动词（审查/检查/扫描）→ 提取目标类型（Dockerfile）→ 生成审查类 skill。

### 2. 重复修复检测

**信号**: AI 用相同方法修复了同类 bug ≥3 次。

**示例**:
```
轮次 5: 修复了 handleError 中的 nil pointer
轮次 9: 修复了 processRequest 中的 nil pointer
轮次 14: 修复了 validateInput 中的 nil pointer
```

**提取策略**: 识别修复模式（nil pointer guard）→ 提取修复模板 → 生成审查类 skill（专门检查此类问题）。

### 3. 重复分析检测

**信号**: 对同类工件进行了 ≥3 次分析/评估。

**示例**:
```
轮次 2: 分析 api/v1/handler.go 的复杂度
轮次 8: 分析 api/v2/handler.go 的复杂度
轮次 15: 分析 internal/service.go 的复杂度
```

### 4. 自定义流程检测

**信号**: 用户设计了 ≥3 步骤的协议，并在 ≥3 轮次中重复使用。

**示例**:
```
每次新 feature:
  1. 生成 prd.md
  2. 跑 lx-code-review
  3. 跑 lx-pre-commit
  4. 用户确认 → commit
```

---

## 误报排除

以下情况**不应**触发提取：

| 误报场景 | 原因 |
|---------|------|
| 用户说了 3 次"继续" | 通用推进词，非特定工作流 |
| AI 自己重复了 3 次相同操作 | AI 的重复不是用户需求 |
| 单次长操作被分割成多轮 | 不是真正的重复模式 |
| 项目特有的单次配置 | 不够通用，不值得提取为 skill |
| 对话寒暄/问候 | 非工作流 |

---

## 评分细则

| 因子 | 权重 | 说明 |
|------|------|------|
| 重复次数 | 3-8 分 | 3次=3分, 5次=5分, 10+次=8分 |
| 步骤可结构化 | 0-2 分 | 线性步骤=2分, 有分支=1分, 高度语境化=0分 |
| 通用性 | 0-2 分 | 跨项目适用=2分, 同类型项目=1分, 本项目特有=0分 |
| 用户明确表达需求 | +2 分 | 用户说过"每次都...""又要..."等信号词 |

**阈值**: ≥5 分 → 可提取。3-4 分 → 附带发现。<3 分 → 忽略。

---

## 信号词列表

用户表达重复/疲惫的信号词（提高置信度）:
- "又..." "再一次..." "还是..." "每次都要..."
- "有没有更快的方法..." "能不能自动..."
- "之前不是做过类似的吗..." "跟上次一样..."
- "again" "every time" "as before" "same as last time"
