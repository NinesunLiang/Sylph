# 工坊变迁录 — 圣器的兴与废

> v6.3.8 · Carror OS


> 📍 弧4：工程：工坊变迁录 | [⬅ 上篇](story-10.md) | [下篇 ➡](story-11.md)


防御再强也只是盾。要创造东西，还需要剑——那些能产出代码、测试、审查、验证的技能。

但工坊不是一成不变的。剑会断。盾会锈。有些圣器被供奉了半年，然后被熔掉——不是因为它坏了，是因为它做的事情被别的圣器吸收了。

这是工坊的变迁录。兴与废，都是因果。

---

## 活着的圣器

四大工坊，18 件活跃圣器：

### 代码审查工坊
**lx-code-review** — 语言无关的通用审查。8 大类别 39 条规则。他不只看"代码写得对不对"——更看"代码放在这个项目里对不对"。

### 测试工坊
**lx-test-gen** — 语言无关的测试生成器。自动检测项目语言（Go/TS/Python/Rust/Java），路由到对应模式。不生成"能通过的测试"——生成"能发现 bug 的测试"。

### 任务工坊：从轻到重的五级升级链
```
lx-todo (轻量) → lx-task-spec (中等) → lx-rpe (feature级) → lx-oma (PRD/项目级) → lx-stepwise (原子级)
```
升级不是惩罚——是承认当前工具的复杂度天花板。用小刀削苹果，用大斧砍树。

**lx-rpe** 是 feature 级重武器，四文档闭环：

| 文档 | 职责 | 一句话 |
|------|------|--------|
| **prd** | 需求澄清、意图澄清、未知项澄清 | 决定方向 |
| **research** | 调研可用资源、评估实施路径 | 决定路径 |
| **plan** | Phase 一级 + Step list 二级，串行/并发控制 | 管理进度 |
| **executor** | executor_log（执行记录 + error_log 防重走错路）+ checklist（验证通关点）+ report（退出无人模式的钥匙） | 执行与验收 |

**lx-oma** 是链上的项目级编排器：将超大型 PRD 按功能域 MECE 拆分为多个 Sub PRD，每个 Sub PRD 委托 lx-rpe 执行。OMA 四站流水线：hier（分层拆解）→ split（Feature 正交拆分）→ orch（管线编排）→ gov（治理同步）。

**lx-stepwise** 是链尾的原子刀：强制将高难度 bug 拆分为不可再分的原子步骤，每步独立验证，不可跳过。

### 运维工坊：门禁与监控
**lx-pre-commit** — 四步质量门：项目类型检测 → 编译 → 测试 → 代码审查。在代码进入 git 历史之前拦住有问题的一切。

**lx-pre-push** — 比 commit 更深、更慢、更彻底。commit message 规范校验 → 测试覆盖分析 → 安全扫描 → 最终判定。

**lx-status** — 4 面板健康可视化：Token 节省、任务通过率、拦截错误、升华知识点。他是飞轮的数据可视化端。

**lx-varlock** — 隐私脱敏代理。不是"请你不要看"——是"你根本看不到"。AI 告诉 varlock 需要什么操作，varlock 代理执行，AI 永远看不到明文。

## 消逝的圣器

工坊里有些架子空了。

曾经有 **lx-security-review** ——独立的代码安全审查官。它被熔掉了，不是因为它不够好——是因为 lx-code-review 学会了它的技能，把它吸收进了自己的 39 条规则里。

曾经有 **lx-golang-test** ——Go 语言的专项测试生成器。lx-test-gen 学会了语言检测和路由，变成了多语言通用的测试生成器。Go 专项成了一个路由项，不再需要独立的圣器。

曾经有 **lx-prd** ——产品需求生产工艺，从 Discovery 到 Polish 的完整六阶段。它被移出了工坊。不是因为没用——是因为 PRD 编写属于项目层面，不属于 AI 治理框架的管辖范围。

共 8 件圣器消逝了。没有一件是被"废弃"的。

`lx-security-review` 被 lx-code-review 吸收了——专职安全审查变成了 39 条规则中的一个类别。`lx-golang-test` 被 lx-test-gen 吸收了——专项测试变成了语言路由的一个分支。`lx-prd` 被移出工坊——不是因为没用，是因为 PRD 编写属于项目层面，不由 AI 治理框架管辖。

其余 5 件——`lx-react-review`（React 组件审计）、`lx-web-perf`（Web 性能诊断）、`lx-browser-verify`（视觉回归测试）、`lx-tdd-spec`（行为矩阵）、`lx-debug-spec`（结构调试）——或融入更通用的工具，或因其专业领域过于狭窄而被降级为参考模式。

每一件都经历过同一个命运：**被进化吸收了。**

一个 script 进化成 hook。一个 hook 进化成 skill。一个专项 skill 进化成通用 skill 的一个路由项。

工坊在变薄。但覆盖更密了。

## 新生代：正在淬火的圣器

**lx-dogfood** — 用 Carror OS 测试 Carror OS。启动独立 Claude Code 进程，收集 token 消耗、hook 触发率、completion-gate 拦截率。狗粮报告直接喂入飞轮。

**lx-root-cause-analysis** — 不修 bug，找根因。5 Why 追溯法 + 错误分类器 + 交叉验证。

**lx-skillify** — 扫描操作记录 → 提取可复用模式 → 生成 SKILL.md 候选草案。通过机制采纳门禁三问后才可激活。

**lx-learner** — 从会话历史中提取知识，输出候选知识点等 Oracle 审核。

**lx-sync** — 跨环境同步使者。通过 Source Mirror 协议将治理规则同步到各个目标。

**update-carror-os** — 唯一操作 Carror OS 自身的 skill。系统自噬的执行者。

## 工坊的铁律

永不为增长而增长。每件圣器的诞生背后，都有一道具体且重复的伤口。建造新圣器前，必须通过机制采纳门禁的三问。

工坊在变。但这条铁律不变。
