## Carror OS 是什么？

**Carror OS = AI 时代的开发者操作系统。**

它的目标不是替代 Claude Code、OpenCode 或其他 AI 工具，
而是把这些能力纳入一个更接近真实工程的方法里：

- 有约束
- 有门禁
- 有审查
- 有验收
- 有交付纪律

一句话说：

> **这不是一个更好的 Cursor，而是 AI 时代的 Unix。** [基于知识库 `CARROR-OS-LAUNCH-PLAN.md`]

---

## 它解决的不是“更快”，而是“更稳”

我最关心的不是 AI 多写了几百行代码，
而是它有没有在以下事情上少犯错：

### 1. 防泄露
我不想再把“别泄露”寄托在提示词和模型道德上。
所以 Carror OS 引入了 **DLP / 脱敏防线**。

核心思路很简单：

> **AI 连明文长什么样都不知道，怎么泄露？** [基于知识库 `CARROR-OS-LAUNCH-PLAN.md`]

知识库里这套机制已经落到 `privacy-gate.sh` 与 `lx-varlock` 双向混淆脱敏链路上。[基于知识库 `CHANGELOG.md`]

---

### 2. 防自证偏差
AI coding 里一个很大的坑是：

- 它自己写
- 它自己 review
- 它自己说通过

这不是审查，这是自我催眠。

所以 Carror OS 把 Code Review 变成 **A/B 盲审对抗**：

- 主 Agent 负责执行
- Fresh Context 的 Sub-agent 负责无情质检

> **打破 AI 的自我证实偏差，我们把 Code Review 变成了多智能体攻防对抗。** [基于知识库 `CARROR-OS-LAUNCH-PLAN.md`]

---

### 3. 防上下文失控
如果你长期用过 AI coding，你一定知道：

> 上下文一长，智力会稀释，结构会遗忘，错误会变得越来越“自信”。

所以 Carror OS 引入了 **Context Guard / Hard-Gate**：

- 长上下文时触发守卫
- 超阈值时直接熔断
- 在“甜点区”主动交接，让任务在最干净的时候切换会话

知识库里已经落地了：
- `context-guard` 物理级 Token 熔断 [基于知识库 `CHANGELOG.md`]
- `ctx% > 50%` 时的 sweet-spot handoff / 强制 `/compact` 接力 [基于知识库 `CHANGELOG.md`]

不是“尽量提醒”，而是**必要时强制刹车**。

---

### 4. 防无验收交付
AI 不是写完就算完。
真正贵的，是那些“看起来差不多、其实还没验收”的产物。

Carror OS 的工作流里，我特别强调：

- 文档先行
- 阶段验收
- 强证据
- 生产验收
- 更新文档后再推进下一步

知识库中的 `lx-rpe` 9 步主循环，本质上就是把“生成 → 实现 → debug → 安全复核 → 验收 → 文档更新”串成可执行链路。[基于知识库 `CHANGELOG.md`]

## 1. Carror OS 是什么？

Carror OS 是一套面向 AI coding 的开发者操作系统。

它不是单纯的提示词集合，也不是另一个“更会生成代码”的包装层，而是在 AI 之上增加一层：

- 治理
- 门禁
- 审查
- 验收
- 交付纪律

它的目标不是替代 Claude Code、OpenCode 或其他 AI 编程工具，而是让这些工具在真实工程环境中更可控、更可验证、更可交付。

---

## 2. 它和 Cursor、Claude Code、OpenCode、Devin 这类产品有什么不同？

最核心的区别不是“谁更会写代码”，而是“谁更重视治理”。

Carror OS 的重点不是单纯追求生成能力，而是处理以下问题：

- 防泄露

- 想实际在项目里低成本接入：**Base Edition**
- 想深入使用主动工作流和更完整体系：**Enhanced Edition**

通常不建议第一次就直接从最复杂层开始，除非你已经很熟悉 AI coding 工作流治理。

## 11. 什么是 A/B 盲审？

A/B 盲审的核心思路是：

- 不让同一个 Agent 自己写、自己审、自己给自己通过
- 让拥有 Fresh Context 的另一个角色进行独立审查

知识库里的表述是：

> **“打破 AI 的自我证实偏差，我们把 Code Review 变成了多智能体攻防对抗。”**
> [基于知识库 `CARROR-OS-LAUNCH-PLAN.md`]

这不是为了炫技，而是为了降低“自我催眠式通过”的概率。

---

## 12. 什么是 Context Guard / Hard-Gate？

这是 Carror OS 用来处理长上下文失控问题的一组机制。

知识库记录中包括：
- `context-guard` 的物理级 Token 熔断 [基于知识库 `CHANGELOG.md`]
- `ctx% > 50%` 时的 sweet-spot handoff / 强制重置接力 [基于知识库 `CHANGELOG.md`]

它们的核心目标是：
- 在 AI 还没彻底失控前提醒和切换
- 在高风险时直接阻断危险写入或继续执行

---

## 13. 它和 prompt 工程有什么关系？

Carror OS 不否认 prompt 的价值，但它的核心观点是：

> Prompt 不是安全边界，提示词也不是治理系统。

知识库中的发布思路明确提到：

> “AI 的安全下限不能靠 Prompt 讲道理，必须靠底层的物理 Hook。”
> [基于知识库 `CARROR-OS-LAUNCH-PLAN.md`]

所以你可以把它理解为：
- prompt 是沟通方式
- Carror OS 是治理方式
