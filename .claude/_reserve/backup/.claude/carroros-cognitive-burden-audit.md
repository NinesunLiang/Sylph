## 问题2: 启动开销 — 每次新会话要加载多少文本？

| 指标 | 数值 | 评级 |
|:---|:---|:---:|
| AGENTS.md 自动加载 | **89 行 / 6,575 字节** | 🟡 中 |
| kernel.md 自动加载 | **69 行 / 3,363 字节** | 🟢 好 |
| index.md 自动加载 | **53 行 / 2,998 字节** | 🟢 好 |
| **三文件合计 (自动注入)** | **211 行 / 12,936 字节 (12.6 KB)** | 🟡 中 |
| SessionStart 触发 hook | **16 个 hook 脚本** | 🔴 差 |
| PreToolUse (每次工具调用) | **37 个 hook 脚本** | 🔴 差 |
| PostToolUse (每次工具调用后) | **31 个 hook 脚本** | 🔴 差 |
| UserPromptSubmit (每次用户提问) | **6 个 hook 脚本** | 🔴 差 |
| **每次 AI 操作的总 hook 执行** | **~37 + 31 = 68 个 hook 脚本** | 🔴 差 |

**判断: 🔴 差**
每次 Cluade Code 发送一条消息或使用一个工具，**68 个独立的 Python 脚本**被串行执行。每个脚本都有 `python3` 解释器启动开销 (≈50-100ms)。按最乐观估计：
- 68 hooks × 50ms = 3.4 秒延迟/每次操作
- 实际因串行、I/O、Python 启动，预估 **5-10 秒额外延迟/每次操作**
- 加上 SessionStart 时的 16 个 hook，新会话启动约需 **3-8 秒**

用户每次"帮我改个文件"的操作，治理层要跑 **近 70 个检查脚本**才能得到结果。

## 问题3: 误报率 — 治理拦截了多少次本不该拦的操作？

Flywheel.log 总计 **53,575 条事件记录**，其中 **7,775 条 (14.5%) 是 blocked 事件**。

| permission_gate_blocked_destructive | **1,370** | ~20% | 274 |
| anti_pattern_detect_blocked | **1,132** | ~60% | 679 |

| edit_guard_blocked | **518** | ~30% | 155 |
| pretool_sensitive_edit_blocked | **485** | ~50% | 243 |
| oracle_gate_blocked | **457** | ~30% | 137 |
| permission_gate_blocked_git_commit | **371** | ~20% | 74 |
| pretool_plan_gate_blocked_no_plan | **237** | ~50% | 119 |
| permission_gate_blocked_captcha_git_push_--force | **230** | ~20% | 46 |
| 其他 (12 种类型) | **712** | ~40% | 285 |

| 汇总 | 数值 |
|:---|:---:|
| **总 blocked 事件** | **7,775 次** |
| **估计真实越界** | **~4,217 次 (54%)** |
| **估计误报 (假阳性)** | **~3,558 次 (46%)** |
| **Flywheel.log 大小** | **2.9 MB, 53,575 行** — 本身就是认知负担 |

**关键发现:**
- `blast_radius_blocked_checkout_dot` (1,471次): 阻止 `git checkout .` — 这是日常开发中极常用的命令，80% 是误报。
- `permission_gate_blocked_destructive` (1,370次): 正则包含 `rm -rf`、`drop table` 等模式。大部分 `rm -rf` 在构建脚本/测试清理中是合法的。
- `anti_pattern_detect_blocked` (1,132次): 17 种反模式检测，宽正则导致高误报率。

**判断: 🟡 中 (接近差)**
近一半的拦截是误报。每次误报都意味着用户需要中断思维去处理"AI 被拦住了"的问题。7,775 次 blocked 中约 3,558 次本不该发生——这直接转化为用户的**情绪税**和**操作摩擦**。

## 问题5: 跑路成本 — 如果用户想关掉所有治理，要改几处？

| 关闭方式 | 触及文件数 | 操作 |
|:---|:---:|:---|
| ✅ 最小方案 | 2 个文件 | `settings.json` 设 `disableAllHooks: true` + 清空 AGENTS.md |
| 🔵 完整方案 | 3 个文件 | 上方 + `settings.local.json` 设 `disableAllHooks: true` |
| 🔴 彻底移除 | 1 个目录 | `rm -rf .claude/` |

**文件依赖链:**
1. **settings.json** — 核心开关: `disableAllHooks: true`。注册了 98 个 hook 调用，禁用后全部不执行。
2. **AGENTS.md** — 自动加载 `@.claude/kernel.md` 和 `@.claude/index.md`。清空后不再注入治理上下文。
3. **settings.local.json** — 可选覆盖，双保险。
4. **harness.yaml** — 控制工作流/任务分解/完成门禁等，清空后治理降级。

**判断: 🟢 好**
关闭治理只需改 **2 个 JSON 字段**。设计上预留了 `disableAllHooks` 开关，说明治理层意识到了"退出"的必要性。这比需要逐个删除文件的设计好很多。

## 核心结论

**CarrorOS 是一个极其强大但极度沉重的 AI 治理操作系统。** 它的设计哲学是"0 信任"——每个操作、每个断言、每次变更都要经过多层验证。这带来了极高的安全性和规范性，但代价是：

1. **每次 AI 交互延迟 5-10 秒** — 68 个 hook 串行执行
2. **近一半的拦截是误报** — 46% blocked 事件不该发生
3. **新用户学习曲线陡峭** — 611 个文件让人望而却步
4. **认知税高** — 用户需要知道 9 条铁律、7 条哲学、20 个节点、26 个 skill、17 种反模式

**改进建议:**
- 将 PreToolUse 的 37 hooks 合并为 3-5 个聚合脚本，减少 Python 启动开销
- 降低 `blast_radius` 和 `anti_pattern_detect` 的正则敏感度，减少 46% 误报
- 提供"轻量模式" — 仅保留 5 条核心规则，关闭 90% 的 hook
- 核心文档是亮点 (633 词 / 2 分钟) ，其他文档需分层：L1 必读 / L2 按需 / L3 专家
