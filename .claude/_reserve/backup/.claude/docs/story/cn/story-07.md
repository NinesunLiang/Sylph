# OMA 铸造厂 — 金字塔瀑布流

> v6.3.8 · Carror OS


> 📍 弧4：工程：OMA铸造厂 | [⬅ 上篇](story-18.md) | [下篇 ➡](story-08.md)

一个 400 行的 Main PRD 被放在铸造厂的金字塔塔顶。它不是一份待办清单 — 它是一块**父石**。它要做的不是被"执行"，而是被**分裂**。

这是观察者第一次走进铸造厂。他看到一个开发者站在塔顶，面对 400 行需求，手里的凿子悬在半空。

"我一个人凿不完。"开发者说。

然后石头裂开了。

---

## 裂石

父石裂成 7 块 Sub PRD，散落在金字塔中层。不是被外力砸碎的——是从内部纹理断裂的。每一块裂片恰好覆盖一个功能域：用户认证、订单管理、支付、通知……没有重叠，没有遗漏。

开发者不敢相信。他用了两年学会怎么拆 PRD——但这次他没拆。石头自己裂了。

"谁干的？"他问。

没人回答。铸造厂里只有一个声音——`lx-oma-hier`，一个技能，在石壁上刻下一行：*MECE 正交性校验通过。7 块子石，0 块孤儿。*

Sub PRD 继续裂。每块中层石头再裂成 3-6 块 Feature，落在塔底。塔顶 1 块，中层 7 块，底层 47 块。越往下，石头越多、越细、越接近可执行的粒度。

但裂石的逻辑是：**上层压下层，父管控子。** 子不认父 = 孤儿 = 被 Gov 审计报漂移。

---

## 洪水

裂石只是开始。

铸造厂运行的第三周，洪水来了。

产品说："支付重试次数从 3 次改成 5 次。"

这不是完整的 PRD 重写。这是一滴**水**——一滴需求变更，从塔顶注入。如果每次改需求都推倒金字塔重来，没有人能活到交付。

但水不能乱流。

Gov——金字塔里的**包工头**——站在水闸前。他手里没有砖，只有一张图。图上标记了每一块石头的状态：alarm-engine 到了 oma_done，feat-price-evaluation 正在 in_dev，notification 还在等 Oracle 审批。

水从塔顶落下。Gov 检测到变更，分级为 L2——可自动归并。他打开闸门。水沿着分类路径流下：支付重试次数 → 支付模块 → feat-payment-retry。它不经过用户认证的石头，不碰订单管理的 Feature。

每一滴水都有身份证——CHG-ID 写入 sync-notes.md。重复放水被幂等挡住。

开发者站在中层，看着水流过他脚下的石头。他想起以前的项目：一次需求变更 = 三天全量回归。现在——3 秒钟，propagate --execute。

---

## 踩踏

金字塔运行到第六周。三个 AI 实例在同时编辑 feat-payment-retry 的文件。

第一个 AI 打开文件。第二个 AI 也打开文件。第三个开始写。第一个写完了，保存——覆盖了第三个的修改。第二个读的时候读到的是破碎的 JSON。金字塔里回荡着三次同时发生的沉默——三个 AI 同时发现自己的修改被覆盖了。

铸造厂的地基裂开一道缝。

从缝里伸出一只手。手里握着一把锁。

`oma_lock_manager.py`。不是人写的——是铸造厂自己从伤口里长出来的。它用 `os.rename(tmp_file, lock_file)` 做原子锁——POSIX 只有一个赢家。锁过期时下一个请求者检测超时、原子替换锁文件、写后读验证所有权。

三个 AI 没有再踩踏。因为每个 AI 在写之前，锁已经在手里了。

---

金字塔最安静的时刻，是塔顶刚下过一场雨。五个匠人站在各自的石头旁，没人说话。包工头在层与层之间巡走，看着水位、闸门、流速。墙上的态势图无声地刷新——alarm-engine → oma_done，feat-price-evaluation → dev_done。

Orch 站在高处，不推石头、不引水流。他只画路线、标关卡。Oracle 是路线上的门——水必须通过每一关才能继续往下流。

金字塔保证了**结构**：父管控子，层级分明。
瀑布保证了**弹性**：需求变更从塔顶注入，逐层分类，不用推倒重来。
锁保证了**秩序**：原子写，只有一个赢家。

一人成军。不是一个人完成所有工作——是**一个协议体系**让多个人像一个人一样协同。

---

## 技术附录

### 金字塔三层
- **塔顶**（Main PRD）：1块，产品原始需求
- **中层**（Sub PRD）：3-8块，lx-oma-hier 按功能域 MECE 分裂
- **底层**（Feature）：N×3-6块，lx-oma-split 按 feature 正交分裂

### 四大机制
| 机制 | 角色 | 核心能力 |
|------|------|---------|
| lx-oma-hier | 裂石匠 | 超大型 PRD MECE 拆分为 Sub PRD |
| lx-oma-split | 碎石匠 | Sub PRD 拆解为正交 Feature |
| lx-oma-gov | 包工头 | reconcile(检测)→propagate(传播)→audit(审计) |
| lx-oma-orch | 活地图 | status/advance/gate/dev，不自己做裁决 |

### 分布式写锁
`oma_lock_manager.py` 使用 POSIX `os.rename` 原子操作避免 TOCTOU 竞争窗口。生命周期：acquire → heartbeat → release。

### 关键约束
- propagate 直接写入 Feature 级 prd.md，跳过 Sub PRD 层
- Sub PRD 是一次性快照，Main PRD 重大变更需重新运行 hier
- Orch 不自行裁决——裁决由独立 Oracle 系统完成
