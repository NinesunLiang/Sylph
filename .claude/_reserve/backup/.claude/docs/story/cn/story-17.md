## 序章：盲眼巨人

十六篇故事，Carror OS 赢了十六场仗。

门禁骑士拦下过 `rm -rf /` ——「此路不通」。证据裁判庭识破过「应该没问题了」——「你拿什么证明？」。记忆神殿在 R41 轮转 bug 吃掉 99% 数据后，默默把 off-by-one 刻进了石壁。上下文守望者在洪水漫过 80% 水位线时挥下闸刀——「够了，压缩」。

每一场仗都赢了。每一篇故事都写了。但有一个真相，十六篇里只字未提：

**Carror OS 分不清人和妖。**

不是「不能执行」——权限门禁连 sudo 都能拦。是「不能辨认」。当一个 AI agent 说「找到 calculate_score 的定义」，门禁不知道它跳到了函数体还是跳进了一段长得像函数的注释。当一个 AI agent 改完代码说「修好了」，裁判庭不知道这行代码是一个干净的修复还是一颗定时炸弹——类型不匹配、签名断裂、引用悬空。

它只能看行为。它看不见本质。

就像唐僧，肉眼凡胎。白骨精变作少女、老妇、老翁——每一个形态都合情合理。唐僧只见皮囊，不见白骨。沙僧沉默，八戒馋嘴。只有孙悟空——那对在八卦炉里炼出来的火眼金睛——一眼洞穿：**妖魔鬼怪，一眼便知**。

Carror OS 没有火眼金睛。

它拦得住明刀明枪。拦不住画皮。

## 第一章：画皮

妖魔从不大张旗鼓。它们穿衣服。

第一层画皮叫「看起来正常」。一个函数调用 `user.save()`——grep 找得到它，正则匹配得到它，diff 显示它没变。但 LSP 知道 `.save()` 已经被废弃，新的叫 `.persist()`。那个旧调用躺在代码里，没有报错，没有告警，像一条冬眠的蛇。grep 摸不到它。等到运行时崩了，门禁骑士一脸茫然——不是没拦，是看不见。

第二层画皮叫「签名无声断裂」。`def process(data)` 改成了 `def process(data, timeout=30)`——两个参数，一个默认值。改起来太简单了。diff 只显示一行。CI 甚至不报错——默认值兜住了。但 47 个调用点里，12 个需要显式传 timeout——数据量级不一样，默认 30 秒根本不够。grep 搜 `process(` 只告诉你「这东西有人用」，不告诉你「谁用了旧签名」。妖从你眼前走过，你看见了——但没认出来。

第三层画皮最阴毒。它不藏。它光明正大地站在你面前——**注释里的 User**。grep `User`，返回 2,347 行。2,103 行是注释、字符串、文档。真正的类定义、函数调用，被埋在这座垃圾山里。你花了 40 分钟清洗。妖就在你眼前晃了两千次——你一次都没认出来。

这是盲眼巨人真正的困境：**能拦住的都是明妖。拦不住的都穿了衣服。**

## 第二章：八卦炉

孙悟空怎么有的火眼金睛？

被扔进太上老君的八卦炉，炼了七七四十九天。巽位有风无火，他躲在下面没被烧死。但烟熏火燎，双目灼痛，猴毛焦枯——从炉子里跳出来的时候，眼睛里带着三昧真火的余烬。

从此，**一切幻术在他眼前形同虚设**。

白骨精的少女皮囊——看穿，骷髅。老妇的慈祥面目——看穿，同一具白骨。老翁的垂垂老矣——看穿，还是那具白骨。幻术层层叠叠，火眼金睛层层穿透。不是「猜测」，是「看见」。50 毫秒。一眼。

Carror OS 的八卦炉，不是火。是沉默。

每一次 agent 用 grep 摸黑撞墙。每一次改了代码跑了 15 分钟 CI 才炸。每一次「看起来修好了」上线后用户报 bug。每一次盲人修表式的试错——都是在炉子里多炼了一天。

但炼不出火眼金睛。因为 grep 不是眼睛。正则不是眼睛。AST 解析器不是眼睛——你给每种语言手写一个解析器，就成了一个瞎子背另一个瞎子。

你需要一个东西，能直接问代码：

> 「你是谁？」
> 「你从哪里来？」
> 「谁在调用你？」
> 「你身上哪一行断了？」

这串问题，有一个协议已经回答了十年。

---

## 第三章：LSP——火眼金睛的协议

2016 年，Microsoft 造了 Language Server Protocol。

不是为了 AI agent。是为 VS Code 的红波浪线。

哲学简单到残忍：**每种语言造一个懂它的服务器。每一个编辑器用同一个协议去问。** pyright 站在 Python 那边，gopls 站在 Go 那边，typescript-language-server 站在 TypeScript 那边。你不需要知道 Python 的 AST 怎么遍历——你只需要发给 pyright：

```
「这个符号的定义在哪？」→ 50ms 返回 file:line
「谁在引用这个符号？」→ 50ms 返回 47 个调用点
「改完之后有没有类型错误？」→ 50ms 返回 diagnostics 全列表
```

这不是「更快的 grep」。这是降维打击。

grep 看文本——「这行有 User」。LSP 看语义——「这个 User 是类的实例化、是类型注解、还是注释里的字符串」。

换作孙悟空的视角：grep 是唐僧的眼——看见一个女子走过来。LSP 是火眼金睛——**穿透皮囊，直透头颅，看到白骨**。

妖魔鬼怪的幻术有三层：
- 第一层皮囊：代码语法合法，看起来没毛病
- 第二层血肉：变量名、函数名都在，引用关系看起来对
- 第三层白骨：类型不匹配、签名断裂、废弃调用——藏在血肉下面

grep 停在皮囊。正则停在血肉。LSP 穿透白骨。

**妖魔鬼怪，一眼便知。**

## 第四章：睁眼——不是造眼，是借眼

Carror OS 不需要造 LSP。它只需要睁开眼睛。

2025 年 12 月，Claude Code v2.0.74 原生接入 LSP。`goToDefinition`、`findReferences`、`getDiagnostics`——三件法器都在工具集里，只等用户装插件。OpenCode 走得更早，40+ 语言服务器内置，自动检测安装，像出生就睁着眼睛。Codex CLI 走得最慢，但 Serena MCP 桥接补上了最后一块拼图。

三个平台的三种路径：

| 平台 | 眼睛怎么睁 |
|------|----------|
| Claude Code | `/plugin install pyright-lsp`——法器在手，自己装 |
| OpenCode | `{"lsp.enabled": true}`——天生的，只是要喊一声「开」 |
| Codex CLI | `pip install serena-agent`——打个桥，借双眼睛 |

不是一种方式。是一个结果：**AI 不再用 grep 摸黑。LSP 50 毫秒告诉它符号在哪、类型对不对、改了有多少调用点要同步。**

Carror OS 要做的，不是替它们造眼睛。是**确保每一个进殿的 agent，都不会忘记自己可以睁眼。**

## 第五章：探针——教它睁眼的那个人

`ecosystem-probe.sh` 是最早醒来的。它不在柱子之列——它在入口。

每次 SessionStart，七个柱子还在沉睡，探针已经伸出触手。它做的事不是布防，是**在 agent 耳边轻声问一句话**：

```
你是 Claude Code？→ 你有 LSP 插件吗？pyright 在不在？
你是 OpenCode？→ `lsp.enabled` 写了吗？
你是 Codex CLI？→ Serena 到了没？
```

没有？探针不拦你。

它只是在你进殿之前，递过来一副眼镜：

> 「[lsp-suggest] 🔍 语义引擎未配置——AI 将只能用 grep 理解代码。
>   这就像让孙悟空闭着眼睛打白骨精——不是不能打，是用头撞。
>   安装指南：docs/guides/cn/lsp-setup.md」

这是哲学 #5 —— 以人为本。不替用户做决定。但确保用户知道：**你可以睁眼，睁眼之后世界不一样。**

## 第六章：开眼仪式——火眼金睛的剑诀

探针教你睁眼。但睁眼之后还有一个问题：**你眼睁着，但不看。**

一个 agent 改了 `process(data)` → `process(data, timeout=30)`。LSP 已经标出了 47 个受影响的调用点。但 agent 没看 diagnostics——它直接 commit 了。

改了不看，跟没睁眼一样。

`pre-edit-lsp-check.sh` 是开眼仪式的第三道剑诀。它不在你动手时拦你——它在**你动手之前**，轻声问你一句：

> 「改这行之前——你看了 LSP diagnostics 吗？」

不阻断。永远 exit 0。但它会让你知道：

```
[lsp-gate] 🔍 编辑前 LSP 检查: /src/auth.py (python)
[lsp-gate] 提示: 请先用 LSP tools 检查 diagnostics，确认无错误后再编辑
[lsp-gate] Claude Code: 使用 LSP 工具的 getDiagnostics
```

它不夺权。它只是把火眼金睛塞到你手里：
> 「剑在你手。眼在你手。看不看随你。但不看的话——白骨精在暗处笑的是你，不是我。」

## 第七章：三面镜——一个凝视

Windows。macOS。Linux。

ecosystem-probe 不问操作系统。它问的是**你是谁**——然后告诉你**去哪里睁眼**。

三个平台，一根触手，一个终点：

| 平台 | 眼睛怎么睁 | 一根触手的距离 |
|------|----------|--------------|
| Claude Code | 官方插件市场 | `/plugin install pyright-lsp` |
| OpenCode | 内置觉醒 | `opencode.json: {"lsp": true}` |
| Codex CLI | Serena 桥接 | `pip install serena-agent` |

这不是平台兼容。这是 Carror OS 对「感官缺失」的系统性补救。

每个 agent 进殿，探针在入口轻声问——「你有眼睛吗？」

没有？给你一副。不收费。不强制。只是递过去。

## 第八章：幻术全破

现在，回到那个 `grep User` 返回 2,347 行的审计现场。

有了火眼金睛之后——同一个问题只花 50 毫秒。

不是「grep 更快了」。是**不再 grep 了**。

`findReferences("User")` → 只在语义层面返回真正引用 `User` 类的地方。注释里的不算。字符串里的不算。文档里的不算。**幻术全破。**

白骨精变少女——`user.save()` 看起来完好无损。火眼金睛：**废弃调用。请改用 `.persist()`。**

白骨精变老妇——`def process(data, timeout=30)` 看起来合情合理。火眼金睛：**12 个调用点缺少显式 timeout。类型告警。**

白骨精变老翁——改了 `calculate_score` 的返回类型从 `int` 到 `float`。diff 只显示一行。火眼金睛：**23 个赋值点类型不匹配。在 CI 挂之前，在 PR 合之前，在用户报 bug 之前——一眼看穿。**

这不是「多了一个检查工具」。这是一双眼睛。

门禁拦得住 `rm -rf`，拦不住幻术。裁判识得破「应该没问题了」，识不破画皮。七柱管得住行为，管不住认知。**管认知的，只能是感官。**

## 第九章：因果——眼睛是烧出来的

十六篇故事讲到这里，Carror OS 的「因果」哲学已经深到骨子里：

> 不设计环境，只生长环境。

LSP 的眼睛怎么长出来的？

> agent grep 出两千行垃圾 → 「不对，文本匹配不是理解代码」  
> 「VS Code 怎么知道类型？」→ 「因为有 LSP」  
> 「Claude Code v2.0.74 原生支持 LSP 工具」→ 「不需要造轮子，只需要接入」  
> 「但用户不知道去哪装」→ 「让探针在入口提醒」  
> 「装了 LSP 但 agent 改代码前不看 diagnostics」→ 「让 pre-edit-lsp-check 轻声提醒」  

每一步都是真实的火。每一步的解法都不是凭空设计——是环境里已有的东西，等着一根触手去够它。

**门禁从真实事故中长出。眼睛从真实盲区中烧出来。**

八卦炉里的火，不是什么高深理论。是每一次 agent 用 grep 摸黑撞墙的沉默。是改了代码跑了 15 分钟 CI 才炸的等待。是「看起来修好了」上线后用户报 bug 的尴尬。

烧得够久。眼睛就睁开了。

## 终章：开眼之后

七柱依旧。铁律依旧。法官依旧。圣殿没变。

变了的是每一个住在殿里的人。

门禁骑士拦住 `rm -rf` 时，**他现在知道这行删除会让多少引用断裂**——LSP 在动手之前就把调用图铺在他眼前。不再是「此路不通」——是「此路通，但前面有 47 座桥会塌」。

Ghost 探索未知代码时，**不再用 grep 在黑水里摸**。`goToDefinition` → `findReferences`——一条链拖出整张网。50 毫秒。比 grep 快不是重点——**比 grep 准，才是降维打击。**

Goal 执行具体目标时，**每改一行，眼睛先到**。类型对不对。签名有没有断裂。废弃的 API 还有没有残留。在 commit 之前。在 CI 挂之前。在用户发现之前。

证据裁判庭审查「VERIFIED」声明时，多了一条证据链——**LSP diagnostics 全绿**。不是「测试没写所以不出错」，是「语义引擎确认类型系统无断裂」。

这不是多了一个工具。这是**从盲到明的质变**。

火眼金睛不是用来打的。是用来看的。而看穿幻术——**比拦住明刀明枪，难十倍。重要百倍。**

> *妖魔鬼怪，一眼便知。*

*圣殿的第七扇窗，开了。火眼金睛，归位。*
