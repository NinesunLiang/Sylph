# pretool-gate.py 模块化 ROI 评估

> 分析日期：2026-07-12
> 目标：拆 1 个 571 行文件 → 8 个模块文件 + 1 个调度器

---

## 一、当前状态（基线）

| 指标 | 实测值 |
|------|--------|
| **文件大小** | 571 行 / 20KB |
| **函数数** | 28 个（含 8 个 gate 检查 + 20 个工具函数） |
| **冷启动耗时** | ~0.02s (3 次平均) |
| **全门禁通过** | ~0.02s (3 次平均) |
| **内存占用** | ~12.9 MB RSS |
| **架构模式** | `GATES = [(name, fn), ...]` 顺序执行 + short-circuit |

### 运行时路径分析

```
工具调用 → 读 stdin(0.001s) → 清 stale lock(0.002s) → for gate in GATES:
  gate_1 sensitive-edit: Read 工具→跳过(0.001s)
  gate_2 fallback: 无 token→跳过(0.002s)
  gate_3 action-gate: 无 command→跳过(0.001s)
  gate_4 plan-gate: Read→跳过(0.002s)
  gate_5 edit-scope: Read→跳过(0.001s)
  gate_6 verify-gate: Read→跳过(0.002s)
  gate_7 oracle-gate: L1→跳过(0.002s)
  gate_8 doc-quality: Read→跳过(0.002s)
→ _ok("ALLOW") → 输出(0.001s)
总耗时 ≈ 0.015-0.025s
```

**关键洞察：** 最慢的 gate 是 gate_4 (plan-gate, 读 token.json + 文件系统) 和 gate_8 (doc-quality, 正则匹配)。但即使最坏情况也只 0.025s —— 对于每次工具调用（通常 0.5-5s）来说，hook 开销占比 **< 5%**，完全可以忽略。

## 二、模块化方案

### 方案结构
```
hooks/
├── pretool-gate.py          (调度器, ~50 行)
├── pretool-gates/
│   ├── __init__.py
│   ├── sensitive_edit.py    (gate 1)
│   ├── fallback.py          (gate 2)
│   ├── action_gate.py       (gate 3)
│   ├── plan_gate.py         (gate 4)
│   ├── edit_scope.py        (gate 5)
│   ├── verify_gate.py       (gate 6)
│   ├── oracle_gate.py       (gate 7)
│   ├── document_quality.py  (gate 8)
│   └── harness_lib.py       (共享工具函数)
├── carroros_hooklib.py
├── hook-launcher.sh
└── statusline-command.sh
```

## 三、ROI 计算

### 成本

| 项目 | 估算 |
|------|------|
| **开发时间** | ~30-45 分钟（拆 8 个文件 + 测试 + 验证） |
| **文件数增加** | 1 → 10 个（+900%） |
| **维护面增大** | 所有 gate 逻辑需跨文件追踪 |
| **Base 模型认知负担** | ⬆️ 原来 1 个文件读完即懂，现在需了解 10 个文件的组织 |
| **import 开销** | Python 每多一个模块多一次 `stat()` 系统调用，冷启动增加 ~0.01s |
| **部署风险** | OC plugin 同步、CC settings.json、路径引用全部要更新 |

### 收益

| 项目 | 效果 |
|------|------|
| **运行时性能提升** | **≈ 0%** — 当前已 0.02s，瓶颈不在文件组织而在 Python 启动本身 |
| **可维护性** | **≈ 0%** — 当前 `GATES` 列表已清晰表达门禁管线，拆开反而丢失全局视图 |
| **选择性加载** | **微弱** — CC 每次工具调用都启动新 Python 进程，无法利用选择性 import |
| **代码复用** | **≈ 0%** — 共享工具函数已在 `carroros_hooklib.py` 中 |
| **测试独立性** | **轻微** — 可单独测试每个 gate，但当前 gate 之间无耦合，已可独立测试 |

### ROI 评分

| 维度 | 评分(0-10) | 说明 |
|------|-----------|------|
| 运行时效能 | **0/10** | 0.02s 已接近理论下限（Python 进程启动时间），模块化不改善 |
| 契合 CarrorOS 哲学 | **2/10** | 哲学 #7「少即是多」反对增加文件数；#6「增益原则」收益 < 成本 |
| Base 模型适配 | **1/10** | 当前 1 个 571 行文件对 Base 模型更友好（一次读完即理解全部门禁） |
| 维护成本 | **3/10** | 增加 9 个文件后，跨模块追踪门禁管线更困难 |
| 可扩展性 | **5/10** | 如果未来要支持不同 hook-launcher 选择性加载 gate（如 OC 只加载 action-gate），才可能体现价值 |

**综合 ROI：2/10 — 不建议执行。**

## 四、与 CarrorOS 哲学的契合度

### ✅ 当前设计符合的哲学

1. **哲学 #7 少即是多**（优先级最高）
   - 1 个文件 vs 10 个文件
   - 571 行既是完整门禁管线的单一真相源
   - Base 模型一次读完即理解全部行为

2. **哲学 #6 增益原则**
   - 改动成本 ≈ 45 分钟
   - 收益 ≈ 0（性能不变，维护性不升）
   - 净 ROI 负数

3. **哲学 #2 零信任执行**
   - fail-open 设计：gate 异常时放行不阻断（L546-559）
   - 每步独立验证：每个 gate 不依赖其他 gate 的状态

### ❌ 模块化设计会违反的哲学

1. **哲学 #7 少即是多** — 10 个文件远多于 1 个
2. **哲学 #6 增益原则** — 复杂度增加但无实质收益
3. **哲学 #5 人本决策** — 增加理解成本，没有带来对人更好的结果

---

## 五、当前设计的隐性优势

1. **短路优势** — Read 工具调用只跑 gate_1 + gate_7（约 0.005s），Write 才跑全链
2. **自包含** — 不依赖 pretool-gates/ 包的存在性，不会因缺少模块而崩溃
3. **与 hook-launcher.sh 的配合** — 直接 `hook-launcher.sh pretool-gate.py`，无需预判加载哪些模块
4. **审计全量** — 一次 `_append_audit` 调用即可记录门禁决策，跨文件需要共享 audit 实例

---

## 六、结论

> **不拆分。保持 pretool-gate.py 单文件架构。**
>
> 当前设计已经是"单一入口 + 8 个命名函数 + 顺序 GATES 列表"的模块化风格——只是模块在同一文件内（模块化 in-code，而非 in-filesystem）。对于 Base 模型，这反而是最优形态：
> - 一次读完 = 理解全部治理规则
> - 0.02s 的 hook 开销可忽略
> - 12.9 MB 的内存开销极小
> - 文件级模块化带来的额外认知成本 > 收益
>
> **推荐做法**：保持当前结构，后续如需新增 gate，按 `_check_<name>` 模式在 `GATES` 列表中追加即可。
