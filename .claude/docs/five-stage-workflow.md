# 五阶段决策-执行-验收工作流 v3.1

> 从人类设定到验收闭环的完整管道。AI 自主执行，人类在三个关键节点把关。
> v3.0 新增：方案修订回门、checkpoint 设计判断、ROI 对比裁决、报告质量审查。
> v3.1 新增：Compact 抗性 + 无人值守协议 + 状态恢复 + Gate 超时。

──────────────────────
## 总览

```
Stage 0: 人类设定（gaol 0）
    │  PRD + 铁律 + 材料 + 路径约束
    ▼
Stage 1: AI 出方案
    │  哲学→铁律→现状→ROI → 方案文档
    ▼
【Gate 1: 人类审方案】
    │  人类: 调整/拒绝/打回/补充
    │  通过 → 人类启动 Oracle
    ▼
  Oracle 静态审查 → REVISE? → AI 修订 → 回 Gate 1 人类复核
    │              → PASS    → 人类启动 Meta-Oracle
    ▼
  Meta-Oracle 对抗审查（上限3次）
    │  ACCEPT → 进入执行
    │  REJECT → 打回 Stage 1
    ▼
Stage 2: AI 执行
    │  Token 初始化（串行）→ 各区域并发执行
    │  每 checkpoint → 烟测 + 设计约束 diff → 失败暂停等人类
    ▼
【Gate 2: 人类审执行报告】
    │  人类: 审执行报告
    │  通过 → 人类启动双法官验收
    ▼
  Oracle + Meta-Oracle 验收（上限3次）
    │  ACCEPT → 进入 Stage 3
    │  REJECT → 打回 Stage 2
    ▼
Stage 3: 最终验收
    │  生成验收报告（含预估 vs 实际 ROI 对比）
    │  偏差 >30% → 触发人类裁决
    ▼
【Gate 3: 人类读验收报告 → 闭环】
```

──────────────────────
## Stage 0：人类设定（gaol 0）

**人类定义边界。** AI 不猜需求，人类画好框再开始。

输入物（必须）：
- 目标描述（做什么、为什么）
- PRD / 需求文档（或结构化约束清单）
- 铁律覆盖（哪些铁律特别相关）
- 路径约束（技术栈、目录结构、禁止触及的区域）
- 验收标准（怎么判断成功）

AI 职责：如果发现模糊点，在 Stage 1 方案中结构化提问——**不允许假设 PRD 完备**。

### 运行模式选择

Stage 0 人类必须指明模式：

| 模式 | Gate 1 | Gate 2 | Gate 3 | Compact 恢复 | 适用场景 |
|:---|:---|:---|:---|:---|:---|
| **有人值守** | 人类手动启动 Oracle | 人类手动启动验收 | 人类闭环 | Stage Anchor 自动注入 | 人在旁边，实时决策 |
| **无人值守** | Oracle 自动启动，ACCEPT=推进，REVISE/REJECT=BLOCKED | 自动验收，ACCEPT=归档，REJECT=BLOCKED | 跳过——报告归档 | Stage Anchor + 幂等恢复 | 人不在，批量任务 |
| **混合** | 有人 Gate 1 → 无人执行 → 有人 Gate 2/3 | — | — | 按当前阶段自动切换 | 先审方案再放行无人执行 |

> 无人值守模式下，人类已在 Stage 0 给了充分材料 → 视为「Gate 1 已预授权」。
> Oracle 替代人类做出推进/阻塞裁决。Meta-Oracle 仍上限 3 次。

──────────────────────
## Stage 1：AI 出方案

```
输入: Stage 0 全部材料
处理: 哲学优先级扫描 → 铁律约束检查 → 现状分析 → ROI 预估
输出: 方案文档
```

方案文档强制包含：
1. **约束矩阵**：铁律覆盖 + 技术约束 + 范围边界（结构化，可被 checkpoint 消费）
2. **任务分级**：LLM 初判 L1/L2/L3 → 附判定理由
3. **ROI 预估**：预期收益 + 预估成本（时间/复杂度/风险）
4. **并行区域识别**：哪些子任务互不依赖可并发
5. **模糊点清单**：PRD 中不明确需要人类确认的点

──────────────────────
## Gate 1：人类审查 → 双法官审核

**人类先审，通过后才启动双法官。** 人类是第一道真实对抗——发现了 AI 法官不会发现的战略错误。

### 人类审查清单
- 方案是否符合意图？
- 任务分级是否正确？
- ROI 预估是否合理？
- 范围边界是否恰当？
- 模糊点是否已确认？

人类动作：**调整 / 拒绝 / 打回 / 补充**，通过后人类手动启动 Oracle。

### Oracle 静态审查
审查维度：简洁性、安全性、可验证性、一致性、铁律合规。

裁决：
- **PASS** → 人类启动 Meta-Oracle
- **REVISE** → AI 修订方案 → **回 Gate 1 给人类复核**（v3.0 新增——堵住修订盲区）→ 人类确认后重新进入 Oracle
- **REJECT** → 打回 Stage 1，需重新出方案

> v3.0 关键修正：Oracle REVISE 后的修订方案不再直接进入 Meta-Oracle，必须经过人类复核。
> AI 在「修订」时可能引入新问题——只有人类能发现「修了一个 bug 引入了两个」。

### Meta-Oracle 对抗审查
审查维度：运行时验证、烟测日志、对抗性用例、盲区扫描。

裁决：
- **ACCEPT** → 进入 Stage 2
- **ADVISORY** → AI 自行判断是否采纳建议 → 进入 Stage 2
- **REJECT** → 打回 Stage 1
- 上限 **3 次**，3 次 REJECT → 任务进入 BLOCKED，需人类裁决

──────────────────────
## Stage 2：AI 执行

**方案锁定后全自主执行，不打断人类。** 并行区域并发跑。

### 执行流程
```
1. Token/全局约束初始化（串行——唯一串行依赖）
2. 并行区域拆分子任务 → 各自并发执行
    每个子任务完成后 → checkpoint
```

### Checkpoint 机制（v3.0 强化）

每个区域子任务完成后触发 checkpoint，两个维度：

| 维度 | 检查内容 | 失败动作 |
|:---|:---|:---|
| **技术烟测** | 回归测试、lint、编译、类型检查 | 自动修复（最多3次）→ 仍失败 → 暂停等人类 |
| **设计约束 diff** | 实际实现 vs 方案中的约束矩阵 | 偏差 > 阈值 → 暂停等人类裁决 |

设计约束 diff 检查项：
- 是否引入了方案禁止的依赖？
- 是否触及了范围边界外的文件？
- DOM 结构是否偏离了极简约束（过度包装）？
- 样式是否重复定义了可继承属性？

> v3.0 新增：设计约束 diff 解决了「AI 自认为正确但方向偏了」的问题。
> 纯技术烟测能发现 error，发现不了「用 antd Card 套了一个纯文本 span」。

**Checkpoint 超时策略**：暂停等人类，最长等待 30 分钟。超时 → 跳过该 checkpoint，记录到执行报告，继续下一区域。所有跳过的 checkpoint 在 Gate 2 强制人类复核。

### 三次修复法则
同一问题 3 轮修复仍失败 → BLOCKED，写入执行报告，继续下一区域。不阻塞并发区域。

──────────────────────
## Gate 2：人类审执行报告 → 双法官验收

### 执行报告强制结构
1. 变更清单（文件级 diff 摘要）
2. 每个 checkpoint 结果（通过/跳过/失败+修复记录）
3. 跳过的 checkpoint 清单（如有——原因 + 影响评估）
4. 已知未修复问题（BLOCKED 项）

人类动作：审报告 → 通过后人类启动双法官验收。

### 双法官验收（上限3次）
Oracle 验静态 + Meta-Oracle 验运行时。任一 REJECT → 打回 Stage 2。3次 REJECT → BLOCKED，需人类裁决。

──────────────────────
## Stage 3：最终验收报告

验收通过后生成最终报告。

### 报告强制结构（v3.0 强化）
1. 任务摘要（做了什么、结果）
2. **预估 vs 实际 ROI 对比表**（v3.0 新增）
3. 变更清单（完整）
4. 双法官验收记录
5. 遗留问题清单
6. 建议后续动作

### ROI 对比裁决（v3.0 新增核心机制）

```
实际ROI ÷ 预估ROI = 偏离率

偏离率 ≤ 50%  → 记录，不阻断
偏离率 50-70% → 标注 ⚠️，写入建议
偏离率 > 70%  → 标注 🔴，触发人类裁决
```

ROI 对比维：
- **时间成本**：预估 vs 实际工时
- **复杂度**：预估 vs 实际变更文件数/行数
- **风险**：预估 vs 实际触发的 checkpoint 失败次数

> v3.0 新增：防止 AI 在 Stage 1 低估成本以争取方案通过。
> 偏差 > 70% 意味着「要么预估时骗了，要么执行时失控了」——两种都需要人类判断。

──────────────────────
## Gate 3：人类闭环

人类读最终验收报告 → 确认闭环 / 要求补充 / 打回重做。

──────────────────────
## 报告质量审查（v3.0 新增）

**所有提交给人类的报告（方案文档、执行报告、验收报告）在提交前经过精简 Oracle 结构审查。**

### 精简 Oracle 审查维度（只看结构，不看内容）

| 检查项 | 标准 |
|:---|:---|
| 结构完整性 | 是否包含所有强制章节？ |
| 证据标注 | 每个断言是否有 file:line 或命令输出？ |
| 软完成语 | 是否出现「应该没问题」「看起来正常」？ |
| 可操作性 | 人类读完后是否知道「下一步做什么」？ |
| 长度控制 | 是否 ≤ 合理上限（方案 5KB / 执行报告 8KB / 验收报告 5KB）？ |

**不审内容正确性**——那是双法官的职责。精简 Oracle 只保证「报告本身是可用的输入」。

裁决：
- **PASS** → 提交
- **STRUCTURE_FAIL** → AI 重写报告，补全缺失章节（自动，不打断人类）
- 3次 STRUCTURE_FAIL → 标注 [报告质量受限]，提交但附带说明

## 任务分级

| 级别 | 定义 | 工作流 |
|:---|:---|:---|
| **L1** | 单步：改一行配置、修一个 typo、查一个值 | 跳过完整流程，直接执行 + 烟测 |
| **L2** | 多步：单一模块内的修改 | 轻量版：Stage 1 方案 + Oracle（跳过 Meta-Oracle）+ 执行 + 烟测 |
| **L3** | 架构：跨模块/新功能/重构 | 完整五阶段 + 双法官 |
| **L4** | 关键：不可逆操作/删除/发布/安全 | L3 + Gate 1 强制人类书面确认 |

判定方式：LLM 初判 → 人类在 Gate 1 确认。

──────────────────────
## 反模式

| 反模式 | 表现 | 纠正 |
|:---|:---|:---|
| 跳跃式方案 | 方案写「重构 XX 模块」，没写具体改哪些文件 | 打回，要求约束矩阵 |
| 幽灵修订 | Oracle REVISE → AI 改完直接进 Meta-Oracle | 必须回 Gate 1 人类复核 |
| 软验收 | 烟测失败 3 次后说「影响不大」直接继续 | BLOCKED，暂停等人类 |
| ROI 注水 | Stage 1 预估「1 小时」，实际跑了 4 小时 | 偏差 >70% 触发裁决，写入反模式库 |
| 报告膨胀 | 验收报告 12KB，大量日志原文堆砌 | 精简 Oracle 打回，要求压缩 |
| 串行惰性 | 4 个独立区域排队跑而非并发 | 打回 Stage 1 重新识别并行区域 |
