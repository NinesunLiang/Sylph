     1|# 三重门神谕 — 三端交叉验证的终极审判

> 📍 弧5：审判：三重门神谕 | [⬅ 上篇](story-14.md) | [下篇 ➡](story-16.md)


A 端预言者放下笔。他写了 7 条可证伪预测——不是建议，不是方向，是可以被证明为错的具体断言。

"如果修改 completion-gate.sh:47，(1) stderr 输出消失，(2) 日志写入 .omc/state/completion-gate-autonomous.log，(3) exit code 保持 0。"

B 端盲执行者走进房间。没人告诉他 A 说了什么。他只收到原始需求和一份代码。他打开文件，开始修改。

Oracle 在高台上等两人的产出——左手的预言，右手的执行。比对、裁决、报告。Oracle 不做执行。Oracle 只回答一个问题：这两个人，谁漏了什么？

---

## 为什么需要三重门？

哲学 #6 说："先天对 AI 0 信任。"但"0 信任"不是"不做任何事"——是"任何单一来源都不足以采信"。

一个 AI 可以正确。一个 AI 也可以犯错。但三个 AI——使用三种不同的推理路径——同时犯错且犯相同错误的概率，趋近于零。

三重门的核心设计：**A ≠ B ≠ Oracle。**

- **A 端（预测者）**：从代码层面做预测——"如果改了这个函数，下游的 X/Y/Z 会受影响"
- **B 端（盲执行者）**：不接受 A 的任何信息，独立理解需求，独立执行
- **Oracle（终审者）**：读取 A 的预测 + B 的执行结果，逐条比对，裁决差异

---

## 第一重：A 端 — 可证伪的预言者

A 端的任务是：**做出可以被证明为错的预测。**

这和其他 AI 系统的做法截然相反。大多数 AI 输出的内容是"建议"——不可证伪、模糊、无法判断是对是错。A 端的输出必须是**可证伪断言**：

- ❌ "修改应该不会有问题"（不可证伪）
- ✅ "修改 completion-gate.sh:47 将从 stderr 改为日志，预测：(1) 不再输出到终端 (2) 写入 .omc/state/completion-gate-autonomous.log (3) exit code 保持为 0"（每一步都可以验证）

A 端预测的格式：
```
PREDICTION: [具体断言]
  - [子预测 1] — 可验证方式: [验证步骤]
  - [子预测 2] — 可验证方式: [验证步骤]
```

预言者不执行。预言者的输出是 B 端的输入、Oracle 的比对材料。

---

## 第二重：B 端 — 盲眼的执行者

B 端不接受 A 端的预测。B 端看到的是**原始需求**——与 A 端相同，但不知道 A 端预测了什么。

盲执行的关键在于——B 不能试图"验证 A 的预测"。B 的任务是：尽最大努力独立完成需求，不参考任何其他 AI 的判断。

执行完后，B 端的产出是独立的结果——可能与 A 端的预测吻合，可能不吻合。不吻合不是失败——不吻合是有价值的信息。

---

## 第三重：Oracle — 终端裁决者

Oracle 是整个三重门的最高权威。Oracle 不做任何执行——Oracle 只做一件事：**比对。**

Oracle 收到三份文件：
1. A 端的预测列表
2. B 端的执行结果
3. 原始需求

Oracle 逐条比对：

| A 预测 | B 结果 | 裁决 |
|--------|--------|------|
| 预测正确 | 结果吻合 | ✅ HIT — A 的理解正确 |
| 预测正确 | 结果不吻合 | ⚠️ MISMATCH — B 可能误解了需求 |
| 预测错误 | 结果不吻合 | ❌ MISS — A 的理解有盲区 |
| 未预测到 | 出现遗漏 | 🔴 GAP — A 和 B 都遗漏了 |

Oracle 输出一份详尽的**比对报告**：哪些预测被验证、哪些被推翻、哪些被遗漏、哪些是新发现。

## 双门禁：Oracle 管线的两次裁决

Oracle 不止在一个时间点出现。她在 L3 流水线上有两个门禁：

### Oracle 方案审核（执行前）

在执行开始之前，Oracle 审查：
- 方案的技术可行性
- 方案是否覆盖了所有验收条件
- 方案是否引入了不可接受的风险

不通过 → 方案被打回，AI 带着 Oracle 的具体问题重新设计。

### Oracle 终审（执行后）

执行完成后，Oracle 审查 qa.md（验证报告）：
- 所有验收条件是否逐项验证？
- 每项验证是否有 L1/L2 证据？
- 代码变更摘要是否完整？

通过 → 标记完成。不通过 → 回到执行阶段，带着 Oracle 给出的改进方向。

### Meta-Oracle — 最后守门员

Oracle 不是绝对正确的。她的评分方法论有盲区，她的静态检查会漏掉运行时才会暴露的问题。所以 Carror OS 还有一位更高位的存在 — **Meta-Oracle**。

Meta-Oracle 是 Oracle 的审判官。她只在四个关键节点现身：

| 触发点 | 场景 | 理由 |
|--------|------|------|
| **G1** | 架构决策终审 | 架构错了全盘皆输 |
| **G2** | PRD 方案最后一步 | 蓝图错了执行全错 |
| **G3** | Oracle ACCEPT + 高分 | Oracle 最可能虚高的时刻 |
| **G4** | Release 门禁 | 发布的破坏不可逆 |

Meta-Oracle 不常出现。她是核武器 — 消耗巨大，非必要不使用。Oracle 是常规守门员，每天站岗。Meta-Oracle 是最后守门员，只在最关键的节点才会被请出来。

她的裁决是软门禁 — REJECT 时 AI 可以在明确书面理由下覆写。但连续两次 REJECT，就升级为事实阻断，必须人工介入。

在 A→B→A 三重门中的位置：

```
A 预测 → B 盲执行 → A 自证 → Oracle 审核（常规守门员）
                              ↓
                         Meta-Oracle 最后守门（仅在 G1-G4 触发）
```

---

## 三重门为什么不能被简化为"一个很仔细的 AI"？

因为"仔细"不能被信任。

一个 AI 反复检查同一个自己产出的代码——它看到的仍然是自己的盲区。它的第一个假设决定了搜索方向，反复检查只是在同一条路径上走了两次。

三重门的 A ≠ B ≠ Oracle 物理隔离打破了这种"自我验证"的幻觉：

- A 端产出的预测和 B 端的执行**来自不同的推理路径**
- Oracle 的比对**来自更高的抽象层**（不关注实现细节，关注预测/结果的一致性）
- 三端的差异不是"谁对谁错"——差异本身就是信息

三方全部一致的结论：可认为是可靠的。
两方一致、一方不同的结论：需要人工裁决。
三方全不同的结论：系统性问题，需要重新调研。

---

## 三重门在 Carror OS 中的位置

三重门不是一个独立的 hook 或 script——它是**协议**，由多个节点（nodes/）定义：

- `oracle_terminal.md` — Oracle 的角色、输入、输出、裁决逻辑
- `a_terminal.md` — A 端的可证伪预测格式
- `b_terminal.md` — B 端的盲执行协议
- `gate_checker.md` — 门禁检查规则

这些节点在 L3 开发流水线中被调用，遵循哲学 #7（文档优先，调研先行）— 出方案 → Oracle 审核 → 执行 → 报告 → Oracle 终审 → 完成。

---

## 神谕的重量

三重门不是一个高效的流程。三个 AI 的协调时间 + Oracle 的比对时间，远超一个 AI 独自完成。但它从来不是为了效率而设计的。

它是哲学 #6（0 信任）的最纯粹物化——在信息和信任之间，选择前者。在效率和安全之间，选择后者。在"应该没问题"和"三重验证通过"之间——神谕的裁决不可上诉。
