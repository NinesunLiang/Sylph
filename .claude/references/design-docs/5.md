# CarrorOS 第三轮迭代：第 5/10 次（优化版）

## 迭代主题：VerifyGate 完成裁决硬门

本轮核心问题：

```text
Executor Evidence Ledger 记录执行事实后，如何裁决当前 step 是否真正完成？
```

---

## 1. 核心裁决

**裁决等级：核准**

VerifyGate 的单一职责：

```text
将 plan.md 中当前 step 的 verify 规则与 executor.md 中标准 evidence 进行匹配，裁决 step 状态
```

**允许输出：**
```text
VERIFIED   - 所有 verify 规则匹配，无未解决 failure
WARN       - 部分证据存在，但强度不足
BLOCKED    - 证据缺失或存在未解决 failure
REJECTED   - verify 规则非法或存在软完成语
```

**禁止输出：**
```text
✗ LOOKS_GOOD
✗ PROBABLY_DONE
✗ ACCEPTED_BY_FEELING
✗ SKIP_VERIFY
✗ TRUST_ME
```

**最终裁决：**

```text
plan.md 标记 [x] 的唯一合法入口是 VerifyGate VERIFIED
任何其他路径标记 [x] 均视为审计违规（违反铁律 5：不假完成）
```

---

## 2. 与 CarrorOS 铁律的对齐

VerifyGate 强制执行以下铁律：

### 铁律 2：证据门禁
```text
每步改完必须贴命令输出或 diff
VerifyGate 只接受 executor.md 中标准化的 evidence
```

### 铁律 5：不假完成
```text
没跑 VerifyGate = 没完成
plan.md 的 [x] 只能由 VERIFIED 裁决产生
```

### 铁律 1：不编造
```text
软完成语（"应该好了"、"looks good"）触发 REJECTED
断言必须有明确证据：command exit_code=0、file_assertion、user_confirmation
```

---

## 3. VerifyGate 的职责边界

### 3.1 读取来源（符合 README.md 文档系统）

```text
✅ .omc/tokens/{date}/{task_name}.json        - token 状态源
✅ .omc/tasks/{date}/{task_name}/plan.md      - 计划源
✅ .omc/tasks/{date}/{task_name}/executor.md  - 证据源
✅ .omc/audit/{date}.jsonl                    - 审计尾部（可选）
```

### 3.2 禁止行为

```text
✗ 不执行修复（违反单一职责）
✗ 不扩大 scope（违反铁律 3：范围冻结）
✗ 不新增 verify 规则（verify 由 PlanBuilder 冻结）
✗ 不删除失败证据（executor.md 是追加式账本）
✗ 不将 Oracle ACCEPT 当完成证据（Oracle 是复核，不是证据）
✗ 不将用户"继续"当确认（必须是原子验收项）
✗ 不将 file_change 当 file_assertion（前者证明修改，后者证明断言）
```

## 4. 总体流程（对齐 L1 工作流）

```mermaid
flowchart TD
    A[启动 VerifyGate] --> B[读取 token.json]
    B --> C[读取 plan.md 当前 step]
    C --> D[读取 executor.md evidence]
    D --> E{token.current_step 与 plan pending 一致?}

    E -->|否| F[BLOCKED: state_conflict]
    E -->|是| G[解析 verify 规则]

    G --> H{verify 规则存在?}
    H -->|否| I[REJECTED: missing_verify]

    H -->|是| J[匹配 evidence]
    J --> K{全部 verify 匹配?}

    K -->|是| L{存在未解决 failure?}
    L -->|否| M[VERIFIED]
    L -->|是| N{后续成功证据覆盖?}

    N -->|是| M
    N -->|否| O[BLOCKED: unresolved_failure]

    K -->|否| P{只有弱证据?}
    P -->|是| Q[WARN]
    P -->|否| R[BLOCKED: evidence_missing]

    M --> S[安全标记 plan.md [x]]
    S --> T[更新 token done/current_step]
    T --> U[写 audit verify_decision]
```

**与 L1 工作流的关系：**

```text
L1: Plan → Step → Verify → Archive
         ↑       ↑       ↑
    init     执行   VerifyGate   archive
```

## 5. verify 规则语法（对齐 PlanBuilder）

PlanBuilder 允许 4 类 verify：

```markdown
# plan.md 示例
- [ ] S1: 修复 token 过期判断
  - scope: src/auth.ts
  - verify: command:npm test -- auth.test.ts
  - verify: file:src/auth.ts contains "expiresAt check"
  - verify: assertion:expired token rejects login
  - verify: user:确认登录流程符合预期
```

**规则：**
```text
1. verify 必须有前缀：command: / file: / assertion: / user:
2. 没有前缀的 verify 规则 → REJECTED: invalid_verify_rule
3. 多个 verify 规则必须全部匹配才能 VERIFIED
```

## 6. Evidence 匹配规则（继承第 4 次 Executor Ledger）

### 6.1 command 匹配

**plan.md：**
```text
verify: command:npm test -- auth.test.ts
```

**executor.md：**
```markdown
### EV-20260706-0001
- step: S1
- type: command
- source: npm test -- auth.test.ts
- exit_code: 0
- evidence_level: E3
```

**匹配条件：**
```text
1. type == command
2. source 规范化后与 verify 一致（多空格折叠、首尾空格忽略）
3. exit_code == 0
4. evidence_level == E3
5. evidence.step == current_step
```

**禁止：**
```text
✗ npm test 代替 npm test -- auth.test.ts（命令不完整）
✗ exit_code=1 被解释为通过（违反铁律 2：证据门禁）
✗ output 包含 "passed" 但 exit_code != 0（软完成语）
```

---

### 6.2 file 匹配

**plan.md：**
```text
verify: file:README.md contains "npm install"
```

**executor.md：**
```markdown
### EV-20260706-0002
- step: S1
- type: file_assertion
- file: README.md
- assertion: contains "npm install"
- evidence_level: E2
```

**匹配条件：**
```text
1. type == file_assertion（注意：不是 file_change）
2. file 路径一致
3. assertion 包含 verify 的断言内容
4. evidence_level ∈ {E2, E3}
5. evidence.step == current_step
```

**禁止：**
```text
✗ file_change 代替 file_assertion（前者只证明修改，不证明断言）
✗ "修改了 README" 代替 contains 断言
✗ scope 内文件变更自动算通过（违反零信任）
```

---

### 6.3 assertion 匹配

**plan.md：**
```text
verify: assertion:expired token rejects login
```

**executor.md（方式 1 - file_assertion）：**
```markdown
### EV-20260706-0003
- step: S1
- type: file_assertion
- file: src/auth.ts
- assertion: expired token rejects login
- evidence_level: E2
```

**executor.md（方式 2 - command output）：**
```markdown
### EV-20260706-0004
- step: S1
- type: command
- source: npm test -- auth.test.ts
- exit_code: 0
- output_tail: ✓ expired token rejects login
- evidence_level: E3
```

**匹配条件：**
```text
1. assertion 文本可在 file_assertion.assertion 或 command.output_tail 中匹配
2. 若为 command 证据，必须 exit_code == 0
3. evidence_level ∈ {E2, E3}
4. evidence.step == current_step
```

---

### 6.4 user 匹配

**plan.md：**
```text
verify: user:确认登录按钮文案符合预期
```

**executor.md：**
```markdown
### EV-20260706-0005
- step: S2
- type: user_confirmation
- confirmation: 用户确认登录按钮文案为"继续登录"，符合设计稿
- evidence_level: E1
```

**匹配条件：**
```text
1. type == user_confirmation
2. confirmation 是原子验收项（≥8 字，无软完成语）
3. confirmation 覆盖 verify 语义
4. evidence_level == E1
5. evidence.step == current_step
```

**禁止（违反铁律 1：不编造）：**
```text
✗ "继续"
✗ "可以"
✗ "都行"
✗ "你看着办"
✗ 用户沉默
```

## 7. 裁决语义

### 7.1 VERIFIED

**条件：**
```text
1. token.current_step 与 plan 第一个 pending step 一致
2. 当前 step 存在至少 1 条 verify
3. 所有 verify 规则均匹配到有效 evidence
4. 无未解决 failure
5. 所有 evidence 均绑定当前 step
```

**结果：**
```text
1. plan.md 当前 step 从 [ ] 改为 [x]
2. token.stats.done +1
3. token.task.current_step 移动到下一个 pending step
4. 若无 pending step，token.task.phase = archive
5. 写 .omc/audit/{date}.jsonl
```

**CLI 调用：**
```bash
python3 .claude/scripts/carros_base.py verify --step S1
# 输出：VERIFIED
```

---

### 7.2 WARN

**条件：**
```text
1. 有部分 evidence
2. evidence 类型弱于 verify 规则（如 E1 匹配 command verify）
3. 用户确认语义不够完整
4. file_assertion 缺少 line/snippet_hash 但仍可人工复核
```

**结果：**
```text
1. 不标记 [x]
2. 写 audit
3. 追加 executor.md warning 区块
4. 要求补充证据或用户原子确认
```

**裁决：**
```text
WARN 不是完成
WARN 不推进 current_step（符合铁律 5：不假完成）
```

---

### 7.3 BLOCKED

**条件：**
```text
1. 缺少 evidence
2. token 与 plan 冲突（state_conflict）
3. executor.md 缺失
4. verify 规则无法匹配
5. 存在未解决 failure
6. audit 写入失败
7. 当前 step 不存在
```

**结果：**
```text
1. 不标记 [x]
2. token.task.status = blocked
3. token.task.blocked 写入 reason
4. executor.md 追加 VerifyGate 阻塞记录
5. 等待修复或用户裁决
```

---

### 7.4 REJECTED

**条件：**
```text
1. plan.md 当前 step 没有 verify 规则（违反 PlanBuilder 约定）
2. verify 规则语法非法（缺少前缀）
3. evidence 存在软完成语（违反铁律 1：不编造）
4. executor.md 把 narrative 当 evidence
5. step 被绕过直接标记 [x]（违反铁律 5：不假完成）
```

**结果：**
```text
1. 不标记 [x]
2. token.task.status = blocked
3. 要求：
   - 回到 PlanBuilder 修正 verify 规则
   - 或回到 Executor Ledger 补充合法 evidence
```

## 8. 软完成语拦截（强化铁律 1）

**黑名单（固定）：**
```python
SOFT_COMPLETION_PHRASES = [
    "应该好了", "看起来可以", "基本完成", "大概没问题",
    "已经处理", "完成了", "done", "fixed", 
    "looks good", "should work", "probably fixed",
    "可以了", "没问题", "OK", "ok"
]
```

**裁决：**
```text
软完成语只能作为 narrative（E0）
不得作为 evidence（E1/E2/E3）
命中软完成语且无硬证据时 → REJECTED: soft_completion_phrase
```

**示例（REJECTED）：**
```markdown
### EV-xxx
- step: S1
- type: file_assertion
- assertion: 应该好了
- evidence_level: E2
```

**输出：**
```json
{
  "decision": "REJECTED",
  "reason": "soft_completion_phrase",
  "step": "S1"
}
```

## 9. 未解决 failure 判定

**规则：**
```text
1. failure 不因时间流逝自动解决
2. failure 不因用户说"继续"而解决
3. failure 不因 file_change 而解决
4. failure 只能被对应成功证据覆盖
```

**示例：**

**executor.md：**
```markdown
### FAIL-20260706-0001
- step: S1
- action: command:npm test -- auth.test.ts
- exit_code: 1
- reason: test_failed

### EV-20260706-0002
- step: S1
- type: file_change
- file: src/auth.ts
- change_summary: fixed token expiry check
```

**裁决：**
```text
BLOCKED: unresolved_failure:FAIL-20260706-0001
```

**原因：**
```text
file_change 不能覆盖 command failure
必须有 command:npm test -- auth.test.ts exit_code=0 的 E3 evidence
```


**冲突检测：**
```text

```

```json
  },
  "stats": {
    "done": 1,
    "total": 3
  }
}
```

**最后一个 step VERIFIED 后：**

```json
{
  "task": {
    "id": "my-task-001",
    "status": "verifying",
    "phase": "archive",
    "current_step": null,
    "blocked": null
  },
  "stats": {
    "done": 3,
    "total": 3
  }
}
```

**一致性规则：**
```text
token.stats.done 必须等于 plan.md 中 [x] 数量
token.stats.total 必须等于 plan.md step 总数
不一致 → BLOCKED: token_plan_counter_mismatch
```

## 12. VerifyGate 核心代码（完整版）

完整代码已在前面提供，这里补充关键优化点：

### 12.1 与 carros_base.py 的集成

```python
# .claude/scripts/carros_base.py

def verify(args):
    """验证当前 step（L1 工作流第 3 步）"""
    token_path = Path(f".omc/tokens/{date}/{args.task_id}.json")
    
    # 调用 verify_gate.py
    result = subprocess.run(
        [sys.executable, ".claude/scripts/verify_gate.py"],
        capture_output=True,
        text=True
    )
    
    if result.returncode == 0:
        print("✅ VERIFIED")
    else:
        decision = json.loads(result.stdout)
        print(f"❌ {decision['decision']}: {decision['reason']}")
```

### 12.2 路径兼容性（README.md 要求）

```python
# 支持 macOS / Windows / WSL
from pathlib import Path

token_path = Path(".omc/tokens") / date / f"{task_id}.json"  # ✅ 跨平台
plan_path = Path(".omc/tasks") / date / task_id / "plan.md"  # ✅ 跨平台
```

## 14. 设计优缺点

### 14.1 优点

```text
✅ 完成裁决唯一化（符合铁律 5）
✅ plan.md 标记 [x] 有硬入口（防止绕过）
✅ executor.md 事实与完成裁决分离（单一职责）
✅ 软完成语被系统性拦截（强化铁律 1）
✅ failure 不会被成功叙述覆盖（零信任）
✅ token / plan / executor 三源一致性增强
✅ L1 可用静态规则完成验证
✅ L2 可在 VerifyGate 后接 Oracle 复核
```

### 14.2 缺点

```text
❌ 证据格式要求严格（学习成本）
❌ file verify 需要额外 file_assertion（非 file_change）
❌ 用户确认必须原子化（不能偷懒）
❌ 宽松项目会感觉流程偏硬
❌ Markdown 解析有格式依赖
❌ 命令匹配过严时可能需要补充规范化规则
```

### 14.3 取舍裁决

```text
CarrorOS 选择可审计完成，而非口头完成
严格证据匹配增加成本，但消灭"看起来完成"的灰区
该代价符合"验证 > 零信任 > 守护"的核心灵魂
```

## 15. 与 Oracle 的边界（L2 工作流）

Oracle / Meta-Oracle 可在 L2 中复核：

```text
- 风险盲区
- 架构一致性
- 对抗性失败
- 验收充分性
```

**但 Oracle 禁止：**
```text
✗ 跳过 VerifyGate
✗ 替 VerifyGate 标记 [x]
✗ 用 ACCEPT 覆盖缺失 command evidence
✗ 用主观判断覆盖 unresolved failure
```

**正确顺序（L2）：**
```text
Executor evidence → VerifyGate VERIFIED → Oracle phase_end/final → Archive
```

**裁决：**
```text
Oracle 是高阶复核（L2 特性）
VerifyGate 是完成硬门（L1/L2 共用）
```


**禁止：**
```text
✗ compact 后凭记忆继续标记 [x]
✗ handoff 写"已完成"就跳过 VerifyGate
✗ prompts 中用户说"继续"作为 user_confirmation
```

## 17. 本轮最终规则（14 条铁律）

```text
1. VerifyGate 是唯一后置完成门
2. plan.md 标记 [x] 的唯一合法入口是 VERIFIED
3. VerifyGate 只裁决，不执行修复
4. verify 规则必须有 command:/file:/assertion:/user: 前缀
5. 所有 verify 必须匹配 executor.md 标准 evidence
6. command verify 必须匹配 exit_code=0 的 E3 evidence
7. file verify 必须匹配 file_assertion，不接受 file_change
8. user verify 必须匹配原子 user_confirmation
9. 软完成语命中且无硬证据时 → REJECTED
10. 未解决 failure 时 → BLOCKED
11. WARN 不推进 current_step
12. audit 写入失败时 → BLOCKED
13. token.done/total 必须与 plan.md 一致
14. Oracle 不得覆盖 VerifyGate
```
