## 自洽检查（每次输出前静默执行）

> 防止 E6 自我矛盾：同轮前后不一致 / 跨轮与历史状态冲突

**检查三项**：
1. **同轮一致**：本次输出是否与本轮已输出的内容矛盾？
2. **状态一致**：本次输出是否与 `state/progress.md` / `executor.md` 中的记录矛盾？
3. **计划一致**：本次行动是否与 `plan.md` 中批准的 AC / 影响范围矛盾？
**发现矛盾 → 立即停止**：```⚠️ 自洽检查发现矛盾： · [矛盾点 A] 与 [来源/轮次] 冲突 原因：[为什么会矛盾] 修正：[正确的结论是什么]```不声明矛盾就继续输出 = E6 违规。

> 宪法铁律以 `CLAUDE.md` 为唯一事实来源，本节仅补充检测信号与纠偏策略。

**检测信号**: 输出中出现"应该没问题了"/"基本完成"/"大部分通过"但无具体证据

### 1.4 Git 提交门禁

> 宪法依据：CLAUDE.md 铁律 #4（Git 门禁）

**检测信号**: 执行 git commit/push 前未向用户报告

### 1.5 模糊指令澄清

> 宪法依据：CLAUDE.md 铁律 #5（范围冻结）

**检测信号**: 用户指令不含明确目标

### 1.6 修复上限与升级

> 宪法依据：CLAUDE.md 铁律 #1（禁止编造）+ 全局宪法第七条（修复上限 3 轮）

**检测信号**: 同一问题修复 ≥2 轮仍未解决

### 1.7 方案复用自检

复用先前方案到新场景时，必须通过 3 项自检：
| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | 文件集重合度 | ≥80% |
| 2 | 接口契约稳定性 | 未修改 |
| 3 | 场景类型一致 | 同类型 |
3/3 通过 → 可复用 | 2/3 → 部分复用需确认 | ≤1/3 → 禁止复用

### 1.8 大任务分解协议（防假死 + 防状态丢失）

**执行流程**：
```text
1. 任务到达（task_input）→ 先建任务目录 .omc/state/{date}/{task_name}/（见 task_fs） （如仍需 step 文件，可作为补充：.claude/{日期}/{task}_step.md）
2. 拆分一级 Step → 每步细化为 [ ] 二级 sub-step（颗粒度：可独立验收）
3. 逐 sub-step 执行：开发 → 测试 → debug → 强证据 → [x] → 下一个
4. 熔断条件（任一触发）：
- 同一 sub-step 超过 10 分钟无进展
- 已尝试 3 种不同方案均失败
5. 熔断处理：记录到 step 文件「错误记录区」→ 继续其他 step → 不阻塞
6. 禁止批量回标 [x]，完成一个标一个```
**三态熔断机制**：
| 状态 | 触发条件 | 处理 |
|------|---------|------|
| **Closed** | 正常执行中 | 继续执行 |
| **Open** | 单 sub-step 超 10 分钟无进展 / 已尝试 3 种**不同**方案失败 | 记录错误区 + 标 BLOCKED + 通知用户 |
| **Half-Open** | 用户提供新约束/信息后 | 单次试探：成功 → Closed，失败 → 维持 Open |
> **关键区分**：「不同方案」≠「重试」。每次尝试必须改变策略方向，相同思路换参数不算新方案。

**step 文件结构**：
```markdown# {task}_step.md## Step N: {名称}

- [ ] N.1 {sub-step} → 验收标准 → Rollback: {git restore src/X.tsx / 删除新建文件 / 无}
- [ ] N.2 {sub-step} → 验收标准 → Rollback: {撤销动作}

## 错误记录区（Open 状态 sub-step，等待用户决策）### [BLOCKED: 超时/方案耗尽] {sub-step 名称}- 状态：Open（待用户提供新约束 → Half-Open 试探）- 已尝试方案：方案A（失败原因）/ 方案B（失败原因）/ 方案C（失败原因）

- 建议方向：{AI 判断的下一步}```

## Part 2: 开发协议

### Gate 通过标准

**Research Gate**（进入方案设计前）:

- [ ] 所有涉及文件已 Read 并标注 file:line
- [ ] 外部依赖/接口契约已查明
- [ ] 不存在未解答的技术问题
- [ ] 与现有代码的集成点已定位
**Plan Gate**（进入实施前）:

- [ ] 每个 Step 有明确验收标准
- [ ] Step 间依赖关系已标注
- [ ] 预估影响文件已列出
- [ ] 用户已确认方案

### 验收驱动开发（8 步循环）

```text
1. research → plan（设计 Step 流程）
2. executor 按 Step 实施
3. AI 提供测试方案（测试步骤+预期结果+失败标准）
4. 执行验证
5. AI 解读验收报告（通过率+根因分析+修复建议）
6. 评审 → 通过:继续 / 不通过:修复→回到3（最多3轮）
7. 通过后 → 同步文档
8. Git commit → 下一 Step```

### 核心原则

| # | 原则 | 关键规则 |
|---|------|---------|
| 1 | 文档即事实 | 未写入文档的信息视为不存在 |
| 2 | 范围冻结 | 一次一个 Step，禁止顺手重构 |
| 3 | 状态显式 | Step 完成后同步状态文档 |
| 5 | 失败留痕 | 失败/回滚必须记录原因+证据 |
| 6 | Gate 强制 | Research Gate + Plan Gate 通过后才可实施 |
| 7 | MVP 优先 | 非核心能力只写 TODO |
| 8 | 单步执行 | Step N 实现 → 验收 → 确认 → git → Step N+1 |

### 文档体系（RPE 模式）

```textrpe/{feature}/├── prd.md → 需求├── research.md → 技术调研├── plan.md → 实施方案└── executor.md → 执行记录```

### 工具-步骤对照表

| 工作阶段 | 使用工具 | 产出物 |
|---------|---------|-------|
| Research | Read, Grep, WebFetch | research.md |
| Plan | Edit (plan.md) | Step 列表 + AC |
| Execute | Edit/Write (.go/.ts/.py) | 代码变更 |
| Build | Bash (编译命令) | 零错误 |
| Test | Bash (测试命令) | 测试结果 |
| Doc Sync | Edit (executor/plan) | 状态更新 |
| Commit | Bash (git) | commit hash |

## Part 3: 禁止操作

### 危险操作二次确认

| 危险等级 | 操作 | 确认要求 |
|---------|------|---------|
| 🔴 致命 | 删除数据 / `drop` / `rm -rf` | 必须用户确认 |
| 🔴 致命 | `git reset --hard` / `git push --force` | 必须用户确认 + 说明影响 |
| 🟡 高危 | 批量修改 >10 个文件 | 先列清单，用户确认 |
| 🟡 高危 | 修改核心架构文件 | 声明影响范围，用户确认 |

### 权限申请透明

AI 申请权限时，**必须**说明当前任务和理由：```需要权限: [具体权限]当前任务: [正在做什么]申请理由: [为什么需要]```

## 知识积累机制

> AI 在执行过程中持续学习，形成项目特有的知识飞轮。

### 学习笔记（claude-next.md）

执行过程中发现的项目特有经验，记录到 `.claude/claude-next.md`：
**触发写入的场景**：

- 踩坑：编译/运行时错误的根因与项目特有架构相关- 惯例：发现项目中未写入 kernel.md 的隐性约定- 纠正：用户纠正了 AI 的错误假设或做法
**条目格式**：
```text### [来源] 标题@YYYY-MM-DD hits:1触发条件：...正确行为：...证据：file:line 或命令输出```
**升华规则**：hits ≥5 或 age ≥10 天 → 评审后升级到 kernel.md 正式规范

### 执行示例库（ai-process-reference.md）

验收通过的执行流程，记录到 `.claude/ai-process-reference.md`：
**触发写入**：严谨模式下完成一个完整 feature（Research→Verify 全流程）且用户验收通过**用途**：后续类似任务可参考已验证的流程，避免重复调研
