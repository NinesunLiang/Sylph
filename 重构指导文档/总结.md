# 第三轮重构方案评价

## 总裁决

**裁决等级：附条件核准。**

第三轮方案已经从“治理理念”推进到“可执行治理系统”，整体方向正确，且比第二轮更完整。它把 CarrorOS 的关键治理链路压成了：

```text
Intake → Plan → PreAction → Execute → Verify → Context → Oracle → Fallback → CLI → Archive
```

这条链路的优点很明确：

```text
1. 完成裁决统一收束到 VerifyGate。
2. 长任务恢复统一收束到 Context Engine。
3. 高阶风险复核统一收束到 Oracle / Meta-Oracle。
4. 不可用与降级统一收束到 Fallback Protocol。
5. 外部展示统一收束到 CLI Integration。
6. 最终退出统一收束到 Archive Engine。
```

第三轮最大的价值不是新增了多少模块，而是把原先散落在 executor、acceptance、Oracle、audit、handoff、CLI 里的“完成、恢复、降级、归档”语义做了统一裁决。

但第三轮还不能判定为“完全核准”。原因也明确：它的结构完整度已经很高，但工程可执行性、模块数量控制、真实任务验证基准还没有最终闭环。知识库此前已经指出，真正缺的是验证，不是更多设计；没有真实任务基准跑通前，不能声称完美可执行。

---

## 1. 核心进步

### 1.1 完成语义终于收敛

第二轮已经明确：

```text
Base:
  Plan → Execute → Verify → Archive
```

并指出验证逻辑不能散落在多个地方。第三轮延续并强化了这个方向，把 VerifyGate 定为唯一 step 完成硬门，删除了：

```text
- acceptance.md 独立完成门
- executor.md narrative completion
- Oracle 直接完成裁决
- 用户“继续”作为 user_confirmation
```

这是正确的。  
CarrorOS 最危险的漏洞之一，就是模型用自然语言宣布“完成”，但没有证据链。第三轮把这个口子关上了。

裁决：

```text
完成路径收敛：核准。
```

---

### 1.2 状态源边界更硬

第二轮第 4 次已经裁定：

```text
token.json 是唯一运行状态源。
plan.md 是唯一计划与验收规则源。
executor.md 是唯一执行证据源。
session-handoff.md 是唯一 compact/resume 摘要源。
```

第三轮完全继承了这个原则，并进一步加入：

```text
audit/*.jsonl 是审计源。
final-report.md 是归档报告，不是 evidence。
oracle-verdicts.md 是高阶复核，不是完成证据。
statusline 是展示层，不是治理真相。
```

这很关键。它防止了第三轮新增模块后再次产生“多真源”。

裁决：

```text
状态源设计：核准。
```

---

### 1.3 Context Engine 是第三轮最成功的新增机制

Context Engine 把 compact / resume / state injection 从“提示词技巧”提升为治理机制。

尤其是这几条正确：

```text
1. 上下文不是状态源。
2. compact 前写 session-handoff.md。
3. resume 顺序固定。
4. token / plan / executor 冲突时 BLOCKED。
5. Enhance 水位不可观测时降级 Base。
```

这正好回应了第二轮对 Context 的核心裁决：

```text
磁盘比 context 可靠。
状态只能有一个真源。
Context 只放当前执行所需的最小状态。
```

第三轮在这里没有偏离，反而补上了可执行脚本。

裁决：

```text
长任务恢复机制：核准。
```

---

### 1.4 Fallback Protocol 是必要补强

第三轮第 8 次把不可用、降级、熔断统一成一个 Fallback Protocol，这是必要的。

以前方案最大隐患之一是：

```text
ENHANCE 不可用时，系统可能假装仍在 ENHANCE。
Oracle 不可用时，模型可能口头继续。
Hook 失败时，状态可能悄悄漂移。
```

第三轮明确：

```text
失效必须可见。
降级必须审计。
高风险任务不得静默降级继续。
```

这是治理系统必须具备的反脆弱边界。

裁决：

```text
降级熔断机制：核准。
```

---

### 1.5 CLI 边界正确

第三轮把 Claude Code Statusline Hook 和 OpenCode SQLite 观测都压成展示/观测层，而不是治理真相源。

这点正确：

```text
CLI 可以观察治理系统。
CLI 不得成为治理真相。
```

尤其是 statusline 被明确禁止作为 evidence，这避免了“展示文本污染证据链”。

裁决：

```text
CLI 集成边界：核准。
```

## 2. 主要问题

## 2.1 第三轮重新膨胀出 10 个模块，有熵增风险

第二轮第 1 次曾明确提出：

```text
少即稳。
能合并的合并。
能删除的删除。
能降级为规则的，不保留为 Hook。
```

第三轮虽然说不是新增 Hook，但实际治理模块已经扩展为：

```text
1. IntakeGate
2. PlanBuilder
3. PreActionGate
4. Executor Ledger
5. VerifyGate
6. Context Engine
7. Oracle / Meta-Oracle
8. Fallback Protocol
9. CLI Integration
10. Archive Engine
```

这在 ENHANCE 中可以接受，但对 BASE 来说偏重。

风险不在“模块名多”，而在实现时可能出现：

```text
- 每个模块都有自己的状态判断
- 每个模块都写 audit
- 每个模块都可能 BLOCKED
- failure_type 与 reason 分散
- token 更新路径过多
```

如果不再压缩，第三轮会重新接近第二轮早期的“入口太多”问题。

裁决：

```text
结构完整，但需要实现层合并。
```

建议：

```text
BASE 实现不应真的暴露 10 个独立执行器。
BASE 最终应压成 4 个命令：
1. plan
2. step
3. verify
4. archive

Context / Fallback / Statusline 应作为内部服务，不作为用户认知入口。
```

---

## 2.2 Oracle / Meta-Oracle 仍有过度设计风险

第三轮对 Oracle 的边界比前两轮更清楚：

```text
Oracle 不得替代 VerifyGate。
Oracle 只能做 phase/final 风险复核。
```

这是正确的。

但 L2 Model-pass-curve + L3 Multi-Judge 在工程上仍有一个问题：

```text
如果没有真实强模型调用，只靠 Python 静态评分模拟，Oracle 会变成“伪智能评分器”。
```

第三轮代码里的 `score_l2()` 本质是启发式扣分，不是真正的模型复核。作为占位规则可以，但不能被宣传为真实 L2/L3 语义复核。

裁决：

```text
Oracle 设计核准，静态实现只能作为 fallback stub。
```

强制修正：

```text
ENHANCE 文档必须区分：
- oracle_engine.py static_stub
- model_oracle_review prompt/protocol
- multi_judge 真正由高阶模型执行
```

否则会出现“名字是 Oracle，实际是 if-else”的治理幻觉。

---

## 2.3 Audit 写失败 = BLOCKED 是正确但实现成本高

第三轮多处规定：

```text
audit 写失败必须 BLOCKED。
```

原则正确。  
但实际落地时要面对：

```text
- 文件权限问题
- 并发写入问题
- Windows / WSL 路径差异
- jsonl 半行写入
- 多进程同时触发 hook
```

当前脚本多为简单 append，没有文件锁。  
在单 agent 单任务下可以接受；在多 hook 并发或多 CLI 同时触发下会有审计损坏风险。

裁决：

```text
单进程可核准；并发场景附条件核准。
```

建议增加：

```text
1. audit append atomic lock。
2. audit jsonl repair/check 命令。
3. archive 前 audit integrity scan。
```

---

## 2.4 plan.md 解析契约过脆

第三轮大量代码依赖：

```text
- [ ] STEP_ID: title
- [x] STEP_ID: title
## Scope
## Goal
```

这很简单，适合 BASE。  
但风险是 plan.md 一旦格式轻微偏移，系统就 BLOCKED。

这不是错，但必须承认：

```text
Markdown 现在是半结构化数据库。
```

如果继续走 Markdown，必须冻结模板并提供 lint。

裁决：

```text
可接受，但必须增加 plan_lint。
```

建议：

```text
第三轮后补一个 omc_lint.py：
- 校验 token schema
- 校验 plan.md 格式
- 校验 executor evidence block
- 校验 audit jsonl
- 校验 archive readiness
```

---

## 2.5 BASE / ENHANCE 文件集边界仍需再压

知识库此前已经强调二元法：

```text
L1 = Base 工作流 = 中型任务 = 轻治理 = 稳定完成
L2 = Enhance 工作流 = 复杂任务 = 飞轮增强 = 持久聪明
```

第三轮保留了二元法，这是正确的。

但第三轮 BASE 文件集仍包含：

```text
context_engine.py
verify_gate.py
fallback_engine.py
statusline_renderer.py
archive_engine.py
```

如果再加 IntakeGate、PlanBuilder、PreActionGate、Executor Ledger 的脚本，BASE 会偏重。

裁决：

```text
二元法方向核准，BASE 实现需要瘦身。
```

建议：

```text
BASE 只保留一个 carros_base.py 也可以：
- init
- next
- verify
- archive
- status

ENHANCE 再拆分 oracle / opencode / context 高级能力。
```

## 3. 第三轮对第二轮目标的符合度

第二轮第 1 次要求：

```text
减少分支、减少 Hook、减少状态漂移。
```

第三轮评价：

```text
减少状态漂移：完成得好。
减少 Hook：概念上完成，但模块数偏多。
减少分支：Fallback 矩阵让分支可审计，但分支数量上升。
```

第二轮第 4 次要求：

```text
token / plan / executor / handoff 边界清楚。
```

第三轮评价：

```text
完成得好。
```

知识库复核曾指出：

```text
真正缺的是验证，不是更多设计。
```

第三轮评价：

```text
设计已足够，下一步必须进入真实任务基准验证。
继续设计会熵增。
```

---

## 4. 必须修正的 7 个点

第三轮进入正式实现前，必须补以下 7 项：

```text
1. 增加最小真实任务基准：
   - 文档任务
   - 单文件代码任务
   - 多文件测试任务
   - 失败后修复任务
   - compact/resume 任务
   - fallback 降级任务
   - archive 归档任务

2. 增加 omc_lint.py：
   统一校验 token / plan / executor / audit / archive。

3. 区分 Oracle 静态 stub 与真实模型 Oracle：
   不得把启发式 Python 扣分称为完整高阶复核。

4. BASE 命令入口瘦身：
   用户认知入口不得暴露 10 个治理模块。

5. 增加 audit 完整性检查：
   至少 archive 前扫描 JSONL 是否可解析、事件字段是否齐全。

6. 增加 schema version：
   token.json / audit event / executor evidence 都应带 schema_version。

7. 冻结 Markdown 模板：
   plan.md 与 executor.md 必须有 lint，不允许自由格式漂移。
```

## 5. 最终评价

第三轮是一次有效重构。  
它解决了 CarrorOS 之前最危险的几个问题：

```text
- 软完成语污染
- 多状态源漂移
- compact 后凭记忆续接
- Oracle 越权
- Hook 失败静默继续
- CLI 展示污染治理真相
- 归档无证据封存
```

但第三轮也把系统推到了一个新的边界：

```text
现在不是缺规则。
现在是规则太完整，必须证明它能跑。
```

最终裁决：

```text
第三轮方案：附条件核准。

架构方向：核准。
状态源设计：核准。
VerifyGate 收敛：核准。
Context / Fallback / Archive：核准。
Oracle：附条件核准。
BASE 可执行性：附条件核准。
真实落地：必须通过最小任务基准后才能正式核准。
```

一句话评价：

```text
第三轮已经把 CarrorOS 从“提示词治理方案”推进到了“文件状态机治理方案”。
下一步不应继续扩写机制，而应进入最小实现、lint、基准任务和真实回放。
```
