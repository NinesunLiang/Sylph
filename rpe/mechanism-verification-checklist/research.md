
# Research: 机制验证清单需求分析

  ▎ Phase: Research | Date: 2026-07-08 | Status: Draft

  ---
  1. 问题陈述

  1.1 当前状况

  CarrorOS 项目存在以下现状：
  - ✅ 文件完整性高 - 95%+ 的文档和脚本文件存在
  - ✅ 架构设计完善 - OMA 三层体系、执行引擎、状态管理逻辑清晰
  - ❌ 实际运行验证缺失 - .omc/ 目录不存在，从未真实运行
  - ❌ 机制有效性未知 - 无法确认 skills 是否真的被 Claude 触发

  1.2 核心问题

  命题: 文件存在 ≠ 机制存在 ≠ 机制有效

  具体表现：
  1. lx-goal.py 存在，但从未运行 → 状态管理机制未验证
  2. AGENTS.md 有 70+ 个 @ 引用，但不知道 Claude 是否真的读取 → 集成机制未验证
  3. terminal-safety.md 定义了规则，但不知道 Claude 是否遵守 → 安全机制未验证
  4. OMA 三层定义了调用链，但无运行日志 → 编排机制未验证

  2. 目标定义

  2.1 主目标

  建立一套系统化的机制验证体系，确保 CarrorOS 的每个核心机制都经过四级验证：
  - L1 静态 - 文件/目录存在
  - L2 逻辑 - 文档定义完整
  - L3 运行 - 脚本可执行
  - L4 实战 - 真实场景工作

  2.2 子目标

  1. 建立验证清单 - 覆盖所有核心机制（执行引擎、OMA、状态管理、安全、治理）
  2. 编写验证脚本 - 自动化 L1-L3 验证
  3. 设计实战测试 - L4 端到端场景
  4. 修复阻塞问题 - 优先解决 P0 缺陷（如 .omc/ 不存在）
  5. 持续跟踪 - 建立验证状态仪表盘

  2.3 验收标准 (AC)

  AC1: 验证清单覆盖所有核心机制
  - 执行引擎 4 个（goal/ghost/race/stepwise）
  - OMA 体系 3 层（GOV/HIER/ORCH）
  - 状态管理 2 个（.omc/结构、varlock）
  - 安全机制 2 个（terminal-safety、bash-style）
  - 治理技能 3+ 个（code-review、git-check、RCA）
  - AGENTS 集成 1 个
  - 辅助系统 3+ 个
  
  AC2: P0 验证脚本可自动运行
  bash scripts/verify_p0.sh
  # 输出: 通过/失败/警告 的清晰报告
  # 退出码: 0=全通过, 非0=有失败

  AC3: 发现并记录所有 P0 阻塞问题
  - BLOCK-01: .omc/ 目录不存在
  - BLOCK-02: lx-ghost 缺执行脚本
  - BLOCK-03: 无运行日志证据
  - （其他待发现）

  AC4: 至少 1 个 L4 端到端场景通过
  - 场景: 目标驱动开发全流程
  - 预期: OMA-GOV → lx-goal → HIER → ORCH → code-review → git-check → 交付
  - 验证: 产生完整的工作产物和日志

  3. 范围界定

  3.1 包含范围 (In Scope)

  核心验证对象

  1. 执行引擎
    - lx-goal (P0)
    - lx-ghost (P0 - 缺脚本)
    - lx-race (P1)
    - lx-stepwise (P1)
  2. OMA 体系
    - lx-oma-gov (P1)
    - lx-oma-hier (P1)
    - lx-oma-orch (P1)
  3. 状态管理
    - .omc/ 目录结构 (P0)
    - lx-varlock (P2)
  4. 安全机制
    - terminal-safety.md (P0)
    - bash-style.md (P1)
  5. 治理技能
    - lx-code-review (P1)
    - lx-git-check (P1)
    - lx-root-cause-analysis (P2)
  6. 集成机制
    - AGENTS.md @ 引用 (P0)

  验证层级

  - ✅ L1 静态完整性（全部覆盖）
  - ✅ L2 逻辑自洽性（全部覆盖）
  - ✅ L3 运行测试（核心 skills）
  - ✅ L4 实战效果（至少 1 个端到端场景）

  3.2 排除范围 (Out of Scope)

  - ❌ 性能基准测试（留待后续优化阶段）
  - ❌ 压力测试/并发测试（P2 验证）
  - ❌ 完整的端到端回归测试套件（太庞大）
  - ❌ 用户体验评估（需要真实用户反馈）
  - ❌ 文档美化/重写（只关注机制验证）

  3.3 边界说明

  边界 1: 验证 vs 开发
  - 本清单只验证现有机制是否工作
  - 不负责开发新功能或修复 Bug
  - 但会记录发现的问题，供后续修复

  边界 2: 自动化 vs 手动
  - L1-L3 尽量自动化
  - L4 需要手动对话测试（Claude 交互无法完全自动化）

  边界 3: 完整性 vs 优先级
  - 不要求 100% 覆盖所有 skills
  - 聚焦 P0/P1，P2 可后续补充

  7. 技术调研

  7.1 验证方法论参考

  参考 1: 软件测试金字塔
        E2E 测试 (少量)
         ↑
     集成测试 (适量)
         ↑
    单元测试 (大量)

  映射到本项目:
  - L1/L2 静态检查 ≈ 单元测试
  - L3 运行测试 ≈ 集成测试
  - L4 实战验证 ≈ E2E 测试

  参考 2: 验收测试驱动开发 (ATDD)
  - 先定义 AC（验收标准）
  - 再编写验证脚本
  - 最后实现功能

  应用: 每个机制都有明确的 AC

  7.2 工具选型

  ┌──────────┬────────────────────────┬──────────┬────────────────────────┐
  │   需求   │        候选工具        │   选择   │          理由          │
  ├──────────┼────────────────────────┼──────────┼────────────────────────┤
  │ 脚本语言 │ Bash / Python          │ Bash     │ 简单、直接、无依赖     │
  ├──────────┼────────────────────────┼──────────┼────────────────────────┤
  │ 测试框架 │ pytest / bats          │ 无框架   │ 过重，直接用 bash 脚本 │
  ├──────────┼────────────────────────┼──────────┼────────────────────────┤
  │ 报告格式 │ JSON / HTML / Markdown │ Markdown │ 人类可读 + 可 diff     │
  ├──────────┼────────────────────────┼──────────┼────────────────────────┤
  │ CI/CD    │ GitHub Actions / 无    │ 暂无     │ 后续集成               │
  └──────────┴────────────────────────┴──────────┴────────────────────────┘
