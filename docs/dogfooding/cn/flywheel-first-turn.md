# 飞轮初转：一次狗粮如何反哺 Carror OS 自身

> **所属层级**: 4-数据投喂(血液养分层) — 狗粮反馈循环协议


> **哲学 #4「没通过验证等于没做」的自我证明**
> *当一个框架用来治理它自己的改进时，它就变成了活的系统。*

---

## 序：一次普通的拆解任务

2026年5月12日，一个大型前端项目（26 份 PRD、React/TypeScript 技术栈）启动了 PRD 分层拆解。目标是使用 `lx-oma-hier` skill 将 26 份主 PRD 拆解为 5 个功能域 Sub PRD。

这是 Carror OS 在日常项目中的一次普通调用。普通到没有人预期它会改变 Carror OS 本身。

---

## 第一幕：Oracle 亮起了红灯

拆解完成。5 个域（平台基建、生态资源、体验中心、可视化看板、配置器）结构完整，业务理解准确。看起来一切正常。

然后 Oracle 开始审查。

**第一轮，产物审查：3 个 FAIL。**

- Session 实体归属冲突 — D03 表格行声明「拥有」，D05 散文描述「管理生命周期」。*一个域说 Own，另一个用散文说也 Own，但 grep 脚本只看表格行*。
- 3 处 NFR 数字无来源 — 1500ms、10 tokens/s、2000ms 三个关键性能指标，翻遍主 PRD 找不到出处。违反铁律 #7。
- 依赖类型混淆 — 组件级代码复用在 INDEX.md 中被标成服务依赖，会误导开发顺序。

**第二轮，流程审查：43/100。**

不是 43 分的内容差，是 43 分的流程合规性。§3.2 MECE 检查表完全跳过、§8 Pipeline 集成未执行、§9 可观测性未写入、§11 人工审核门禁从未触发。AI 执行了内容，跳过了框架要求的所有验证步骤。

**第三轮，业务审查：又是 2 个 FAIL。**

知识库注册文档无域归属、D02 内聚性不足（熔断基础设施 + 资源注册 + 市场浏览混在一个域）。

三波审查，波波见红。

---

## 第二幕：狗粮的「回甘」

通常，故事到这里就结束了——发现问题，修正 Sub PRD，项目继续。Oracle 的任务就是做 QA。

但 Carror OS 有一条与众不同的原则：

> **狗粮反馈循环协议（AGENTS.md §狗粮反馈循环协议）**
> *核心循环：发现问题 → 修复问题 → 添加机制 → 传播修复到 source/ 和 packages/*

当 AI 代理发现 Carror OS 自身的问题时，它不只是在客户项目里修一修——它需要把机制改进传回 Carror OS 自身。

于是我们做了一个关键的分类决策：

| 发现 | 归属域 | 处理 |
|------|--------|------|
| Session 归属冲突 | 脚本验证盲区 — **框架问题** | → 写入 claude-next.md DG-01 |
| NFR 来源不可追溯 | 输入阶段验证缺失 — **框架问题** | → 写入 claude-next.md DG-02 |
| 手动/编排模式混淆 | Skill 设计缺陷 — **框架问题** | → 写入 claude-next.md DG-03 |
| checkbox 执行方式 | Skill 设计节奏 — **框架问题** | → 写入 claude-next.md DG-04 |
| 单轮 Oracle 覆盖不足 | 审查方法论 — **框架问题** | → 写入 claude-next.md DG-05 |
| D02 内聚性 | 项目特有业务逻辑 | → 跳过多，不反哺框架 |
| 评估工具独立性 | 项目特有产品决策 | → 跳过多 |

7 个框架级教训从客户项目的 20+ 发现中被分拣出来。

---

## 第三幕：框架本身的进化

然后，这些教训反哺到 Carror OS 的核心 skill 中。

`lx-oma-hier` 从 v1.2.0 升级到 v1.3.0，7 处修改：

| 章节 | 从 | 到 | 为什么 |
|------|---|----|--------|
| **§2** | 默认路径 + 无模式检测 | 执行模式检测 + 路径优先级规则（kernel.md > 用户指定 > 默认） | 狗粮发现 D02 的「路径三不符」问题和 DG-03 的模式混淆 |
| **§3.1** | 实体识别隐式完成 | 强制显式输出实体归属表 + 冲突预检 | Oracle 首轮就发现 Session 冲突——如果有显式输出就不会漏 |
| **§3.2** | 逐域 10 项 checkbox（50 次操作） | 全部域生成后统一 MECE 摘要表 | DG-04 — AI 不按人类「边做边查」的节奏执行 |
| **§3.3** | 依赖图无类型区分 | INDEX.md 增加「依赖类型」列（服务/代码） | P2 — 代码依赖标为服务依赖会误导开发顺序 |
| **§8** | 出口写入始终执行 | 手动模式跳过 + 报告注明 | DG-03 — 没有 pipeline.yaml 时不应报错 |
| **§9** | 无前置检查直接写入 | 环境检测 + 目录不存在时静默跳过 | DG-03 — 没有 .omc/state 时 telemetry 不报错 |
| **§11** | 有门禁章节但从未触发 | 强制每次输出审核清单 + 待裁决项表格 | Oracle 审查发现 §11 自始至终零次触发 |

与此同时，`claude-next.md` 增加了 10 条新教训（5 条 DG + 5 条 META），source 镜像同步更新。

## 第四幕：飞轮的数据

整个过程形成了一个完整的飞轮循环：

```
  客户项目（20+ 发现）
        ↓
  Oracle 审查（3 轮过滤）
        ↓
  分拣 → 7 个框架级教训 + 5 个元教训
        ↓
  lx-oma-hier v1.2.0 → v1.3.0（7 处修改）
        ↓
  claude-next.md +10 条教训
        ↓
  source 镜像同步（lx-skills-v5 + harness-kit）
        ↓
  下一次狗粮 → 飞轮再转
```

| 指标 | 数值 |
|------|------|
| 原始发现数 | ~20+ |
| 框架级教训数 | 7（DG-01~DG-05 + META-01~META-02 合并）|
| Skill 升级版本 | v1.2.0 → v1.3.0 |
| 代码变更文件 | 5（SKILL.md ×2 + claude-next.md ×2 + 狗粮记录）|
| 处理时间 | ~1 会话（含阅读、分析、修改、记录）|

## 终章：活着的系统

这个故事最有趣的地方不在于技术细节，而在于它证明了 Carror OS 的核心设计假设：

> **一个系统如果用来治理它自己的改进，就是活的。**
>
> ——哲学 #6（先天对 AI 0 信任）的终极物化

Carror OS 的 Oracle 在被调用时发现了 3 个 FAIL——其中 2 个是「自动脚本漏报了散文描述的实体冲突」和「NFR 来源未在输入阶段验证」。这两个问题不是客户项目的 bug，是 **Carror OS 框架自身的验证盲区**。

如果 Carror OS 只是一个被动的工具集，这两个盲区永远不会被发现。但因为它是狗粮模式——用它自己来治理它自己的开发——这些盲区在客户项目中暴露出来，然后立刻被反哺回框架本身。

这不是一个关于「项目 A 完成了 PRD 拆解」的故事。
这是一个关于「框架 X 在一次调用中发现了自己的盲区，并在同一次会话中修复了自己」的故事。

飞轮初转。这只是第一次。
