# Carror OS 新手尝试教材

## 第一次使用，从这里开始

---

## 你将得到什么

Carror OS 不是插件。不是 prompt 模板。也不是一个新的聊天工具。

**它是一个运行在 AI 工具之下的治理层。**

你继续用原来的工作方式。它在后台运行，在关键节点替你拦住那些最容易出问题的地方。

大多数时候，你不会明显感知到它的存在。你更可能感知到的是：
- AI 更不容易"假完成"了
- 敏感信息不会轻易被读进去
- 规则不会那么容易在长对话里漂移
- 新会话开始时，不需要反复重讲上下文
- 你没有以前那么频繁地核查 AI 的每一句话了

---

## 先理解两个版本

### 基础版（适合第一次使用的人）
- 安装后即可开始工作
- 大部分机制静默运行
- 基础版几乎不需要额外配置，也不需要先学完整体系
- 适合先体验"它在后台替你盯着"的感觉

### 进阶版（适合想深入掌控的人）
- 机制开关管理
- OMA 并发开发
- 更完整的工作流治理能力
- 更细的规则配置与知识沉淀

**建议：第一次使用，从基础版开始。先感受它，再决定要不要深入。**

---

## 第一步：安装

```bash
# 克隆到你的项目
git clone https://github.com/[carror-os]/harness-kit .claude

# 初始化
bash .claude/install.sh

# 验证安装
bash .claude/scripts/harness-smoke-test.sh
```

看到这个，说明你好了：

```
✅ harness-kit 已就绪
✅ 32 个 hook 已注册
✅ privacy-gate     已激活
✅ completion-gate  已激活
✅ error-dna        已激活
✅ context-guard    已激活
```

**就这些。** 打开你的 Claude Code，开始工作。

---

## 第二步：正常开始工作

安装完成后，不需要刻意测试所有功能。直接像平时一样打开 Claude Code，开始工作。

你不需要主动"调用"Carror OS。它的设计目标就是：**在你继续原有工作方式的同时，在关键动作发生前替你盯着。**

---

## 第三步：你第一次感受到它，通常会发生在这些时刻

### 场景一：AI 说"完成了"，但系统先要证据

你的任务快结束时，AI 准备标记 completed。这时你可能会看到：

```
⛔ COMPLETION BLOCKED

任务标记 completed，但没有找到验证证据。

请先运行验证，将结果写入证据文件：
  .omc/state/.completion-evidence-20260601

证据必须包含：
  · 实际运行的命令和输出
  · 明确的通过标记（exit 0 / PASS / ✅）
```

这不是普通报错。这是系统在替你问 AI：**"你真的做完了吗？请证明。"**

以前，"完成"常常靠 AI 自己宣告。现在，"完成"必须经过验证。

---

### 场景二：AI 想读敏感文件，但系统先拦住

当 AI 尝试读取 `.env` 或其他敏感配置时，你可能会看到：

```
🚫 [Privacy Gate 触发]

禁止直接读取敏感文件：.env

明文配置一旦进入 AI 上下文，
就已经不再安全。

请通过环境变量注入。
```

这意味着敏感信息没有进入 AI 上下文。你不需要事后担心"它是不是已经看到了"。系统在风险发生之前就拦住了。

---

### 场景三：对话已经很长了，但规则没有被忘掉

在第 20 轮甚至更长的对话里，AI 往往开始漂移。而 Carror OS 会在关键动作前，把规则重新锚定：

```
📌 [第20轮·规则锚定]
①禁止编造——技术断言必须引用 file:line
②完成前需 VERIFIED 证据
③git 操作需用户批准
④只改当前任务范围
⑤同一问题最多修 3 轮
```

这不是 AI 自己突然变得更可靠。而是系统在它即将写代码前，把该记住的事情重新放回它眼前。

---

## 第四步：第一次新会话

Carror OS 的价值，不只体现在当前会话。它也体现在"第二天继续工作"的时候。

当你关闭对话、第二天重新打开新会话时，你可能会看到：

```
[上次会话快照]
分支：feat/ecosystem-tab
未提交修改：src/pages/ecosystem/index.tsx
下一步：完成 ResourceCard 组件的 hover 状态

[错误记忆]
未解决的错误：
  · [3次] tsc: TS2345 — src/types/ecosystem.ts:47
  · [2次] MSW handler 未注册导致 404
```

这意味着：
- 上次做到哪里，被接住了
- 哪些坑已经踩过，被记住了
- 你不需要从头把背景再讲一遍

**直接继续工作就行。**

---

## 第五步：一周后，你会真正意识到变化

用了一周之后，你可能会发现一个变化：**你没有以前那么频繁地去核查 AI 了。**

不是因为你盲目更信任它。而是因为你知道：
- 没有证据，它过不了 completion gate
- 敏感文件读不到
- 长对话里规则会被重新注入
- 老错误和未完成事项会被接住

那些以前一直悬在背景里的"不知道"，少了一些。不一定全部消失。但少了一些。**这就是基础版最重要的价值之一。**

---

## 第六步：如果你想要更多，再进入进阶版

当你已经熟悉基础版的工作方式后，再去读进阶文档会更自然。

进阶版包括：
- **OMA 一人成军**：一个人用多个终端并发开发
- **lx-rpe 9步主循环**：更系统化的特性开发流程
- **flywheel 飞轮机制**：从错误到改进的治理闭环
- **知识升华路径**：把一次次经验沉淀成长期可复用的规则与资产

你不需要一次全学会。也不需要一开始就全部打开。**先开始用，再逐步掌控。**

---

## 使用后的三种可能

### 第一种：你感受到了它的价值

也许是在某个深夜，completion-gate 拦住了一次"假完成"。也许是在新会话开始时，错误记忆让你避开了上周踩过的坑。也许是某个周末，你发现自己没有像以前那样把工作的焦虑带进生活里。

如果这些发生了——欢迎留下来，一起把它做得更好。

---

### 第二种：你还没明显感受到

也许你还没遇到最典型的场景。也许你的工作方式和它的设计还没有完全贴合。没关系。继续用。

Carror OS 的很多价值，本来就不是"立刻炫技式出现"的。它更像是一种在关键时刻接管风险的基础设施。

---

### 第三种：你用了，但没有感受到解脱

如果你用了，但仍然很累。仍然很焦虑。仍然觉得高压没有减轻。

**请不要捐款。** 告诉我们哪里不够好。

这个反馈，比任何支持都更重要。因为 Carror OS 存在的理由，不是让功能列表变长，而是让你真的能从高压工作中回归一点生活。如果没做到，它就还不够好。

**我们需要知道，然后把它做好。**

---
