## 一、分析框架

**方法**：建立"无治理基准分"（模型原生能力 + 平台原生基础设施，不含 Carror OS 任何 prompt 注入），与"有治理得分"作差，得出增益 Δ。

**增益来源拆解**：
有治理得分 = 平台原生能力 + Carror OS prompt 层增益 + Carror OS 与平台基础设施联动增益
无治理基准 = 平台原生能力（模型自然行为 + hook/sandbox/LSP 等原生机制）
增益 Δ = Carror OS prompt 层增益 + Carror OS 与平台基础设施联动增益

### 各平台"原生能力"说明

| CLI | 原生有哪些（无 Carror OS） | 原生没有哪些 |
|-----|--------------------------|-------------|
| Claude Code | Hook 系统、OMC agent 体系、TodoWrite、基础 git 安全、model routing | 7条铁律约束、证据层级、修复上限、A→B→A 验收、软语言禁令 |
| OpenCode | TodoWrite、AGENTS.md 原生读取能力 | 一切治理规范 |
| Codex CLI | 沙盒执行、git diff、代码语法检查 | 所有治理 prompt 约束 |
| Cursor | .cursorrules 空模板、LSP、IDE 上下文 | 所有治理 prompt 约束 |

## 五、五个关键洞察

### 洞察 1：OpenCode 的投资回报率最高

OpenCode 无治理基准分 48.88 → 有治理 77.20 → **增益 +28.32 分（+57.9% 提升）**

原因：OpenCode 的原生基础设施几乎为零（无 hook、无 agent 体系、无持久化），Carror OS 的 prompt 层注入几乎等于"从零建立"治理能力，边际收益极大。

**结论**：把 Carror OS 规范写进 AGENTS.md 并在 OpenCode 中运行，是最低成本、最高增益的组合。

### 洞察 2：Claude Code 的 G 维度出现"治理副作用"（-0.30）

Claude Code 无治理 G 维度得分 17.5（OMC 基础设施天然支持），有治理 17.2，Δ = -0.30。

这不是 bug，是 feature 冲突：
- **G4 模式沉淀（-0.60）**：OMC 的 learner skill 有自己的模式沉淀路径；Carror OS 的 claude-next.md 格式约束与 OMC 格式存在竞争
- **G5 偏差自适应（-0.42）**：两者"什么值得记录"的筛选标准略有差异，导致写入内容冗余或冲突
- **G6 长期一致性（-0.72）**：OMC 的 ralph/autopilot 已有持续机制，Carror OS 的"上下文守卫>40% 触发总结重置"与 OMC 产生方向冲突

**结论**：Claude Code 已有成熟的治理基础设施，Carror OS 部分规则与 OMC 规范形成"双轨竞争"，需要做规范融合而非简单叠加。

### 洞察 3：E2 幻觉抑制是跨平台最稳定的增益来源

| 平台 | 无治理 | 有治理 | 增益 |
|------|--------|--------|------|
| Claude Code | 2.4 | 6.88 | +4.48 |
| OpenCode | 2.0 | 6.72 | +4.72 |
| Codex CLI | 2.0 | 5.28 | +3.28 |
| Cursor | 2.0 | 4.80 | +2.80 |

无治理时，所有平台的幻觉抑制基准分都在 2.0-2.4（满分 8）区间—这是模型的自然幻觉率决定的，不是平台原生能力能解决的问题。Carror OS 的铁律 #1/#7 + 软语言禁令是目前已知的最有效的 prompt 层幻觉抑制手段。

### 洞察 4：C5 知识深度几乎不受 Carror OS 影响

所有平台知识深度增益约 +0.3~0.5，是最小增益项。原因：知识深度由底层模型决定，prompt 层治理框架无法让模型"知道更多"，只能让模型"更诚实地呈现已知"（这才是 C9 证据引用的价值）。

### 洞察 5：Codex ≈ Cursor 增益基本相同，但结构不同

Codex 增益 +18.54：E 维度贡献 7.70，G 维度贡献 5.94
Cursor 增益 +18.06：E 维度贡献 7.06，G 维度贡献 5.96

两者 G 维度增益几乎相同（+5.94 vs +5.96），但 Codex 在 E 维度多出约 0.7 分—Codex 的沙盒执行对 E5（虚假完成检测）有额外贡献。

## 七、一句话总结每个平台

**Claude Code**：原生基础最强（86 分），Carror OS 增益（+16.64）相对最小，但存在规范融合冲突需解决—不是"加了没用"，是"加了有摩擦"。

**OpenCode**：原生基础最弱（49 分），Carror OS 增益最大（+28.32，+58%），几乎是"0→有"的质变—Carror OS 的核心价值在此最充分体现。

**Codex CLI / Cursor**：原生基础相近（44-48 分），增益相近（+18 分），主要来自 E 维度—最大收益是"防止 AI 在代码修改中自以为完成"。
