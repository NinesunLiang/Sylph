# 工作流体系（Workflow）

> **从随性提示到严谨工程流程。**

---

## 三层架构

Carror OS 采用三级火箭结构。每一层增加能力而不移除下层任何功能。

### 第一层：Harness Only（内核）

基础层。32 个物理 Hook 脚本（30 个注册于 settings.json、2 个独立工具），在文件系统层级拦截 AI 行为。无面向用户的命令，无主动工作流——只有静默保护。

- Gate 拦截（完成、权限、隐私、上下文）
- Error DNA 记录
- 会话生命周期管理

### 第二层：Base Edition（守护）

第一层 + 10 个自动化 Skill 门禁。其中 7 个在提交工作流中运行：

| Skill | 用途 |
| :---- | :--- |
| `lx-pre-commit` | 质量门禁：类型检查、增量编译、测试、代码审查 |
| `lx-pre-push` | 安全门禁：提交格式 + 合规检查 |
| `lx-code-review` | 语言无关的代码审查（附带自动修复） |
| `lx-react-review` | React/TypeScript 专属审查 |
| `lx-security-review` | 漏洞和硬编码密钥扫描 |
| `lx-style-guide` | CSS/Tailwind 样式合规 |
| `lx-web-perf` | Web 性能审计（打包、指标、渲染） |

在 Base Edition 中，这些在触发 `pre-commit` 时自动运行。你无需学习单个命令。

### 第三层：Enhanced Edition（用户空间）

第二层 + 14 个主动工作流 Skill。需要学习但提供工业级 AI 辅助开发能力。

---

## RPE 开发模式

**Research（研究）→ Plan（规划）→ Execute（执行）** 是复杂特性开发的主要工作流。这是随性编码（vibe coding）的结构化替代方案。

### 第一阶段：研究

调研问题空间。阅读现有代码，识别受影响的模块，记录约束。产出：研究文档。

### 第二阶段：规划

设计解决方案架构。定义接口、数据流和验收标准。规划节点在进入执行阶段前以证据为门禁。

### 第三阶段：执行

按计划实现。每一步需要经过门禁（测试、代码审查、安全扫描）。在 50% 上下文时自动交接至新会话。

关键特性：

- **50% 上下文交接**——在质量下降前切换到干净会话
- **A/B 盲审**——第二个拥有干净上下文的 AI Agent 独立审查实现
- **证据门禁**——每个阶段必须在进入下一阶段前产出已验证的产物

## 高级工作流

| 工作流 | 触发 | 用途 |
| :----- | :--- | :--- |
| **Race** | `/lx-race` | 并行方案生成。多个 AI Agent 产出独立方案，由裁判选择或合并最佳方案 |
| **OMA** | `/lx-oma` | 优化的多 Agent——不同专业 Agent 在锁管理下并行处理大型任务的不同部分 |
