# 审计追踪（Audit Trail）

> **每一次 AI 行为都被记录、不可篡改、可回放。**

---

## 为什么要审计？

当 AI 自主编写代码时，你需要**法医级证据**来回答：

- AI 读取了生产环境凭据吗？
- AI 删除了或修改了任务范围外的文件吗？
- 会话交接保留了完整的决策历史吗？

信任不是感觉，是账本。

---

## 核心组件

### read-tracker（读取追踪）

记录 AI 在会话中读取的每一个文件——路径、时间戳和目的。为安全审查提供完整的访问日志。

- 位置：`.omc/state/read-tracker.txt`
- 格式：`时间戳 | 文件路径 | 触发原因`

### turn-counter（轮次计数器）

追踪当前会话中每一次 AI 交互。由 `context-guard` 用于估算上下文消耗并触发交接或熔断。

- 阈值 1：50% 轮次 → 主动交接信号
- 阈值 2：80% 轮次 → 硬停止

### session-snapshot（会话快照）

每次交接或终止时写入快照，包含：

- 完整轮次历史摘要
- 会话中修改的文件
- 触发的 Gate 事件
- 当前任务进度

快照使会话可以不从头开始恢复。

- 位置：`.omc/state/sessions/<session-id>/`

## 审计面板

`/lx-status` 命令渲染三面板审计面板：

| 面板 | 数据 |
| :--- | :--- |
| Token 节省 | 渐进式披露 vs 全量 System Prompt 节约的 Token 估算 |
| 自愈率 | Error DNA 匹配率——重复出现的错误有多少被自动修正 |
| 执行画像 | Gate 事件、文件修改、会话交接的时间线 |

---

## SHA256 完整性

所有审计记录在写入时以 SHA256 哈希。如果记录在创建后被篡改，哈希不匹配可在验证时发现。

这确保：

- 记录不会被未来的 AI 会话静默修改
- 安全审计员可以验证整条链的完整性
- 关键操作（凭据访问、破坏性命令）的不可否认性
