# V. 10 Suggestions Ranked by Importance

Below are what I consider the **ten most important suggestions**, ordered by priority.

---

## 1) Downgrade All Strong Claims to "Evidenciable Claims"
### Why This Matters
This is the foundation of credibility.

### What to Do
Unify the treatment of the following expressions:
- 19,280 tokens/session
- ~75% savings
- 100% feature completeness
- Fully visible
- True concurrency
- Industry-first
- Claude default tokenizer

Change them to:
- Verified / Pending Verification / Conceptual / Needs Benchmark Support

---

## 2) Do Repository Reality Check Before Large-Scale Documentation and Code Overhaul
### Why This Matters
Your knowledge base has already shown the risk of doc-code divergence.

### What to Do
- First verify canonical files
- First check for empty files, mismatched files, outdated references
- Then let the local model iterate

---

## 3) Establish a Unified Claim Registry
### Why This Matters
This transforms the knowledge base from "narrative" to "governance."

### What to Do
Annotate each claim with:
- Status
- Evidence level
- Whether it is public-facing
- Whether it requires benchmark
- Whether it is internal research only

---

## 4) Make Feature Registry a Machine-Readable Master Table
### Why This Matters
This resolves state inconsistency among hooks, skills, and editions.

### What to Do
Unify the recording of:
- Whether enabled by default
- Config key
- Version belonging
- Code path
- Evidence source
- Probe method

---

## 5) Prioritize Benchmark Over Narrative Stacking
### Why This Matters
The aspect of your product most vulnerable to challenge is not the concept, but the numbers and results.

### What to Do
Prioritize verification of:
- Whether progressive disclosure actually saves tokens
- Whether race mode is effective
- Whether proactive handoff actually triggers
- Whether OMA is actually stable
- Whether error-dna is actually usable

---

## 6) Build the System Governance Evidence Chain Before Product Marketing
### Why This Matters
Carror OS's core moat is governance.

### What to Do
Turn these capabilities into demonstrable evidence:
- Gate
- Audit
- Snapshot
- Error DNA
- Context control
- Handoff
- Lock lifecycle

## 7) Improve Usability, Lower the Expert Barrier
### Why This Matters
The current product is strong, but "too strong" also creates comprehension barriers.

### What to Do
- Front page should only explain what it is, why it matters, and how to start
- Deeper documentation can address philosophy and architecture
- Reduce first-screen terminology density
- Create persona-based navigation

## 8) Change External Documentation to "Methodology Review" Rather Than Self-Scoring
### Why This Matters
This is a brand credibility issue.

### What to Do
- Remove "self-scoring" tone
- Remove internal reasoning-style commentary
- Switch to public methodology, acceptance data, reproducible citations
- Separate internal scores from external scores

---

## 9) Strengthen Visualization and Agentic UI
### Why This Matters
Governance capabilities are hard for users to perceive if they exist only in text.

### What to Do
Prioritize:
- lx-status trend charts
- Error DNA distribution
- Flywheel timeline
- Numbered-choice menus
- Audit dashboard

---

## 10) Clarify the Commercial Path as "Open Source Base Layer + Professional Enhancement Layer + Consulting/Enterprise Trial"
### Why This Matters
This best fits your current product form.

### What to Do
Define three-layer commercial path:
- Open source base layer: build trust
- Enhanced: sell governance capabilities and efficiency
- Consulting / trial: sell implementation and experience
