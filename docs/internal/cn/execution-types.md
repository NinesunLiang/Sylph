# 执行类型路径定义（lx-todo）

## Bug Fix 路径

**行为约束**：加载 `@../../.claude/nodes/behavior_rules.md`（证据门禁 + Git 门禁 + 修复上限）。

**根因方法**：加载 `@../../.claude/nodes/execute_node.md`（5-Why 快速根因法 + 降级触发矩阵）。

本 skill 的修复上限为 **2 轮**（比通用 3 轮更严格，因 todo 是轻量模式）。

```
1. readFile 相关代码 ±30 行上下文
1a. 引用原始代码（必须粘贴 readFile 输出中与 bug 直接相关的代码 ≤5 行原文，标注文件:行号）
2. 根因判断（30 秒规则）：
   ├─ 明显（满足以下任一）→ 直接修复（最小化变更）：
   │  · 缺 nil 检查 / 漏 error / 类型错误 / 边界缺失
      仅 Phase 1（定位）+ Phase 4（修复），max 2 轮
3. go build ./... 编译通过
4. 症状/根因自检（lx-todo 特有）：
   - 本次修复是否仅在调用点加了防御？
     - 是 → 检查是否有其他调用点同样缺少防御
       - 有 → 系统性问题，升级到主分支使用 /root-cause-analysis
       - 无 → 单点防御可接受，继续 Step 3
     - 否 → 修复针对数据产生源头，继续 Step 3
```

**升级条件**：debug 2 轮仍未定位根因 → **必须升级**，不可继续尝试。传递上下文：已收集的证据 + 已排除的假说。

## Feature 路径

```
1. grep 现有同类代码，提取模式
1a. 引用参考代码（必须粘贴 grep/readFile 输出中被复用的模式代码 ≤5 行原文，标注文件:行号）
1b. 模式适用性检查：现有模式是否适用于当前场景？
    - 适用（输入/输出类型匹配 + 无已知缺陷）→ 复用
    - 不适用（类型不匹配 / 现有模式本身有问题 / 当前场景有特殊约束）→ 从零实现（仍须 ≤3 文件）
    - 不确定 → readFile 现有模式的调用方和测试，确认后再决定
    └─ 工具降级：grep 无结果 → AST grep → readFile 手动查找
2. 复用或从零实现，最小化变更
3. go build ./... 编译通过
```

**升级条件**：实现过程中发现需要新增接口/修改 proto → **必须升级**。

---

## Refactor 路径

```
1. 确认测试覆盖：
   go test -v ./affected/package 2>&1 | tail -5
   ├─ 有测试 → 继续
2. 小步重构，每步 go build ./... 编译通过
2a. 引用重构变更（必须粘贴修改前后的关键代码各 ≤3 行，标注文件:行号）
3. 运行已有测试：go test ./affected/package
```

**升级条件**：重构导致测试大面积失败（>3 个） → **必须升级**。

---

## Docs 路径

```
1. 直接修改文档/注释
2. → 跳到 Step 4（无需 Step 3 验证）
```

---

## 全类型完成标准

- 关键代码已引用（Docs 除外）
- `go build ./...` 编译通过（Docs 除外），引用实际编译输出
- 变更文件 ≤3（`git diff --name-only | wc -l` 验证）
- 变更文件数 >3 → **立即停止**，升级 lx-task-spec
- debug/fix 重试 ≤2 次（见降级触发矩阵，本 skill 上限 2 轮）
- 证据门禁 + 置信度标注
