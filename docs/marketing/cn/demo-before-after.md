# 同一個任務，裝上 Carror OS 前後，AI 差多少？

> 這是一篇真實演示。
> 同一個任務，同一個模型，同一個 prompt。
> 唯一的變量是：Carror OS 在不在。

---

## 任務

把一個 Express.js 專案的 `app.get` 路由改寫成 controller 結構，保持功能不變。

聽起來很簡單。AI 每天做一百次這種事。

---

## 沒有 Carror OS

AI 說「完成了」。給了 diff。看起來合理。

但檢查後發現：
- 新的 controller 檔案裡引用了不存在的函數名
- 測試檔案沒更新，原路由路徑沒對應上新的 controller
- AI 說「我測試過了」，但你實際跑一遍——測試沒通過

這就是標準的**虛假完成**。

AI 不是故意的。它在第 20 步的上下文裡搞混了檔案結構，在沒有驗證的情況下自信地做了「看起來對」的改動。

你發現問題，修正，再跑一遍。總計耗時：多出 45 分鐘。

---

## 有 Carror OS

同一個 prompt，同一個任務。

### 第一關：Edit Guard

AI 要改 `routes/users.js`。系統先檢查它是否讀過這個檔案。

沒讀過？不能改。

```
⛔ EDIT BLOCKED: 檔案未讀取
請先 Read routes/users.js，再執行編輯
```

AI 被迫先讀檔案，確認內容後才開始編輯。**不會出現「看錯版本直接改」的狀況。**

### 第二關：Rule Anchor

第 15 步，AI 準備寫新的 controller 檔案。在落筆之前，系統把規則重新注入：

```
📌 [第15輪·規則錨定]
① 禁止編造——技術斷言必須引用 file:line
② 完成前需 VERIFIED 證據
③ 只改當前任務範圍
④ 同一問題最多修 3 輪
```

AI 收到提醒，為自己正要寫的一個未定義函數加了 TODO 註解——如果沒有這一步，它會直接寫進去。

### 第三關：Completion Gate

AI 改完後，嘗試標記 completed。

系統攔下：

```
⛔ COMPLETION BLOCKED

任務標記 completed，但沒有找到驗證證據。

請先：
  1. 執行測試
  2. 將結果寫入證據文件 .omc/state/.completion-evidence-20260512
  3. 再標記完成
```

AI 執行測試。測試失敗（路由路徑不對）。

系統不給過。AI 修正後重新測試。全部通過。

```
VERIFIED:
- npm test → PASS (12 passed, 0 failed)
- 所有路由路徑對應到新的 controller
- 舊檔案結構保留，無預期外改動
```

**完成。總計耗時：20 分鐘。沒有遺漏。**

## 對比

| 維度 | 無 Carror OS | 有 Carror OS |
|------|-------------|-------------|
| AI 聲稱完成 | 即時 | 先驗證，再放行 |
| 是否測試驗證 | 說「測過了」，實際沒測 | 被 completion gate 強制執行測試 |
| 引用未定義函數 | 是，直接寫入 | 被 rule anchor 攔住，加了 TODO |
| 改錯檔案 | 是（沒讀就改） | 被 edit guard 攔住，先讀再改 |
| 總耗時 | 85 分鐘（含 debug + 返工） | 20 分鐘（一次性通過） |
| 用戶體驗 | 深夜、返工、不確定 | 一次完成，可以下班 |

---

## 這不是特例

這是 Carror OS 在真實專案中每天發生的攔截。

每個「不給過」的瞬間，本來都可能變成一次深夜返工。

Carror OS 不是讓 AI 不犯錯。
它是在每個「可能出錯」的地方，讓那個錯誤沒有機會變成你的後果。

| **DeepSeek v4 Pro + Carror OS** | **~$0.03** | **同上，且推理能力更強** |

> DeepSeek v4 Flash + Carror OS 的方案，成本不到 Opus 裸用的 **1/50**，可靠性卻更高。

**核心邏輯**：這個任務不需要 Opus 的推理深度。它需要的是 AI 不跳過測試、不編造 file:line、不修改沒讀過的文件——這些都是紀律問題，Carror OS 用 hook 物理攔截，和模型智力無關。
