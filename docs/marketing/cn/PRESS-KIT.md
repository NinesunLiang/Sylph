# Carror OS — 给 AI 装上刹车

> **一句话**：Carror OS 是第一个在 Hook 应用层面对 AI 进行物理行为约束的开源治理框架。它不是"建议"，是"拔电源"。

---

## 问题

2026 年，AI 辅助编程已经成为主流。但每个工程师都经历过：

- AI 在你没注意时执行了 `rm -rf /var/www`
- AI 说"搞定了"，但什么都没做
- AI 把你的 `.env` 文件内容发给了云端 API
- 长对话 30 轮后，AI 开始改它不该改的文件
- 同一个错误修了 8 次，每次都说"这次应该没问题了"

**现有方案全部失效**——Cursor Rules、Claude Code Hooks、Copilot Instructions 都是在 Prompt 层说话，靠"请"和"建议"。AI 可以礼貌地忽略它们。

---

## 解决方案

Carror OS 选择了完全不同的路径：**不跟 AI 说话，直接拦截它的工具调用**。

```
正常流程:  用户 → AI → 工具调用 → 执行
Cursor:    用户 → AI → [Prompt 建议] → 工具调用 → 执行
Carror OS: 用户 → AI → [⚠️ Hook 拦截] → 拒绝 / 授权 / 审计
```

### 三层架构

| 层 | 做什么 | 怎么做的 |
|------|------|---------|
| **Harness**（内核） | 32 个应用层 Hook，在 AI 工具调用前拦截 | `Exit 2` 硬阻断，不是 Prompt 请求 |
| **Skills**（用户态） | 24 个工作流 Skill，从 PRD 到提交的全周期 | Markdown 指令集 + 19 个 Python 工具 |
| **Profiles**（桥接） | 5 种语言环境的一键适配 | YAML 配置，零代码切换 |

## 关键能力

### 1. 你在 AI 要 `rm -rf` 时拔了它的电源

32 个 Hook 覆盖六大安全域：

| 域 | 代表 Hook | 拦截什么 |
|------|----------|---------|
| 权限门禁 | `permission-gate.sh` | `rm -rf`、`DROP TABLE`、`git push --force` |
| 隐私防线 | `privacy-gate.sh` | `.env`、`*.pem`、`id_rsa`、明文 Token |
| 上下文熔断 | `context-guard.sh` | 上下文 ≥80% 时物理锁死一切写入操作 |
| 交付验收 | `completion-gate.sh` | AI 无 VERIFIED 证据声称"搞定了" |
| 读写时序 | `edit-guard.sh` | 未读文件不允许编辑 |
| 范围冻结 | `pretool-edit-scope.sh` | 超出任务范围的文件修改 |

**所有 Hook 经过 L1-L4 四层测试体系（手动验收 + 自动 Hook 校验 + 代码扫描 + 格式门禁）全部通过，通过 ShellCheck/Bandit 真实安全扫描（0 真实业务缺陷），对照 OWASP ASVS v4.0.3 / MITRE ATLAS / NIST AI RMF 1.0 三项行业标准完成自评合规对照（75/75 覆盖）[内部自评，非第三方认证]。**

### 2. AI 全程不知道你的真实密钥

`varlock` 是开源社区的双向脱敏代理：

```
AI 看到的:    curl -H "Authorization: {API_KEY}"
真实执行的:   curl -H "Authorization: sk-ant-abc123..."
AI 写回的:    API_KEY={API_KEY}
文件存储的:   API_KEY=sk-ant-abc123...
```

AI 始终在数据隔离沙箱中工作，从未接触过任何明文凭证。

### 3. 长对话不再是"慢性智障"

三层防漂移机制自动运行，不需要你手动 `/compact`：

| 层级 | 触发条件 | 动作 |
|------|---------|------|
| SessionStart | 每次新会话 | 铁律注入上下文 |
| 每 10 轮 | 轮次计数器 | 7 条思想钢印复诵 |
| 每次写文件 | 写操作前 | 范围冻结 + 规则锚定 |
| 50% 甜点区 | 上下文过半 | 强制交接，状态最干净时重置 |

### 4. 不是说"搞定了"，先拿证据来

`completion-gate.sh` 要求 AI 提供 `>= 20 字符` 的 VERIFIED 证据文件才能标记任务完成。没有测试日志、没有输出截图、没有验证命令 → 任务永远处于"进行中"。

### 5. 你说一句话，它长出你需要的技能

Carror OS 不只是「盾」——它是活的生态系统。`/skillify` 命令接收自然语言描述，自动产出通过 11 条原子化规则验证的完整技能。

**真实狗粮记录（2026-05-16）**：用户说「`/skillify` 一个描述就能生产一个专业 skill」，当时 `skillify` 和 `learner` 还是空壳名字。二十分钟后，它们变成了两个通过全部验证、注册在案、带完整脚本和参考文档的生产级技能。用户零代码干预——系统自主探索 29 个已有 skill 找架构模式、按 TEMPLATE.md 生成 SKILL.md、写 Python 脚本、跑 syntax check、跑 lx-validate-skill、注册到生态。

这意味着 Carror OS 的能力边界不是预装的——是从你的使用中生长的。今天没有「审查 Dockerfile」的 skill？描述一下就有了。`/learner` 更进一步：它在对话中观察你的重复操作，主动提议提取为可重用 skill，附带来源证明。

这是 Carror OS 说的「生态」：不是装了多少插件，是**能不能自己长出新器官**。

## 模型成本自由：用更便宜的模型，获得高端可靠性

**AI 编码的真正瓶颈不是模型智力，是可靠性。** 

日常开发 90% 的任务——CRUD、重构、bug 修复、测试编写——并不需要顶级模型的深度推理。它们需要的是一套可靠的纪律系统：不编造 file:line、不在没验证时说"完成了"、不在上下文漂移时写出偏离目标的代码。

| 方案 | 模型成本（每百万 token） | 可靠性保障 |
|------|------------------------|-----------|
| Opus 4.6 裸用（仅 CLAUDE.md） | ~$15 | 依赖模型自觉 |
| **DeepSeek v4 Flash + Carror OS** | ~$0.27 | **物理 hook 拦截 + 双审验证** |
| **DeepSeek v4 Pro + Carror OS** | ~$1.10 | **物理 hook 拦截 + 双审验证** |

> DeepSeek v4 Flash 的价格不到 Opus 的 **1/50**。Pro 不到 **1/13**。

Carror OS 在三个层面弥补了模型能力差距：

1. **纪律补偿智力** — 41 次显式 hook 拦截事件（`skipped-errors.md` 记录），拦截的全部是纪律问题（虚假完成、危险命令、上下文漂移），不是智力问题。更聪明的模型一样会犯这些错。

2. **双审补偿盲区** — Oracle + Meta-Oracle 独立 agent 审查体系，两次审查发现的问题**零重叠**。用算力换覆盖率，弥补单模型审查盲区。

3. **记忆补偿遗忘** — 76 条实战教训跨会话累积注入，knowledge-condenser 自动将高频教训升华为永久规则。知识随使用时间增长，不依赖模型版本。

**谁最适合这个方案：**
- **个人开发者** — DeepSeek v4 Flash 的价格，获得接近 Opus 级别的日常可靠性
- **小团队 / 初创公司** — 用 DeepSeek v4 Pro 覆盖 90% 日常任务，Opus 只留给架构决策和复杂算法
- **任何对 AI 账单敏感的人** — Carror OS 的渐进式加载 + 上下文压缩在实际会话中节省 14%-53% token（20 轮实测数据，`benchmark-report.md:178-179`）

> **模型是引擎，Carror OS 是刹车、安全带和气囊。** 一辆装了安全系统的家用车，比一辆没有刹车的赛车更能让你安全到达目的地。

## 差异化竞争力

以下设计在开源社区中**没有已知同类实现**：

1. **varlock 双向脱敏代理** — AI 全程在数据隔离沙箱中工作
2. **completion-gate 证据门禁** — 物理阻止 AI 无证据声称完成
3. **三层防漂移机制** — 从 SessionStart 到每次写文件的全周期覆盖
4. **50% 甜点区主动交接** — 在 AI 状态最干净时强制重置
