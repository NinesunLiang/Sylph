# 已知限制

> Carror OS 建立在"先证据，后完成"的原则上。项目将自己同样的标准应用于自身的声明。
> 本页记录了当前的限制，以便用户做出知情决策。

---

## 架构

| 限制 | 影响 | 状态 |
|------|------|------|
| Token 节省数字依赖于基准测试 | 不应被视为通用保证 | ⏳ 基准测试待定（RPE-002） |
| 上下文监控依赖本地文件追踪，而非 API 级别的 Token 数据 | Token 百分比是近似值 | ⏳ 正在加固（RPE-003） |
| 高级工作流（RPE、Race）需要手动调用 | 不适合被动用户 | ✅ 设计如此 — 基础版/进阶版分离 |

---

## 门禁与钩子

| 限制 | 影响 | 状态 |
|------|------|------|
| Completion-gate 依赖自然语言触发检测 | 可能无法捕获所有形式的假完成 | ✅ 正常工作——宁可误判不可漏判 |
| Context-guard 50% 主动交接需要步骤完成检测 | 可能在部分会话模式中不触发 | ⏳ 正在加固（RPE-003） |
| Privacy-gate 使用关键词+模式匹配进行 DLP | 可能无法检测混淆或编码后的密钥 | ✅ 设计如此——模式集可扩展 |
| Permission-gate 使用正则模式扫描 | 非常规破坏性命令可能不匹配默认规则 | ✅ 设计如此——规则可在 harness.yaml 中配置 |

---

## 审计与可观测性

| 限制 | 影响 | 状态 |
|------|------|------|
| 审计追踪来自多个源；某些源不可用时可能显示降级状态 | 看板可能显示不完整数据 | ✅ 设计如此——兼容降级 |
| 读文件日志无界增长 | 长会话可能产生大日志文件 | ⏳ 轮转待定（RPE-003） |
| 无图形化 Web 看板 | 审计复查仅限 CLI 通过 lx-status | ✅ 设计如此——CLI 原生 |

---

## 高级功能

| 限制 | 影响 | 状态 |
|------|------|------|
| Race 模式是一种编排模式，不是确定性并行执行 | 不保证并发任务完成 | ✅ 已记录为设计选择 |
| OMA 锁生命周期加固正在进行中 | 复杂的多步骤协调可能遇到过期锁 | ⏳ 正在加固（RPE-014） |
| Error DNA 跨会话记忆依赖本地 JSONL 文件 | 无云同步或团队共享错误数据库 | ✅ 设计如此——本地优先 |

---

## 平台支持

| 限制 | 影响 | 状态 |
|------|------|------|
| 主平台：Claude Code | 其他 CLI 工具（Codex CLI、Gemini CLI）的 Hook 支持可能不完整 | ✅ 已记录为 Claude Code 优先 |
| Windows：仅限 WSL | 不支持原生 Windows Shell | ✅ 设计如此 |
| 性能：Hook 脚本每次工具调用增加约 10-50ms | 对大多数操作可忽略；在快速工具序列中可能可感知 | ✅ 可接受的权衡 |

---

## 为什么要公开限制

1. **先证据，后完成** — 我们将同样的标准应用于 Carror OS 自身
2. **信任** — 透明的限制比过度承诺更能建立信任
3. **清晰** — 帮助用户判断 Carror OS 是否适合他们的工作流
4. **聚焦** — 限制指导我们的开发优先级

---

## 如何处理限制

- **查看当前状态**：每个 RPE 周期审查；在此页面更新进展
- **提交问题**：如果某个限制阻塞了你的工作流，请告诉我们
- **贡献代码**：Carror OS 是开源的 — 欢迎解决限制的 PR
- **变通方案**：大多数限制在相应的功能文档中有记录的变通方案
