# 我的真实看法

我坚定支持这个方向，但我认为**当前最危险的不是模型能力不够，而是你们可能会把自治目标定得太早、把治理风险压得太晚**。

基于现有 `AGENTS.md` 可以确认的事实，我会更偏“保守自治、先闭环再提速”：技术栈明确是 **Vite 8 · React 19 · TS 6 strict · Sass + CSS Modules · React Router v7 · Zustand v5 · Axios v1**，不是 antd 驱动；UI 还原已经规定 Playwright 精测 + chrome-devtools 对比，颜色必须走 `src/styles/tokens/`；C7 红线也已经给出文件体量与反魔法数约束。[已验证:AGENTS.md:项目概览/UI还原铁律/C7红线]

下面是我真正的判断。

---

## 1. 这件事值得做，但先别幻想“全面替换前端”

可行，而且窗口很大。原因不是模型突然聪明了，而是你们已经有一整套相对完整的治理底座：

- 项目宪法权威链明确
- 任务模式与执行引擎已定
- 置信度标记已定
- UI 工具链已定
- 页面目录和数据层边界已有雏形

这比“从头教 AI 写前端”强得多。真正的跃迁，不是让模型更会写页面，而是让组织接受一种新的交付单元：

> **PRD + API + 原型 + 验收证据**，而不是“手写组件数量”。

但我反对一上来就压 95%/90%/80% 作为运营 KPI。前期更应该盯：

- 错误完成声明率
- 越界修改率
- 状态可恢复率
- 输入缺陷识别率
- 同失败无效重试率

这些指标健康后，95% 才会自然上来。顺序反了，系统就会学坏：先学会吹牛和绕门禁。

---

## 2. 我认为对的决策

### 2.1 以 AC 通过率定义“完成”

这是对的。  
“95% 代码由 AI 写”没意义，只有“95% 验收项能被工具证明”才有管理价值。

### 2.2 实现与验收分离

这也是对的。  
同一个上下文里“写完并自己宣布完成”，企业级不可接受。尤其夜间无人值守时，没有独立验收就等于没有质量系统。

### 2.3 Kimi 少用、只诊断、不写代码

这也是对的，而且必须更硬。  
现有项目已经有 Playwright + chrome-devtools 的确定性视觉工具链，视觉问题应当先走测量和对比，再让昂贵模型做语义诊断。[已验证:AGENTS.md:还原工具链]

Kimi 的正确位置不是“视觉主力”，而是：

> 机器已经测出问题，但 DeepSeek 轮修后仍说不清结构语义时的外脑。

### 2.4 夜间只出 Draft PR

我坚持这条。  
普通工程团队会嫌慢；但在 AI 主体交付场景里，Draft PR 是最低成本保险。  
只要一次夜间误合并污染主干，治理体系信誉会立刻崩掉。

### 2.5 技术栈必须以现有 AGENTS 为准，而不是角色设定里的 antd v6

这是我认为**最该先拍死**的点。

角色描述里有 ant-design v6，但项目宪法明写的是 **Sass + CSS Modules + 自定义 shared 组件**，样式从 tokens 出，原则与 antd 主导体系完全不同。[已验证:AGENTS.md:技术栈/目录结构/UI还原铁律]

如果你两边都不明确，低阶模型会随机混搭：

- 一边写 `.module.scss`
- 一边又假设 Form/Table 写法和主题配置
- 一边遵守 Token
- 一边又从“通用 antd 经验”硬编码

这会把视觉一致性和架构一致性一起打碎。

**我的立场：**  
先按自定义组件 + CSS Modules 体系治理；除非产品明确要求迁移 antd v6，否则不要在方案里并行两套 UI 世界观。

---

## 3. 我认为危险、需要收紧的地方

### 3.1 L2/B2 直接进夜间，我倾向更保守

一开始可以白天跑：

- L0/L1
- V0/V1
- B0/B1

然后再放开到：

- L2
- V2
- 部分 B2

原因很简单：  
B2 以上开始出现写操作、状态流转、接口副作用。AI 写“看起来对”的 UI 不难，真正难的是它以为成功了、其实搞坏了数据流和用户路径。

**我的修订建议：**

- 夜间 Phase-1：B0/B1 全开，B2 只允许“有完整 E2E 契约 + Mock/联调稳定”的白名单任务
- B3 永不进入无人值守自治，只允许生成设计和风险报告，等人工授权

### 3.2 Flash 两次失败再升 Pro，粒度还不够

“连续失败 2 次升级”太粗糙。  
应该改成失败分类路由：

| 失败类型 | 立即动作 |
|---|---|
| Compile / 简单 Type / Lint | Flash |
| 跨文件类型传导 / 状态流错误 | 立即 Pro |
| 交互失败且根因不清 | 立即 Pro |
| PRD 与接口冲突 | 停止，不升级模型 |
| 环境抖动 | 恢复 Agent，不改业务代码 |
| 视觉语义偏差 | 先 Pro；仍不收敛再 Kimi |

不要用“再试一次更贵模型”掩盖分类错误。

### 3.3 输入治理比模型路由更决定成败

你们高估了模型，容易低估输入质量。  
基于现有工作模式，Goal/Ghost 本身已经暗示：目标不清时最多调研两轮，否则标信息不足。[已验证:AGENTS.md:工作模式选择]

我建议把这一点升级成硬制度：

- 没有可执行 AC，不开编码
- 没有 API 契约，不写业务逻辑
- 没有原型，V2/V3 视觉门不启动
- D2 缺口直接阻塞，禁止“先写再说”

一句话：

> **输入不完整时，最好的 AI 行为是停，而不是聪明地编。**

### 3.4 C7 红线要成为自动门，不要只是“文档要求”

现有 C7 很对：`.tsx ≤300`、`.module.scss ≤300`、禁裸色值和魔法 px。[已验证:AGENTS.md:C7红线]

但如果这只是 Agent 记在心里，没有机器执行，夜间一定会慢慢腐化。  
我建议：

- 对改动文件做行数和 Token 扫描
- radio/颜色硬编码扫词
- page 内功能块粗分检查
- 超线直接 fail，不允许“先提交再拆”

治理系统最怕的不是一次错误，而是错误能悄悄过线。

---

## 4. 我对“全员全栈 + AI 顶前端”这件事的判断

这个组织转变能成立，前提是角色重新分配，而不是幻想前端工种消失后零成本。

你们消灭的应该是：

- 大批量重复页面手写
- 机械性样式对齐
- 低价值 CRUD 编码

不该消灭的是：

- Design System 所有权
- 交互规范所有权
- 公共组件契约所有权
- 前端质量门槛所有权

否则会出现一个诡异局面：

- AI 产出越来越快
- 页面风格越来越碎
- shared 越来越脏
- 全栈同事谁都不会系统性修前端债

所以我主张保留一个轻编制：

- **前端治理 Owner 1 人**
- **CarrorOS Owner 1 人**
- **全栈任务 Owner 若干**

AI 可以顶“页面生产”，顶不了“体系设计”。

---

## 5. 如果只按我的意见，接下来怎么定

如果我拍板，最终方案就 8 条硬规则：

1. **以现有 AGENTS 技术栈为准**：自定义组件 + CSS Modules + Tokens，不默认 antd。
2. **以 AC 证据定义完成，不以代码量定义完成。**
3. **Flash 跑低风险，Pro 主实现，Kimi 只做稀缺视觉诊断。**
4. **功能和交互门先于视觉门；工具判断先于模型判断。**
5. **夜间只出隔离分支或 Draft PR，自动合并永不默认开启。**
6. **D2 输入缺口一律阻塞，不让模型补产品决策。**
7. **C7、Token、截图路径、任务目录全部自动化校验。**
8. **先做治理闭环 benchmark，再谈 95%/90%/80% 运营指标。**

顺序上，我不谈“如何立刻更自治”，我谈“如何先不失控”：

```text
统一规则冲突
→ 输入契约与证据门
→ 白天监督试点
→ 受限夜间
→ 再扩并行和业务深度
```

---

## 6. 我最想让 Opus / Grok 同行继续打的，其实只有三刀

别再争框架完整性了。真正决定能不能上线的是这三个问题：

1. **没有专职前端时，谁来长期拥有 Design System 和 shared 组件合同？**
2. **当 PRD/API/设计三者冲突时，系统是会停，还是会“聪明地编出一个看起来能跑的前端”？**
3. **夜间发现公共组件本身有历史病时，是局部绕开，还是允许 AI 触碰公共面？**

三者都能给出确定答案，方案就能落地。  
答不出来，再加再多模型路由表也没用。

---

## 最后一句

我的总看法是：

> **这套方案方向对，时机也对，但成熟度的关键不在模型混编，而在你们愿不愿意把“自主开发”收敛成“受限、可证、可恢复的生产系统”。**

如果只选一个原则压死整件事，我选这句：

> **让 AI 在边界内高产，而不是让 AI 在自由中看起来很能干。**

真正伟大的工作，不是让 deepseek 夜里把页面写满仓库，而是早上打开 Draft PR 时，你敢点进验收报告，并且相信：

- 没越界
- 没编造
- 失败可解释
- 通过可复现

那才叫 AI 自主完成前端交付，而不是 AI 夜里加班写代码。
