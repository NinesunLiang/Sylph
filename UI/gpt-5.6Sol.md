# CarrorOS AI 前端自主开发方案

> **文档状态**：评审稿 v0.9  
> **评审对象**：Opus 4.8、Grok 4.5 及项目治理负责人  
> **方案边界**：本稿只定义治理体系、Agent 架构、任务协议、无人值守 Loop、质量门禁、模型路由和落地计划，不执行具体项目开发。  
> **核心目标**：在全栈开发模式下，让 AI 自主完成前端静态页面开发 ≥95%、UI 交互 ≥90%、在具备合格 PRD 和 API 文档时完成业务逻辑 ≥80%。  
> **成本约束**：Kimi K3 仅用于少量高价值视觉还原与视觉差异裁决，不进入常规编码循环。

---

# 1. 执行摘要

本方案将 AI 前端开发定义为一条受 CarrorOS 治理的、证据驱动的自动化生产线，而不是一个模型持续自由编码：

```text
需求输入
  ↓
输入质量检查与契约化
  ↓
任务拆分、风险分级、模型路由
  ↓
DeepSeek V4 Pro 主实现
  ↓
DeepSeek V4 Flash 机械执行、验证、低风险修复
  ↓
确定性工具门禁
  ↓
按需调用 Kimi K3 做视觉诊断
  ↓
DeepSeek 根据诊断修改
  ↓
独立验收 Agent + CarrorOS 证据归档
  ↓
通过 / 阻塞 / 人工升级
```

系统的核心原则是：

1. **CarrorOS 掌握状态，模型不掌握状态。** 所有进度、决策、失败、证据必须落盘，模型上下文可以随时丢失并恢复。
2. **工具判定事实，模型处理歧义。** 编译、类型、Lint、测试、页面可访问性、网络错误、视觉像素差异优先由工具判断。
3. **实现与验收分离。** 同一 Agent 不得只凭自己的描述宣布任务完成。
4. **便宜模型跑量，能力模型攻坚，昂贵视觉模型只做高价值判断。**
5. **无人值守不等于无限自治。** Loop 必须有预算、重试、熔断、回滚、升级和停止条件。
6. **没有证据就没有完成。** 继承历史 AGENTS.md 的铁律：编码前有 `research.md + plan.md`，完成一步立即更新 `executor.md`，失败立即留痕。[已验证:AGENTS.md:任务工作流章节]
7. **不允许模型靠反复尝试掩盖输入不足。** PRD、API、设计稿不完整时，应输出阻塞项或采用预先授权的默认策略。

[推断,待确认] 本方案建议先把目标定义为“95%/90%/80% 的验收项自动通过率”，而不是“95% 的代码由 AI 编写”。前者可测量、可审计，也更符合业务交付目标。

# 2. 目标、非目标与成功定义

## 2.1 建设目标

### G1：静态页面自主交付 ≥95%

“静态页面完成”必须同时满足：

- 路由可访问；
- 页面主要区域完整；
- 使用正确组件和 Design Token；
- 目标视口布局符合要求；
- Loading、Empty、Error、No Permission 等约定状态存在；
- 无阻塞级控制台错误；
- 视觉验收达到页面等级对应阈值；
- 交付证据完整。

### G2：UI 交互自主交付 ≥90%

包括：

- 表单输入、校验、提交和重置；
- 搜索、筛选、排序、分页；
- Tab、折叠、展开、Popover、Modal、Drawer；
- 表格行操作、批量操作；
- 成功、失败、取消和二次确认；
- 键盘、焦点和禁用态等基础可用性；
- 状态刷新和返回路径。

### G3：业务逻辑自主交付 ≥80%

成立前提：

- PRD 中业务规则明确；
- API 契约结构完整；
- 枚举和状态机有定义；
- 权限边界明确；
- 异常响应及处理策略明确，或项目已有统一策略。

不具备这些条件时，不把失败归因于模型开发能力，而归类为“输入契约缺陷”。

### G4：夜间无人值守连续运行

夜间 Loop 应当能够：

- 从磁盘恢复任务；
- 自动执行一个或多个隔离任务；
- 自动验证和有限修复；
- 遇到不可逆、高风险或信息不足时安全停止；
- 早晨输出任务摘要、通过项、失败项、证据和人工待办。

## 2.2 非目标

第一阶段不追求：

- AI 自主决定产品需求；
- AI 自主修改后端 API 契约；
- AI 自主执行数据库破坏性操作；
- 所有页面都达到营销官网级像素还原；
- 完全取消人工验收；
- 用昂贵模型覆盖整个开发流程；
- 在输入明显缺失时强行生成“看起来能用”的业务逻辑。

## 2.3 指标口径

建议使用以下统一公式：

```text
静态页面自主完成率
= 无人工修改且通过页面验收的页面数 / 纳入评估页面总数

UI 交互自主完成率
= 无人工修改且通过交互 AC 的交互项数 / 全部交互 AC 数

业务逻辑自主完成率
= 无人工修改且通过业务 AC 的业务规则数 / 输入合格的业务 AC 数

首次通过率
= 首轮实现后通过全部门禁的任务数 / 总任务数

自治闭环率
= 经自动修复后通过且没有人工介入的任务数 / 总任务数

人工接管率
= 需要人工修改代码或产品裁决的任务数 / 总任务数
```

[推断,待确认] “人工查看报告但未修改代码”不应计为人工介入；“人工补充缺失需求”应计入输入治理，不计入 AI 编码失败。

# 3. 总体架构

## 3.1 五层架构

```text
┌──────────────────────────────────────────────┐
│ L1 输入与契约层                              │
│ PRD / API / 设计稿 / 路由 / 权限 / 项目规范 │
├──────────────────────────────────────────────┤
│ L2 CarrorOS 治理控制层                       │
│ 任务状态 / CAS / 预算 / 审批 / 恢复 / 审计  │
├──────────────────────────────────────────────┤
│ L3 Agent 编排层                              │
│ Planner / Implementer / Visual / Verifier    │
├──────────────────────────────────────────────┤
│ L4 工具执行层                                │
│ Git / IDE / TS / ESLint / Test / Browser     │
├──────────────────────────────────────────────┤
│ L5 质量与证据层                              │
│ AC / 截图 / Trace / 日志 / Diff / 报告       │
└──────────────────────────────────────────────┘
```

## 3.2 控制面与执行面分离

### 控制面：CarrorOS

负责：

- 任务身份和状态；
- 阶段迁移；
- Agent 权限；
- 模型路由；
- Token、时间和调用预算；
- 重试与熔断；
- 证据索引；
- 高风险审批；
- 任务恢复；
- 最终裁决。

### 执行面：Agent + 工具

负责：

- 读取允许范围内的文件；
- 修改计划范围内的代码；
- 运行验证命令；
- 启动页面和执行浏览器验证；
- 生成截图、Trace、日志和 Diff；
- 将结果写回 CarrorOS 任务目录。

模型不得绕过控制面自行宣布成功，也不得在没有任务租约的情况下修改仓库。

# 4. Agent 角色设计

不建议建立一个“万能前端 Agent”。建议以角色隔离降低上下文污染和自证风险。

## 4.1 Intake Agent：输入审查员

职责：

- 检查 PRD、API、设计稿和项目规范是否齐全；
- 提取页面、角色、状态、字段、动作和异常路径；
- 生成缺失项清单；
- 判定 Goal 或 Ghost 模式。

历史规则已定义：目标清晰时使用 Goal；方向模糊时使用 Ghost，并且调研澄清最多两轮，仍不明确则标记信息不足。[已验证:AGENTS.md:工作模式选择章节]

输出：

- `input-assessment.yaml`
- `requirement-contract.md`
- `open-questions.md`
- `assumptions.yaml`

禁止：

- 修改业务代码；
- 擅自补充高风险业务规则。

## 4.2 Research Agent：仓库研究员

职责：

- 定位路由、组件库、请求层、状态管理和权限体系；
- 查找同类页面；
- 确认 Ant Design v6 使用方式和项目封装；
- 提取 Token、布局、表格、表单等既有模式；
- 形成复用建议和风险清单。

输出：`research.md`。

## 4.3 Planner Agent：任务规划员

职责：

- 将需求转化为可独立验收的 Work Unit；
- 为每个 Work Unit 定义 AC、文件边界、依赖、风险、回滚方案；
- 选择 direct、stepwise 或 race 执行引擎；
- 给任务标注模型等级和视觉等级。

历史规则中，L2+ 任务需要任务目录，包含 `research.md`、`plan.md`、`executor.md`、`state/progress.md` 和 `state/acceptance_report.md`。[已验证:AGENTS.md:任务目录章节]

## 4.4 Implementer Agent：实现者

主要由 DeepSeek V4 Pro 承担，职责包括：

- 页面骨架；
- 组件拆分；
- 状态及交互；
- API 接入；
- 表单和表格；
- 权限和路由；
- 错误态及边界处理；
- 必要测试。

限制：

- 只能修改 `plan.md` 中声明的文件或匹配的路径范围；
- 新增越界文件前必须申请计划变更；
- 不能修改 API 契约掩盖前端错误；
- 不能删除失败测试来让门禁通过。

## 4.5 Fixer Agent：低成本修复者

默认由 DeepSeek V4 Flash 承担：

- TypeScript 报错；
- ESLint/格式问题；
- 明确的导入和类型问题；
- 根据结构化诊断执行样式微调；
- 补充低复杂度测试；
- 修复已定位的交互问题。

Flash 不负责：

- 未明确的跨域架构决策；
- 大规模组件重构；
- 复杂业务状态机；
- 自主改变产品行为。

## 4.6 Visual Analyst：视觉分析员

Kimi K3 只承担该角色：

- 设计稿区域理解；
- 关键页面视觉结构提取；
- 设计截图与实现截图差异诊断；
- 输出按优先级排序的修正建议；
- 对高价值页面做最终视觉裁决。

原则上只分析、不直接编码，以避免昂贵模型进入反复修改循环。

## 4.7 Verifier Agent：独立验收员

职责：

- 读取 AC，不读取实现者的主观成功结论；
- 运行全部适用门禁；
- 检查证据是否对应当前 Commit；
- 输出 pass、fail 或 blocked；
- 生成验收报告。

建议使用与实现者不同的上下文；高风险任务可使用不同模型。

## 4.8 Recovery Agent：恢复与故障分类员

职责：

- 判断失败是环境、输入、代码、测试、视觉、工具还是模型问题；
- 恢复中断的任务；
- 检查是否允许重试；
- 防止同一失败无变化循环；
- 触发升级或熔断。

# 5. 模型路由与成本策略

## 5.1 默认路由

| 任务 | 默认模型 | 升级条件 |
|---|---|---|
| 文件检索、简单修改、格式与类型修复 | DeepSeek V4 Flash | 两次失败或涉及跨文件设计 |
| 页面实现、组件抽象、API 和状态 | DeepSeek V4 Pro | 架构冲突或连续失败 |
| 普通视觉调整 | DeepSeek V4 Pro/Flash | V2/V3 页面且自动视觉不收敛 |
| 视觉结构理解、截图差异诊断 | Kimi K3 | 不再自动升级，超预算转人工/高阶裁决 |
| 复杂架构、规则冲突、失败裁决 | GPT-5.6-sol / Opus 4.8 / Grok 4.5 | 人工决策 |

## 5.2 Kimi K3 调用政策

### 视觉等级

| 等级 | 页面类型 | Kimi 配额 |
|---|---|---:|
| V0 | 无设计稿、低视觉要求 | 0 |
| V1 | 标准 CRUD、项目已有模板 | 0 |
| V2 | 有明确设计稿的普通业务页 | 默认 0，失败后最多 1 次 |
| V3 | 核心工作台、复杂高保真页面 | 最多 2 次 |

### Kimi 触发条件

必须同时满足：

1. 任务标记为 V2 或 V3；
2. 当前构建、类型和交互门禁已经通过；
3. 视觉问题不是页面无法加载、接口报错或明显 CSS 编译错误；
4. DeepSeek 已完成至少一轮视觉还原；
5. 调用预算仍有余额。

### Kimi 输入最小化

每次仅提供：

- 目标设计截图；
- 同视口实现截图；
- 页面区域和视口信息；
- Design Token 摘要；
- 已知限制；
- 固定输出 Schema。

不提供整个仓库，不提供无关历史对话，不让 Kimi自行搜索大量源码。

### Kimi 输出契约

```json
{
  "verdict": "pass | fail | uncertain",
  "similarity_estimate": 0.0,
  "issues": [
    {
      "priority": "P0 | P1 | P2",
      "region": "header/main/table/form/modal",
      "category": "layout | spacing | typography | color | component | content",
      "observation": "可观察差异",
      "suggestion": "不绑定实现细节的修改建议",
      "confidence": 0.0
    }
  ],
  "stop_recommended": false
}
```

输出交给 DeepSeek Pro 修改；修改后的验证默认由工具和 Verifier 完成，不自动再次调用 Kimi。

## 5.3 初始调用量预算

[推断,待确认] 建议以任务调用次数而非 Token 金额先行限制：

- DeepSeek V4 Flash：65%–75%；
- DeepSeek V4 Pro：20%–30%；
- Kimi K3：2%–5%；
- 高阶规划和裁决模型：1%–3%。

同时设置：

```yaml
budget:
  task_max_model_calls: 20
  task_max_auto_fix_rounds: 4
  same_failure_max_retries: 1
  visual_fix_rounds: 3
  kimi_default_calls: 0
  kimi_max_calls_v2: 1
  kimi_max_calls_v3: 2
  wall_clock_timeout_minutes: 120
```

具体数值应通过试点校准，而不是直接永久固化。

# 6. 输入契约

AI 自治能力高度依赖输入标准化。开发开始前，将原始材料转换为机器可消费契约。

## 6.1 PRD 最低要求

每个页面至少包含：

- 页面目标；
- 用户角色；
- 进入条件；
- 页面区域；
- 字段定义；
- 用户动作；
- 业务规则；
- 状态转换；
- 成功反馈；
- 失败反馈；
- 权限限制；
- 验收条件。

## 6.2 API 最低要求

- Method 和 Path；
- 请求字段、类型、必填和枚举；
- 响应字段和空值规则；
- 分页结构；
- 错误码；
- 权限失败；
- 幂等要求；
- 示例请求与响应；
- Mock 可用方式。

## 6.3 设计输入最低要求

- 页面截图或可访问设计稿；
- 基准视口；
- 关键状态截图；
- 弹窗、抽屉和下拉状态；
- 图标或资源来源；
- 可复用组件说明；
- 哪些区域要求像素级，哪些允许系统化适配。

## 6.4 缺失信息处理

按风险分三级：

### D0：可安全默认

例如项目已有统一的 Loading 和空状态。Agent 可引用项目惯例，并记录假设。

### D1：可回滚假设

例如普通筛选项是否即时查询。可按项目惯例实现，但必须记录在 `assumptions.yaml` 和验收报告中。

### D2：必须阻塞

例如删除是否不可恢复、金额计算规则、权限边界、状态能否逆转。禁止自主假设。

# 7. 任务分级与拆分

## 7.1 继承并扩展 L0–L3

历史 AGENTS.md 已定义：L0 微操、L1 单文件小改、L2 2–5 文件中等改动、L3+ 架构或新功能。[已验证:AGENTS.md:工作流速查章节]

建议扩展前端风险维度：

```yaml
classification:
  complexity: L0 | L1 | L2 | L3
  visual: V0 | V1 | V2 | V3
  business_risk: B0 | B1 | B2 | B3
  reversibility: R0 | R1 | R2
```

- `B0`：纯展示；
- `B1`：普通查询或本地交互；
- `B2`：创建、编辑、批量操作；
- `B3`：资金、删除、权限、安全、不可逆操作。

无人值守默认允许：

- L0–L2；
- V0–V2；
- B0–B2；
- 可回滚修改。

L3、B3 或不可逆动作需要预审批，夜间不可自行扩大范围。

## 7.2 Work Unit 粒度

一个 Work Unit 应满足：

- 目标单一；
- 修改范围可声明；
- 能独立验证；
- 理想执行时间 15–60 分钟；
- 失败可局部回滚；
- 最多包含一个核心不确定性。

不建议把“完成整个业务模块”作为一个任务。建议拆为：

1. 路由和页面骨架；
2. 查询区；
3. 列表和分页；
4. 创建/编辑表单；
5. 详情抽屉；
6. API 对接；
7. 权限及异常态；
8. 视觉对齐；
9. 回归验收。

## 7.3 执行引擎

继承现有规则：

- `direct`：单文件、低风险、范围明确；
- `stepwise`：步骤有依赖；
- `race`：至少三个互不依赖的子任务并行。[已验证:AGENTS.md:执行引擎章节]

前端场景中，race 只能在文件所有权不冲突时使用。例如三个独立页面可以并行；共享 Token、路由表和公共类型不可多人同时写。

# 8. 标准自主开发流水线

## Phase 0：任务接收

1. 创建任务 ID；
2. 记录目标 Commit；
3. 获取仓库租约；
4. 保存输入材料引用；
5. 判定 Goal/Ghost；
6. 创建任务目录。

## Phase 1：输入审查

1. 提取页面和交互清单；
2. 检查 PRD/API/设计输入；
3. 标记 D0/D1/D2 缺失项；
4. D2 存在则阻塞；
5. 生成需求契约。

## Phase 2：仓库研究

至少调查：

- 路由组织；
- 页面目录；
- 公共组件；
- Ant Design v6 封装；
- 样式和 Token；
- API 层；
- 错误处理；
- 状态管理；
- 测试设施；
- 相似页面。

输出 `research.md`，禁止边研究边无计划修改。

## Phase 3：规划冻结

`plan.md` 每一步必须包含：

```yaml
- id: FE-03
  goal: 实现订单筛选区
  files_allowed:
    - src/pages/order/**
  dependencies: [FE-01]
  acceptance:
    - 支持订单号和状态筛选
    - 重置后恢复默认参数并重新查询
    - 参数与 API 契约一致
  verification:
    - pnpm typecheck
    - pnpm test OrderFilter
    - playwright order-filter.spec.ts
  rollback: 回退该步骤 Commit
```

计划冻结后，执行 Agent 不得隐式扩展范围。

## Phase 4：实现

执行顺序建议：

1. 数据类型与 API 适配；
2. 页面状态模型；
3. 页面结构；
4. UI 组件；
5. 用户交互；
6. 异常和边界状态；
7. 测试；
8. 样式和视觉还原。

每完成一步：

- 更新 `executor.md`；
- 保存修改摘要；
- 保存命令和输出引用；
- 提交原子 Commit 或创建可回滚检查点；
- 更新任务状态。

## Phase 5：确定性验证

按从便宜到昂贵执行：

1. 变更范围检查；
2. 格式检查；
3. TypeScript；
4. ESLint；
5. 单元测试；
6. 构建；
7. 页面启动与健康检查；
8. 浏览器交互；
9. 可访问性冒烟；
10. 视觉回归；
11. 独立验收。

任何前置门禁失败，都不进入昂贵视觉分析。

## Phase 6：有限自动修复

失败后先分类，再修复：

```text
CompileError     → Flash
TypeError        → Flash，复杂泛型转 Pro
LintError        → Flash
UnitTestFailure  → Flash/Pro
InteractionError → Pro
ApiContractError → Pro；若文档冲突则阻塞
VisualError      → Pro/Flash；达到门槛才调用 Kimi
EnvironmentError → Recovery，不改业务代码
RequirementGap   → 阻塞
```

每次修复必须产生变化摘要；失败指纹相同且代码、环境、输入均无有效变化时，不允许再次重试。

## Phase 7：独立验收

Verifier 对每条 AC 给出：

- `PASS`：有对应证据；
- `FAIL`：实测不符合；
- `BLOCKED`：环境或输入不足；
- `NOT_APPLICABLE`：不适用并说明理由。

不允许用“看起来没问题”“理论上可用”作为证据。历史规则明确要求做完要验，且不越界修改、不编造、不确定时询问。[已验证:AGENTS.md:铁律与三源验证章节]

## Phase 8：交付或升级

通过后生成：

- 验收报告；
- 变更文件清单；
- 测试摘要；
- 截图和 Trace 索引；
- 剩余假设；
- 风险和回滚方式；
- 成本统计。

失败则生成最小人工待办，不把完整日志直接丢给人员自行阅读。

# 9. 前端质量门禁

## C0：输入门禁

- PRD、API 和设计输入达到任务要求；
- D2 问题已解决；
- AC 可执行。

## C1：范围门禁

- Diff 只涉及允许文件；
- 无无关格式化；
- 无锁文件异常变更；
- 无生成物或敏感文件误提交。

## C2：代码门禁

- TypeScript 通过；
- ESLint 通过；
- 构建通过；
- 无新增高危依赖；
- 无 `any`、禁用规则、忽略错误等逃逸行为，除非项目规则允许并有说明。

## C3：架构门禁

检查：

- 是否复用已有组件；
- 是否遵循目录和依赖方向；
- 是否将页面专属逻辑错误提升为全局抽象；
- 是否破坏 API、权限、路由或状态约定；
- 是否出现超大组件、循环依赖和重复实现。

## C4：功能门禁

覆盖：

- 正常路径；
- 空数据；
- 加载；
- API 失败；
- 无权限；
- 重复提交；
- 取消；
- 返回和刷新；
- 关键边界值。

## C5：交互门禁

- 点击目标有效；
- 禁用态正确；
- Modal/Drawer 关闭行为正确；
- 表单焦点和错误提示合理；
- Loading 期间防重复操作；
- 成功后列表或详情状态正确刷新；
- 筛选、分页和 URL/状态保持符合项目约定。

## C6：视觉门禁

建议分两层：

### 确定性检查

- 固定视口截图；
- 像素差异；
- 元素 Bounding Box；
- 溢出和遮挡；
- 字体加载；
- 控制台错误；
- 网络资源失败。

### 语义视觉检查

- 信息层级；
- 对齐关系；
- 组件密度；
- 设计系统一致性；
- 差异是否影响使用感知。

Kimi 只处理第二层中的高价值复杂判断。

## C7：证据门禁

- 所有 AC 均有证据；
- 证据对应当前 Commit SHA；
- 命令输出未被截断到无法判断；
- 截图包含视口、时间和页面标识；
- 失败也有留痕；
- `executor.md` 与实际状态一致。

## C8：回归门禁

- 受影响页面测试通过；
- 公共组件变更覆盖其消费者；
- 路由、权限、请求层和 Token 变更执行扩展回归；
- 没有通过更新基准图掩盖视觉回归。

# 10. 视觉还原专项方案

## 10.1 视觉还原分层

### 第一层：Design System 对齐

先保证：

- 颜色来自 Token；
- 字号和行高统一；
- 间距遵循尺度；
- 圆角、阴影和边框统一；
- Ant Design v6 主题配置优先于页面覆盖；
- 常用模式复用项目组件。

### 第二层：结构对齐

- 页面栅格；
- Header、Sider、Content 关系；
- 卡片和区域宽高；
- 表格、筛选区、操作区相对位置；
- Modal/Drawer 尺寸；
- 响应式行为。

### 第三层：像素微调

仅对 V2/V3 页面：

- 局部间距；
- 文本基线；
- 图标尺寸；
- 边框和阴影；
- 细节状态。

## 10.2 建议视口

[推断,待确认] 若历史项目是桌面管理系统，可先采用：

- 1440×900：主验收视口；
- 1280×800：紧凑桌面；
- 1024×768：最小支持桌面。

具体断点必须由项目规范或设计输入确认，不能由 Agent 永久自行决定。

## 10.3 视觉通过规则

不建议只用单一像素相似度判定。建议组合：

```text
visual_score
= 结构权重 40%
+ 组件与间距 25%
+ 字体与颜色 15%
+ 像素差异 10%
+ 语义视觉审查 10%
```

P0 结构差异无论总分多高都不能通过；动态区域、时间、随机数据必须 Mask。

## 10.4 防止“截图作弊”

禁止：

- 隐藏未实现区域；
- 用静态图片替代真实组件；
- 通过固定绝对定位仅适配一个视口；
- 更新基线截图来消除失败；
- 在测试环境关闭真实交互；
- 通过删除文本或数据减少差异。

# 11. 夜间无人值守 Loop

## 11.1 状态机

```text
CREATED
  → INPUT_CHECK
  → RESEARCHING
  → PLANNING
  → READY
  → IMPLEMENTING
  → VERIFYING
  → FIXING
  → ACCEPTING
  → PASSED

任意阶段可进入：
  BLOCKED_INPUT
  BLOCKED_ENV
  BUDGET_EXHAUSTED
  NEEDS_REVIEW
  FAILED
  ROLLED_BACK
```

## 11.2 每轮 Loop 协议

1. 获取任务租约；
2. 校验仓库 SHA 和任务版本；
3. 读取唯一状态源；
4. 选择下一合法动作；
5. 检查预算和权限；
6. 调用一个 Agent 或工具步骤；
7. 保存完整输出到 artifact；
8. 提取结构化结果；
9. 使用 CAS 更新状态；
10. 释放或续租；
11. 判断继续、完成、阻塞或熔断。

## 11.3 任务租约与并发

```yaml
lease:
  owner: agent-run-uuid
  acquired_at: timestamp
  expires_at: timestamp
  repository_sha: commit
  file_scopes:
    - src/pages/order/**
```

并发要求：

- 同一路径只能有一个写租约；
- 共享文件修改必须序列化；
- race 子任务使用独立分支或 worktree；
- 合并前重新执行受影响门禁；
- CAS 版本冲突时重新读取，不得覆盖状态。

## 11.4 重试规则

允许重试必须满足至少一个条件：

- 代码已变化；
- 输入已补充；
- 环境已恢复；
- 使用不同的修复策略；
- 升级到更适合的模型；
- 失败被重新分类。

禁止：

- 原 Prompt 原上下文原代码重复调用；
- 无限提高模型等级；
- 通过放宽测试或删除断言解决失败。

## 11.5 熔断条件

满足任一条件立即停止：

- 相同失败指纹重复两次且无有效变化；
- 超过总调用预算；
- 超过墙钟时间；
- 发现需求与 API 契约冲突；
- 需要执行破坏性操作；
- Diff 超出计划范围；
- 测试环境异常且无法归因；
- 模型输出出现高风险逃逸行为；
- Kimi 配额耗尽仍有 P0 视觉问题。

## 11.6 夜间安全边界

无人值守期间禁止默认执行：

- 合并主分支；
- 发布生产；
- 删除数据库数据；
- 修改权限模型；
- 修改 CI 密钥；
- 大版本升级依赖；
- 大范围公共组件重构；
- 自动接受视觉基线更新。

夜间成果应进入隔离分支或 Draft PR，等待早晨审查。

# 13. 证据体系

历史治理强调三源验证：方案自审、证据自检、实际运行验证。[已验证:AGENTS.md:三源验证章节]

前端任务将证据具体化为：

| 结论 | 最低证据 |
|---|---|
| 编译通过 | 命令、退出码、Commit SHA |
| 页面可访问 | URL、HTTP/浏览器结果、截图 |
| 交互通过 | Playwright 断言、Trace 或视频 |
| API 接入正确 | 请求参数、响应摘要、Mock/测试记录 |
| 视觉通过 | 同视口目标图、实现图、Diff、阈值 |
| 无控制台错误 | 浏览器日志 |
| AC 完成 | AC 与证据路径一一映射 |

置信度规则沿用：

- `[已验证:文件:行号]`：读取到的静态事实；
- `[已测试:命令+输出]`：实际运行事实；
- `[推断,待确认]`：尚未验证的结论。[已验证:AGENTS.md:置信度标记章节]

建议在机器报告中增加：

```yaml
claim:
  text: 筛选重置后自动重新查询
  confidence: tested
  evidence:
    - playwright/order-filter.spec.ts#reset
    - artifacts/traces/order-filter-reset.zip
  commit: abc123
```

# 15. 评测体系

上线治理前必须建立固定 Benchmark，避免用单个真实项目主观判断。

## 15.1 基准任务集

建议至少 30 个任务：

- 5 个纯静态页面；
- 5 个标准 CRUD；
- 5 个复杂表单；
- 5 个表格和批量操作；
- 3 个权限页面；
- 3 个复杂状态页面；
- 2 个高保真核心页；
- 2 个故障和恢复任务。

每类任务都应包含：

- 完整输入版本；
- 有意缺失输入版本；
- 标准答案或人工验收基线；
- 允许修改范围；
- AC；
- 成本上限。

## 15.2 核心指标

### 质量指标

- AC 通过率；
- 首次通过率；
- 自动修复后通过率；
- 视觉 P0/P1 问题数；
- 回归缺陷数；
- 越界修改率；
- 错误完成声明率。

### 效率指标

- 单页墙钟时间；
- 单 AC 模型成本；
- 平均修复轮次；
- 夜间吞吐；
- 人工审查时间；
- Kimi 调用收益。

### 治理指标

- 证据完整率；
- 可恢复率；
- 熔断准确率；
- 输入缺陷识别率；
- 同失败无效重试率；
- 人工升级准确率。

## 15.3 Kimi ROI 指标

```text
Kimi 有效调用率
= 调用后一次 DeepSeek 修改即消除 P0/P1 的次数 / Kimi 调用次数

Kimi 边际收益
= 使用 Kimi 后视觉分提升 - 不使用 Kimi 对照组视觉分提升

Kimi 单次有效成本
= Kimi 总成本 / 被其有效解决的视觉问题数
```

如果 Kimi 只给出泛化建议或不能显著降低修复轮次，应进一步缩减调用范围。

# 16. 分阶段落地路线

交付：

- 唯一版 AGENTS.md；
- CarrorOS 前端任务 Schema；
- 任务状态机；

- 输入契约模板；

- 任一新 Agent 能依据文档生成一致任务目录；
- 任务能从磁盘恢复。

范围：

- V0/V1 页面；
- B0/B1 任务；

- 不调用 Kimi；
- 人工观察每个状态迁移。

目标：验证控制面、证据和门禁，而不是追求吞吐量。

## 阶段 2：受限夜间运行，2–4 周

范围：

- 允许 L1/L2；
- 允许自动修复；
- 使用隔离分支；
- 单夜最多 3–5 个任务；
- Kimi 只用于一个 V2/V3 试点任务。

验收建议：

- 状态恢复成功率 ≥98%；
- 越界修改率为 0；
- 错误完成声明率 <2%；
- 无无限循环；
- 早晨报告可在 15 分钟内完成审查。

## 阶段 3：规模化并行，4–8 周

增加：

- Worktree 隔离；
- race 并行；
- 文件租约；
- 自动 Draft PR；
- 模型成本仪表盘；
- 故障指纹和 Error DNA；
- 基准集持续回归。

## 阶段 4：业务逻辑扩展

范围：

- B2 任务；
- 复杂表单；
- 权限与状态机；
- API 契约自动校验；
- 更丰富的端到端测试。

B3 仍保留人工审批和重点审查。

# 18. 主要风险与缓解措施

## 风险 1：PRD 不足导致 AI 编造业务

缓解：输入门禁、D0/D1/D2 分类、强制假设登记、D2 自动阻塞。

## 风险 2：实现者自证完成

缓解：Verifier 独立运行；AC 与工具证据一一映射；Commit SHA 绑定证据。

## 风险 3：夜间无限循环消耗成本

缓解：失败指纹、变化检测、预算、墙钟超时、有限重试和熔断。

## 风险 4：低价模型产生低质量大改

缓解：Flash 只处理低风险机械任务；范围租约；Diff 门禁；复杂任务升级 Pro。

## 风险 5：Kimi 成本失控

缓解：默认零调用；V2/V3 白名单；单页最多 1–2 次；只传两张关键图片和摘要；只诊断不编码。

## 风险 6：视觉通过但业务不可用

缓解：功能和交互门禁先于视觉门禁；浏览器操作和 API 契约验证不可跳过。

## 风险 7：为了过门禁修改测试或基线

缓解：测试和基线文件单独权限；更新需审批；Verifier 检查弱化断言行为。

## 风险 8：并发 Agent 冲突

缓解：Worktree、文件租约、CAS、共享文件串行和合并后回归。

## 风险 9：规则过多导致模型无法遵循

缓解：内核规则保持短小；细节通过 Skill 按需加载；每个阶段只注入当前所需规则；Schema 机器校验。

## 风险 10：评测数据失真

缓解：冻结 Benchmark；区分输入缺陷和编码失败；保留人工修改量；持续对照不同模型策略。

# 19. 需要 Opus 4.8 与 Grok 4.5 重点评审的问题

建议评审不要只评价“方案是否完整”，而应分别攻击以下假设：

## A. 自治边界

1. L2/B2 是否适合无人值守？
2. 哪些文件和操作必须默认禁止？
3. 夜间是否只能生成 Draft PR，还是允许自动合并低风险任务？

## B. 状态与恢复

1. 状态机是否存在无法恢复的中间态？
2. CAS、租约和 Worktree 是否足以防并发覆盖？
3. Agent 崩溃后是否能仅凭任务目录恢复？

## C. 验收可信度

1. 证据是否可被执行 Agent 伪造或弱化？
2. 哪些 AC 必须工具化，哪些可以模型判定？
3. 如何检测删除测试、放宽断言和更新基线等逃逸行为？

## D. 模型路由

1. Flash/Pro 的任务边界是否清晰？
2. 两次失败后升级是否合理，还是应按失败类型升级？
3. Kimi 的默认零调用和 V2/V3 配额是否足够节约成本？

## E. 指标口径

1. 95%/90%/80% 的分母是否合理？
2. 输入不合格任务如何剔除而不形成指标操纵？
3. 人工只补需求、不改代码是否应计为 AI 失败？

## F. 前端专项

1. Ant Design v6 主题和项目封装的优先级是否明确？
2. 响应式、可访问性和视觉验收阈值如何项目化？
3. 没有专职前端后，Design System 的治理所有权由谁承担？

## G. 最坏情况演练

请两位模型重点推演：

- API 文档与真实接口不一致；
- 设计稿状态不全；
- 公共组件本身存在缺陷；
- 测试环境间歇性失败；
- Agent 修改范围失控；
- Kimi 给出错误视觉建议；
- 多 Agent 同时修改共享类型；
- 任务在 Compact 或服务重启后恢复。

# 21. 最终决策建议

建议采用以下总体策略作为评审基线：

1. **以 CarrorOS 为唯一控制面**，统一任务状态、目录、预算、证据和恢复协议。
2. **以 Work Unit 而非整页/整模块驱动执行**，确保每一步可验证、可回滚。
3. **DeepSeek V4 Pro 负责主要实现，Flash 负责高频低风险任务和明确修复。**
4. **Kimi K3 默认不调用，只在 V2/V3 页面视觉不收敛时调用，单页最多 1–2 次。**
5. **先功能、交互和运行验证，后视觉判断**，避免昂贵模型诊断非视觉问题。
6. **实现和验收上下文隔离**，所有成功声明都必须绑定当前 Commit 的真实证据。
7. **夜间运行仅进入隔离分支或 Draft PR**，初期禁止自动合并和生产发布。
8. **建立固定 Benchmark 和成本指标**，通过数据调整模型路由，而不是凭印象选择模型。
9. **保留前端治理 Owner**，即使不再保留专职页面开发岗位，也不能取消 Design System、交互规范和质量门禁的所有权。
10. **先治理闭环，再追求自治比例。** 第一阶段最重要的不是达到 95%，而是确保 AI 不编造、不越界、可恢复、可审计、失败会停止。

一句话总结：

> **CarrorOS 管状态和边界，DeepSeek 承担绝大多数开发，确定性工具负责证明事实，Kimi K3 只解决少量高价值视觉判断，高阶模型负责规划与争议裁决；通过受限自治而非无限自治，实现可规模化的 AI 前端自主交付。**
